# v2.5 Live E2E Release Gate Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add a release gate command that proves the deterministic local beta runs through the core live API state machine before release.

**Status:** Implemented. Verification is recorded in `docs/ai/VERIFY.md`.

**Architecture:** Implement a small dependency-light HTTP release gate script in `apps/api/scripts/` that talks to a running API with `httpx.Client`. Keep the gate API-first and evidence-focused; do not start servers, run browsers, or call optional model providers. Tests use fake request responses so the release-gate logic is deterministic.

**Tech Stack:** Python standard library, `httpx`, pytest, existing FastAPI demo API.

---

## Task 1: Release Gate Script Core

**Files:**

- Create: `apps/api/scripts/release_gate.py`
- Test: `apps/api/tests/test_v25_release_gate.py`

- [x] **Step 1: Add tests for a successful gate**

Create a fake responder that returns the required payloads for health, readiness, auth, demo seed, plan generation, plan approval, event page, merchant packages, public interactions, merchant workbench, review report, and metrics.

Expected assertions:

- `run_gate(...)` returns exit code `0`;
- all emitted JSON lines have `ok=true`;
- final evidence status is `passed`;
- evidence does not contain cookies, `demo1234`, or token-like values.

- [x] **Step 2: Implement request and step helpers**

Add:

```python
@dataclass
class GateResponse:
    status_code: int
    payload: dict[str, Any]

@dataclass
class GateStep:
    step: str
    ok: bool
    status_code: int | None
    summary: str
```

Use an injectable request function for tests and an `httpx.Client` implementation for real runs.

- [x] **Step 3: Implement the success flow**

Call the required endpoints in order:

1. health;
2. ready;
3. unauthenticated protected route;
4. organizer login and `me`;
5. seed/generate/approve;
6. event page draft/publish;
7. merchant-edge package generation;
8. public event projection;
9. public scan/claim/redeem;
10. merchant login/workbench;
11. organizer login/review report;
12. metrics.

- [x] **Step 4: Verify focused success test**

Run:

```powershell
python -m pytest apps/api/tests/test_v25_release_gate.py -q
```

Expected: focused tests pass.

## Task 2: Failure Semantics And Redaction

**Files:**

- Modify: `apps/api/scripts/release_gate.py`
- Modify: `apps/api/tests/test_v25_release_gate.py`

- [x] **Step 1: Add failure tests**

Cover:

- readiness returns `status=not_ready`;
- public event payload includes forbidden terms;
- required metrics are missing;
- request exception returns non-zero and redacted summary.

- [x] **Step 2: Implement failure handling**

Stop at the first hard failure, emit the failed step JSON line, and return `1`.

Redact:

- `authorization`;
- `cookie`;
- `token`;
- `password`;
- `secret`;
- local absolute paths.

- [x] **Step 3: Verify failure tests**

Run:


Expected: success and failure tests pass.
