# Zhiyin Haojiang v0.8 Agent Capability Spec

Date: 2026-06-12
Project root: `<PROJECT_ROOT>`
Status: design spec for next phase

## 1. Purpose

v0.7 made the current MVP easier to judge: it packaged the deterministic closed loop, screenshots, demo narrative, competition brief, and vision-to-MVP gap review.

v0.8 should now strengthen the actual Agent capability behind the product.

The competition emphasis is not only "a working web demo"; it is also whether the project shows meaningful Agent construction. The current system already has a solid deterministic backend loop and an `AgentTrace` object, but the Agent layer is still thin:

- planning is mostly deterministic service logic
- named agent steps are recorded as evidence, but not yet a richer runtime
- tool calls are not first-class audited objects
- model output is not yet schema-parsed or stored as controlled drafts
- Qwen integration exists only as an optional adapter that falls back to deterministic output
- QwenPaw-style workflow orchestration has not been verified or implemented

The purpose of v0.8 is to turn the current trace-like Agent layer into a visible, testable, competition-legible Agent capability layer while preserving the stable deterministic demo path.

v0.8 must not turn model APIs into a required local dependency. The correct direction is:

```text
deterministic closed loop remains the source of truth
+ specialist Agent runtime and tool-call evidence
+ optional model-backed draft/candidate generation
+ strict schema validation
+ human approval before operational state changes
```

## 3. Problem Statement

The project is now a credible product skeleton, but the Agent layer can still be challenged as mostly rule-based workflow plus trace labels.

That is acceptable for an early deterministic MVP, but not enough for the next phase. If v0.8 only adds more UI polish or presentation material, the project risks looking like:

```text
a polished event management demo with Agent terminology
```

The desired v0.8 target is:

```text
a controlled multi-agent orchestration system where specialist agents decompose tasks,
call tools, produce structured drafts/candidates, pass schema validation,
leave auditable traces, and require human approval before operational changes.
```

The challenge is to increase Agent depth without damaging the current strengths:

- local demo reliability
- deterministic reproducibility
- role-separated product flow
- human approval boundary
- public H5 safety
- no mandatory model key

## 4. Positioning Decision

v0.8 is an Agent capability buildout, not a model-first rewrite.

### 4.1 What Counts As An Agent In This Project

For this project, an Agent is not merely an LLM prompt or a text generator.

An Agent capability must include most of the following:

- a named operational objective
- explicit inputs and input references
- task decomposition or specialist responsibility
- tool calls with structured inputs and outputs
- structured output validated by backend schemas
- confidence or risk notes where useful
- a decision rationale suitable for organizer review
- audit trace persistence
- human approval boundary for operational or public-facing changes
- deterministic fallback when model output is unavailable or invalid

### 4.2 What Does Not Count

The following should not be presented as substantial Agent construction:

- free-form chatbot text pasted into a page
- a model response that bypasses schema validation
- a model response that directly approves plans or publishes notices
- a hidden API call that has no trace, tool record, or fallback
- UI labels that say "Agent" without backend behavior
- a Qwen/QwenPaw mention without verified implementation

### 4.3 Main Technical Stance

The recommended stance for v0.8 is:

```text
Agents may propose, draft, explain, and score.
Only the backend workflow and human approvals may commit operational state.
```

That means model-assisted output can create:

- plan candidates
- route story drafts
- recovery explanation drafts
- public notice drafts
- review summary drafts
- risk notes

But model-assisted output must not directly:

- approve a `PlanVersion`
- close an `Incident`
- mutate `MerchantRuntimeState`
- publish a `PublicNotice`
- modify auth/session data
- overwrite authoritative demo seed data

## 5. Scope

### 5.1 In Scope

v0.8 may define and later implement:

- a first-class Agent runtime contract
- richer Agent run and step objects
- logged tool-call abstractions for current deterministic tools
- specialist Agent modules for planning, recovery, public notice, and review
- structured Agent drafts/candidates
- deterministic Agent pipeline for the current demo event
- optional Qwen-backed draft generation behind an environment flag
- strict schema validation and fallback behavior
- organizer-visible Agent evidence panels if needed after backend work
- tests that prove Agent behavior and safety boundaries
- documentation that clearly distinguishes deterministic, model-assisted, and future QwenPaw capabilities

### 5.2 Out Of Scope

v0.8 must not require:

- mandatory `DASHSCOPE_API_KEY`
- full QwenPaw production workflow rewrite
- real merchant onboarding
- real hardware, POS, payment, or map APIs
- real traffic prediction
- model training or fine-tuning
- broad frontend redesign
- open registration
- autonomous publishing
- autonomous inventory mutation
- autonomous approval

Vector RAG, long-term memory, and multi-event learning should remain future work unless the implementation plan later isolates a very small documentation-only or deterministic retrieval slice.

## 6. Target Agent Architecture

v0.8 should introduce a clearer conceptual architecture. It can be implemented incrementally, but the target design should be explicit.

### 6.1 Coordinator Agent

Name:

```text
CoordinatorAgent
```

Responsibilities:

- receive trigger, event id, current state references, and mode
- decide which specialist agents should run
- enforce allowed actions
- collect tool-call outputs
- merge specialist outputs into structured drafts or proposals
- create final `AgentRun` / `AgentTrace`
- mark fallback behavior when model output is unavailable or invalid

The Coordinator must not directly approve plans, publish notices, or mutate merchant runtime state.

### 6.2 Cultural Narrative Agent

Name:

```text
CulturalNarrativeAgent
```

Responsibilities:

- use curated old-district knowledge files and route point metadata
- draft activity theme/story copy
- produce visitor-facing route story candidates
- avoid backend terminology in public-facing copy
- cite input references such as `knowledge:fulong-new-street`

Initial v0.8 implementation can be deterministic with optional Qwen draft enrichment.

### 6.3 Route Planning Agent

Name:

```text
RoutePlanningAgent
```

Responsibilities:

- choose route points from existing mock route data
- call deterministic route tools
- explain route ordering and fallback points
- identify route risks such as weather exposure, crowding, or merchant dependency
- output structured route candidate or route patch

It must not call real map APIs in v0.8.

### 6.4 Merchant Matching Agent

Name:

```text
MerchantMatchingAgent
```

Responsibilities:

- score candidate merchants using existing merchant profiles and runtime states
- explain why selected merchants fit the event
- identify backup merchants for high-risk points
- produce merchant task candidates

It must not represent real merchant participation beyond demo data.

### 6.5 Risk And Recovery Agent

Name:

```text
RiskRecoveryAgent
```

Responsibilities:

- read `Incident`, `MerchantRuntimeState`, plan, tasks, and route points
- call deterministic incident and recovery tools
- produce a `RecoveryProposal` candidate
- draft an organizer-facing explanation
- draft a visitor-facing public notice candidate
- mark required human approval

This is the most important v0.8 Agent because it directly supports the strongest demo claim: live exception recovery.

### 6.6 Public Notice Agent

Name:

```text
PublicNoticeAgent
```

Responsibilities:

- convert approved or pending recovery context into visitor-safe copy drafts
- remove internal terms such as raw schema names, backend status values, and model labels
- produce zh-Hans, zh-Hant, and en copy candidates if implementation scope allows
- mark whether the notice is ready for organizer approval

It must not publish a notice by itself.

### 6.7 Review Agent

Name:

```text
ReviewAgent
```

Responsibilities:

- read metrics, incidents, public notices, approvals, and plan versions
- produce structured review insights
- distinguish evidence-backed findings from recommendations
- draft competition-readable review prose
- reference input metrics explicitly

The Review Agent should make the "event becomes operational memory" claim stronger.

### 6.8 Merchant Edge Agent

Name:

```text
MerchantEdgeAgent
```

Responsibilities:

- future natural-language merchant onboarding
- future image/voice profile extraction
- future low-friction small merchant interface

This should remain future work for v0.8 unless a tiny deterministic demo slice is explicitly approved later.

## 7. Proposed Data Contracts

The implementation plan may choose the smallest viable schema changes, but v0.8 should use the following target contracts as the design reference.

### 7.1 AgentRun

Purpose:

```text
One execution instance of the Agent runtime for a specific trigger.
```

Proposed fields:

```text
run_id: str
event_id: str
trigger: planning_generation | incident_recovery | public_notice_draft | review_generation
mode: deterministic | qwen_draft | qwenpaw_workflow
status: running | completed | failed | fallback_completed
started_at: str
completed_at: str | None
fallback_used: bool
fallback_reason: str | None
final_output_ref: str | None
error_summary: str | None
```

Notes:

- `mode=qwenpaw_workflow` must not be implemented or claimed until official workflow behavior is verified.
- `fallback_used=true` is acceptable and should be visible, not hidden.

### 7.2 AgentStep

Current `AgentStep` already exists. v0.8 should either extend it or wrap it with richer metadata.

Proposed additional fields:

```text
step_id: str
run_id: str
agent_name: str
objective: str
input_refs: list[str]
tool_call_refs: list[str]
model_call_ref: str | None
schema_name: str
validation_status: passed | failed | fallback
structured_output: dict
decision_reason: str
confidence: float
requires_human_approval: bool
```

If migration cost is high, the implementation may preserve current `AgentStep` and add the missing fields at the `AgentRun` or trace payload level.

### 7.3 AgentToolCall

Purpose:

```text
Auditable record of a deterministic or model-assisted tool invocation.
```

Proposed fields:

```text
tool_call_id: str
run_id: str
step_id: str
tool_name: str
input_payload: dict
output_payload: dict
status: success | failed
latency_ms: int | None
error_summary: str | None
```

Initial tools can wrap existing deterministic functions:

```text
budget.split_budget
merchant.select_night_merchants
route.build_static_route
weather.mock_forecast
incident.from_runtime_state
recovery.build_recovery_proposal
publication.build_public_projection
review.generate_review_report
```

### 7.4 AgentDraft

Purpose:

```text
A model-assisted or deterministic draft that is not yet authoritative state.
```

Proposed fields:

```text
draft_id: str
event_id: str
source_run_id: str
draft_type: plan_candidate | route_story | recovery_explanation | public_notice | review_summary
locale: zh-Hans | zh-Hant | en | mixed
content: str
structured_payload: dict
status: draft | accepted | rejected | superseded
reviewed_by: str | None
reviewed_at: str | None
created_at: str
```

Important boundary:

```text
AgentDraft is not PlanVersion, RecoveryProposal, PublicNotice, or ReviewReport.
```

Draft acceptance may help fill an authoritative object, but it should still pass normal backend workflow and approval gates.

### 7.5 AgentModelCall

Purpose:

```text
Minimal safe record of optional model interaction.
```

Proposed fields:

```text
model_call_id: str
run_id: str
provider: deterministic | dashscope
model: str
prompt_template_id: str
input_refs: list[str]
response_status: skipped | success | invalid_json | schema_failed | provider_error
parsed_output: dict | None
fallback_used: bool
error_summary: str | None
created_at: str
```

Do not persist API keys, raw secrets, or large prompt dumps that may leak private context. If raw prompt/output storage is needed later, it should be redacted and explicitly approved.

### 7.6 AgentEvaluation

Purpose:

```text
Small safety and quality checklist for an Agent output.
```

Proposed fields:

```text
evaluation_id: str
run_id: str
schema_pass: bool
fallback_used: bool
unsafe_mutation_attempted: bool
human_approval_required: bool
forbidden_public_terms_present: bool
public_copy_ready: bool
notes: list[str]
```

This can start as a deterministic validation helper before becoming a persisted object.

## 8. Agent Runtime Modes

### 8.1 deterministic

Default mode.

Requirements:

- runs without network
- runs without `DASHSCOPE_API_KEY`
- produces all required demo outputs
- logs specialist steps and tool calls
- produces drafts using templates or deterministic composition
- all tests must pass in this mode

This remains the required competition demo baseline.

### 8.2 qwen_draft

Optional mode.

Requirements:

- enabled only when explicitly configured
- used for drafts/candidates only
- strict JSON/schema parsing required
- fallback to deterministic draft if response fails
- fallback must be recorded in `AgentRun` or trace
- model output must not commit operational state

Recommended first use cases:

```text
recovery explanation draft
public notice draft
review summary draft
route story draft
```

Avoid using Qwen first for full plan generation until draft parsing and safety checks are verified.

### 8.3 qwenpaw_workflow

Future experimental mode.

Requirements before implementation:

- verify official QwenPaw documentation and current API surface
- confirm license, install path, and runtime assumptions
- confirm whether it fits FastAPI backend orchestration or should be separate
- define how it preserves deterministic fallback
- define how it logs tool calls and human approval boundaries

v0.8 should not claim QwenPaw workflow execution unless this verification is complete and tests pass.

## 9. Core Capability Slices

### 9.1 P0: Deterministic Multi-Agent Runtime

This is mandatory for v0.8 implementation.

Required behavior:

- create an Agent run for plan generation and incident recovery
- run at least six typed specialist steps for the main demo flow
- record tool calls from existing deterministic tools
- produce structured outputs for each specialist step
- store final output references
- mark human approval requirements where relevant
- keep current deterministic demo behavior intact

Minimum expected steps:

```text
CoordinatorAgent
CulturalNarrativeAgent
RoutePlanningAgent
MerchantMatchingAgent
RiskRecoveryAgent
PublicNoticeAgent
ReviewAgent
```

The exact trigger may not need all seven agents every time, but the full demo loop should show at least six meaningful steps across planning, recovery, and review.

### 9.2 P0: Recovery Draft And Public Notice Draft

This is mandatory because it strengthens the best current product behavior.

Required behavior:

- when a merchant sold-out update triggers an incident, the Agent runtime produces:
  - recovery explanation draft
  - public notice draft
  - tool-call evidence
  - human approval flag
- the RecoveryProposal remains authoritative only after normal workflow creation
- the PublicNotice remains unpublished until organizer approval

### 9.3 P0: Review Draft With Evidence References

Required behavior:

- after the demo loop, the Agent runtime produces a review summary draft
- the draft references:
  - plan versions
  - incidents
  - approved recovery proposals
  - public notices
  - operational metrics
- the draft must separate evidence from recommendations

### 9.4 P1: Optional Qwen Draft Backend

Optional if P0 is stable.

Required behavior if implemented:

- use Qwen only for `AgentDraft`
- reject invalid JSON or schema failures
- log provider/model/status
- fall back to deterministic draft
- preserve tests without API key

### 9.5 P1: Organizer Agent Evidence UI

Optional if backend P0 is complete.

Possible UI additions:

- Agent run mode and fallback badge
- specialist step timeline
- tool-call evidence drawer
- draft cards for recovery/public notice/review
- clear "requires organizer approval" boundary

The UI must not become another marketing page. It should feel like operational evidence inside the organizer product.

### 9.6 P2: QwenPaw Workflow Spike

Future work.

Before implementation:

- research official QwenPaw workflow and tool-calling support
- document feasibility and integration risks
- choose whether to integrate directly or only reference as a future architecture

## 10. Trigger Flows

### 10.1 Planning Generation

Trigger:

```text
organizer requests or refreshes plan generation
```

Flow:

```text
1. CoordinatorAgent starts AgentRun(trigger=planning_generation)
2. CulturalNarrativeAgent reads event brief and knowledge references
3. RoutePlanningAgent calls route tool
4. MerchantMatchingAgent calls merchant selection tool
5. RiskRecoveryAgent scans risk inputs
6. CoordinatorAgent assembles PlanVersion candidate
7. backend validates PlanVersion
8. organizer approval remains separate
9. AgentRun final_output_ref points to plan candidate or PlanVersion draft
```

### 10.2 Incident Recovery

Trigger:

```text
merchant runtime state changes to sold_out, overloaded, unavailable, or equivalent risk state
```

Flow:

```text
1. runtime update is saved through existing merchant API
2. incident service creates or updates Incident
3. CoordinatorAgent starts AgentRun(trigger=incident_recovery)
4. RiskRecoveryAgent calls recovery proposal tool
5. PublicNoticeAgent drafts visitor-safe notice
6. AgentEvaluation checks schema, public terms, and approval boundary
7. RecoveryProposal is stored as pending
8. organizer approves or rejects through existing workflow
9. only after approval does backend create PlanVersion v2 and published PublicNotice
```

### 10.3 Public Notice Draft

Trigger:

```text
RecoveryProposal pending or approved
```

Flow:

```text
1. PublicNoticeAgent receives incident/proposal/plan diff context
2. produces draft in visitor-safe language
3. checks forbidden public terms
4. stores AgentDraft(type=public_notice)
5. organizer may use draft content in approval workflow
6. published PublicNotice remains a separate authoritative object
```

### 10.4 Review Generation

Trigger:

```text
organizer opens or refreshes review center after event loop
```

Flow:

```text
1. ReviewAgent reads metrics, incidents, approvals, public notices, plan versions
2. calls review report service or evidence summarizer
3. produces AgentDraft(type=review_summary)
4. backend validates evidence references
5. ReviewReport remains metric-backed and deterministic
```

## 11. Safety And Governance Requirements

### 11.1 Forbidden Direct Mutations

No model-backed Agent output may directly:

- approve `PlanVersion`
- close `Incident`
- publish `PublicNotice`
- update `MerchantRuntimeState`
- create or modify user accounts
- modify sessions
- write demo seed data
- bypass API validation

### 11.2 Required Human Approval

Human approval is required for:

- plan approval
- recovery proposal approval
- public notice publication
- any route change visible to tourists
- any merchant task patch that changes operational assignment

### 11.3 Public Copy Safety

Tourist/public surfaces and drafts must not expose internal/backend terms such as:

```text
PlanVersion
RecoveryProposal
Incident
AgentTrace
backend
schema
fallback
Qwen
QwenPaw
deterministic
runtime state
```

Public copy should explain only what visitors need:

- what changed
- where to go next
- whether a merchant/task is paused
- whether the route has changed
- what action the visitor should take

### 11.4 Secrets And Provider Safety

The system must not persist:

- `DASHSCOPE_API_KEY`
- Authorization headers
- raw secrets from environment variables
- full provider request bodies if they contain sensitive data

If model call logging is added, store redacted metadata and parsed structured output first.

## 12. Knowledge And Retrieval Boundary

The current repo has curated knowledge files under:

```text
data/knowledge/
```

v0.8 may use deterministic retrieval from these files by file reference or small keyword matching. It should not introduce a vector database unless a later implementation plan proves it is necessary and low-risk.

Acceptable v0.8 behavior:

```text
knowledge.lookup("fulong-new-street") -> curated snippet refs
knowledge.lookup("inner-harbour") -> curated snippet refs
```

Not required in v0.8:

```text
embedding pipeline
semantic search service
external knowledge crawler
automatic web ingestion
```

### 13.2 Merchant Visibility

Merchant pages should show only operationally useful results:

- task status
- required preparation
- fallback instruction
- submitted runtime status

Merchant pages should not show model internals or raw agent traces.

### 13.3 Tourist And Public H5 Visibility

Tourist/public pages should show only:

## 14. Testing Requirements

### 14.1 Backend Tests

v0.8 implementation should add or update tests for:

1. deterministic Agent runtime creates an Agent run without API key
2. planning run records at least six meaningful specialist steps across the loop
3. tool calls are recorded with structured input/output payloads
4. merchant sold-out update triggers Incident and incident recovery Agent run
5. recovery run creates a pending RecoveryProposal and AgentDraft without auto-approval
6. approval creates PlanVersion v2 through existing workflow, not directly from model output
7. public notice draft has no forbidden backend terms
8. review draft references metrics/incidents/approvals
9. `AGENT_BACKEND=qwen` without `DASHSCOPE_API_KEY` falls back deterministically
10. invalid model response fails schema validation and falls back
11. model output cannot mutate authoritative state directly
12. existing v0.2-v0.6 backend tests still pass

### 14.2 Frontend Tests

Frontend tests are required only if v0.8 modifies UI.

If UI is changed, tests should cover:

- organizer can see Agent evidence
- merchant cannot see raw Agent internals
- public H5 does not show backend/model terms
- locale switching still works
- sold-out quick action still triggers organizer review

## 16. Stop Gates

Stop and ask for review if:

- the implementation would make model API access mandatory
- QwenPaw integration cannot be verified from official/current materials
- schema parsing for model output is unreliable
- model output starts bypassing human approval
- public/tourist pages would expose backend or model terms
- adding Agent runtime requires a large backend rewrite
- adding Agent visibility requires a broad UI redesign
- tests cannot preserve the current deterministic loop
- scope drifts into real merchants, hardware, real maps, POS, payment, or traffic prediction

## 17. Recommended Implementation Phases

### Phase A: Contracts And Tests

Write or update backend tests for Agent runtime contracts, tool-call logging, draft boundaries, fallback behavior, and forbidden direct mutation.

### Phase B: Deterministic Agent Runtime

Implement a small runtime that can orchestrate specialist agents and current deterministic tools without changing the existing product flow.

### Phase C: Recovery And Public Notice Drafts

Add Agent-generated draft/explanation objects around the sold-out incident recovery path.

### Phase D: Review Agent Draft

Add an evidence-backed review draft that references metrics, incidents, approvals, and plan versions.

### Phase E: Optional Qwen Draft Adapter

If P0 is stable, add strict schema-parsed Qwen draft generation behind a flag. Keep deterministic fallback visible and tested.

### Phase F: Organizer Evidence UI

Only after backend Agent capability is stable, expose a focused organizer evidence panel. Do not redesign the whole product shell.

### Phase G: Handoff

Update status, verification, next-step docs, and commit the verified result.
