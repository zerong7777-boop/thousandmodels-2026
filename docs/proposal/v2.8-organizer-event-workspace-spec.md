# v2.8 Organizer Event Workspace Spec

Status: planned.

## Goal

v2.8 turns the v2.7 real event API into a usable organizer workflow. The organizer should be able to create a non-demo event, see it in the event portfolio, open its workspace, and keep moving through the existing planning, approval, event-page, exception, and review surfaces with the selected `event_id`.

This phase is a product usability step, not a backend platform expansion.

## Background

v2.7 added the backend foundation:

- `POST /api/events` creates draft non-demo events.
- `PATCH /api/events/{event_id}` updates draft events before planning.
- Non-demo plan generation requires an explicit event brief and summary.
- Plan approval and public publication are separated.
- Non-demo public mutations require a published current event page.
- Frontend API/types expose `createEvent` and `updateEvent`.

The remaining gap is that the organizer UI still feels demo-centered. The app has an event list route, but it does not yet provide a practical event creation flow, selected-event workspace context, or dynamic organizer shell navigation for non-demo events.

## Scope

v2.8 should deliver a narrow event workspace layer:

1. Upgrade `/organizer/events` into the primary event portfolio and creation entry.
2. Add a focused create-event form backed by `api.createEvent`.
3. Navigate to `/organizer/events/{event_id}` after successful creation.
4. Make the organizer shell's workspace, exceptions, and review links follow the currently selected event id when the current path contains one.
5. Make `OrganizerEventWorkspacePage` show the selected event context, not just generic route-plan copy.
6. Preserve all deterministic demo routes and release gates.

## Non-Goals

- No merchant assignment editor.
- No event archive/delete/clone.
- No full multi-step wizard.
- No marketing landing page.
- No real merchant onboarding, POS, payment, hardware, map, traffic, weather, or visitor identity integration.
- No QwenPaw production orchestration.
- No broad visual redesign.

## Product Requirements

### Event Portfolio

- `REQ-HUB-001`: `/organizer/events` must be the organizer's event portfolio page.
- `REQ-HUB-002`: The page must load events from `api.getEvents()` and render each event's title, area, date, time window, lifecycle status, public release status, and current plan version.
- `REQ-HUB-003`: The event list must not fabricate a real non-demo event when the backend returns an empty list.
- `REQ-HUB-004`: The page may show a demo event action, but it must be labeled as demo and must call the explicit demo seed path.
- `REQ-HUB-005`: Each event row must link to workspace, exceptions, and review for that event id.

### Create Event Flow

- `REQ-CREATE-001`: `/organizer/events` must include a compact create-event form.
- `REQ-CREATE-002`: The form must collect the v2.7 required payload fields:
  - optional `event_id`;
  - `title`;
  - `area`;
  - `date`;
  - `time_window`;
  - `budget_mop`;
  - `target_audience`;
  - `event_goal`;
  - `theme_preferences`;
  - `constraints`;
  - `priority_rules`.
- `REQ-CREATE-003`: List-like fields may use comma-separated or newline-separated input and must be parsed into non-empty string arrays before submit.
- `REQ-CREATE-004`: The form must prevent submit when scalar required fields are empty, `budget_mop <= 0`, or any list-like required field parses to an empty array.
- `REQ-CREATE-005`: On success, the app must navigate to `/organizer/events/{created.event_id}`.
- `REQ-CREATE-006`: On backend validation or conflict errors, the form must show a concise error message and keep user-entered values.
- `REQ-CREATE-007`: The form must not call `/api/events/demo/seed` or create demo records.

### Selected Event Navigation

- `REQ-NAV-001`: When the current path contains `/organizer/events/{event_id}`, the organizer shell must use that event id for workspace, exceptions, and review links.
- `REQ-NAV-002`: When the current path does not contain an event id, organizer shell links may fall back to `demo-night-tour`.
- `REQ-NAV-003`: The app must keep `/review/{event_id}` compatibility, but organizer-owned navigation should prefer `/organizer/events/{event_id}/review`.

### Workspace Context

- `REQ-WORKSPACE-001`: `OrganizerEventWorkspacePage` must load the selected event summary using `api.getEvent(eventId)`.
- `REQ-WORKSPACE-002`: The workspace header must display the selected event title, area, date, time window, status, and public release state.
- `REQ-WORKSPACE-003`: For a draft event with no plan, the workspace must explain that the next action is to build the route plan.
- `REQ-WORKSPACE-004`: Existing plan generation, approval, event-page draft/publish, merchant package generation, operation suggestions, evidence, and task panels must continue to operate on the selected event id.
- `REQ-WORKSPACE-005`: Workspace failures for missing event, missing plan, or invalid transitions must surface as user-visible feedback instead of silent empty panels.

### Demo Boundary

- `REQ-DEMO-001`: Existing demo path `/organizer/events/demo-night-tour` must remain deterministic.
- `REQ-DEMO-002`: Demo seed must remain explicit through `api.seed()`.
- `REQ-DEMO-003`: Browser and API release gates must not need a newly created non-demo event unless v2.8 tests explicitly create one.

### Internationalization and UI

- `REQ-I18N-001`: New visible text must use the existing dictionary system for `en`, `zh-Hans`, and `zh-Hant`.
- `REQ-UI-001`: The UI should stay operational and dense. No hero page, no landing-page composition, and no broad palette or shell redesign.
- `REQ-UI-002`: Form controls must have stable layout on desktop and mobile. Buttons must not resize the page after submit.

## Data Flow

1. Organizer opens `/organizer/events`.
2. UI calls `api.getEvents()`.
3. Organizer submits create form.
4. UI validates the payload locally and calls `api.createEvent(payload)`.
5. Backend returns `EventSummary`.
6. UI navigates to `/organizer/events/{event_id}`.
7. Workspace loads event summary and existing workspace data for that event.
8. Organizer builds and approves the plan through existing v2.7 endpoints.

## Error Handling

- Empty event list: show an empty state with create form and explicit demo seed action.
- Duplicate event id: show backend conflict text in the form area.
- Invalid create payload: keep the draft form values and show validation feedback.
- Workspace event not found: show a compact error state with a link back to `/organizer/events`.
- Draft event with no plan: show no-plan state and keep the build-plan action available.

## Test Requirements

- Add frontend tests for event portfolio rendering.
- Add frontend tests for create-event validation and successful navigation.
- Add frontend tests that the organizer shell uses the selected event id in nav links.
- Add frontend tests that a non-demo draft workspace shows event context and calls `api.generatePlan(eventId)`.
- Keep existing route, auth, shell separation, v1.2 event interaction, and public projection tests passing.
- Run frontend tests and build before closeout.
- Run backend focused or full tests only if v2.8 changes backend code. The expected path is frontend-only.

## Acceptance Criteria

v2.8 is complete when:

1. A logged-in organizer can create a non-demo event from `/organizer/events`.
2. The app navigates to the new event workspace after creation.
3. The organizer shell links follow the selected event id.
4. The selected workspace shows real event context and can start route-plan generation for that event.
5. Demo routes still work.
6. New tests cover the event workspace flow.
7. `npm.cmd run test` and `npm.cmd run build` pass for `apps/web`.

## Follow-Up
