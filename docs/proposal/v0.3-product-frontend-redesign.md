# 智引濠江 v0.3 Product Frontend Redesign Spec

版本：v0.3 design draft  
日期：2026-06-10  
项目根目录：`<PROJECT_ROOT>`  
状态：下一阶段设计约束，不代表当前已实现能力  
前置状态：v0.2 后端 deterministic 闭环和多入口路由已完成

## 1. 背景

v0.2 已经解决了两个基础问题：

- 后端有 deterministic 状态闭环，可以跑通 `PlanVersion`、`Incident`、`RecoveryProposal`、游客端 projection 和指标复盘。
- 前端从单页 demo shell 改成多个 URL 入口，商户端、游客 H5、复盘中心可以通过独立路由访问。

但 v0.2 仍然没有达到产品成熟度要求。主要问题不是“功能不够”，而是：

1. 整体视觉仍像通用管理后台，审美和比赛展示说服力不足。
2. 当前“分端”只是 URL 分开，不是身份、权限、导航和信息架构真正分开。
3. 如果在这个基础上直接接 Qwen/QwenPaw，会把问题从产品框架转移成模型接入，无法解决核心体验缺陷。

因此 v0.3 的目标不是继续补几个页面，而是重新定义前端产品形态。

## 2. v0.3 目标

v0.3 要把当前前端从“多页面后台 demo”推进到：

> 一个有登录入口、角色身份、分端 shell、可信视觉风格和清晰演示路径的旧区文旅活动协同产品。

v0.3 必须先完成调研和产品设计，再进入实现。调研不是附属任务，而是 v0.3 的第一阶段交付物。

### 2.1 必须解决的问题

- 找到适合本项目的开源模板、页面结构、组件风格和视觉参考，避免继续在丑陋后台壳上硬改。
- 建立真实的角色分端模型：
  - 主办方端
  - 商户端
  - 用户/游客端
- 增加 demo login/session 机制，让不同角色进入不同端，而不是共享一个后台导航。
- 让三端在视觉、信息密度、操作方式上明显不同。
- 保持 v0.2 deterministic 闭环和 API 合约不被破坏。
- 明确 Qwen/QwenPaw 只在前端框架稳定后接入。

### 2.2 不解决的问题

v0.3 不做以下内容：

- 不接真实商户。
- 不接硬件、AI 眼镜、IoT 或机器人。
- 不做真实交通预测。
- 不训练或微调模型。
- 不把 Qwen/QwenPaw 作为本地 demo 的必需路径。
- 不做完整 SaaS 权限系统、计费、多租户、组织管理。
- 不做营销 landing page。
- 不重写后端核心 deterministic 状态闭环，除非前端集成暴露明确 API 缺口。

## 3. 核心判断

### 3.1 多路由不等于分端

当前 v0.2 已经有这些入口：

```text
/organizer
/organizer/events/demo-night-tour
/organizer/events/demo-night-tour/exceptions
/merchant/m001
/public/events/demo-night-tour
/review/demo-night-tour
```

但这还不等于真正分端。真正分端至少包括：

- 不同身份进入不同默认首页。
- 不同身份使用不同导航。
- 不同身份看到不同信息密度。
- 不同身份拥有不同操作权限。
- 用户/游客端不能出现后台术语和后台导航。
- 商户端不能像主办方系统的一个子页面。

v0.3 必须把“入口分开”升级为“身份和产品端分开”。

### 3.2 视觉方向不能继续沿用通用后台

当前 UI 最大问题是“像某个通用管理后台”。这会削弱比赛项目的独特性：

- 看不出澳门旧区文旅场景。
- 看不出活动运营和城市协同的专业感。
- 看不出游客端和商户端是独立产品体验。
- 视觉上没有记忆点，也没有演示时的可信质感。

v0.3 不要求做炫技页面，但必须摆脱默认后台模板感。目标是“有产品气质的文旅活动协同系统”，不是“Ant Design CRUD 后台”。

### 3.3 Qwen/QwenPaw 必须后置

Qwen/QwenPaw 的接入价值在 Agent 工作流解释、文化叙事、恢复建议表达和复盘文案生成。它们不能用来掩盖前端产品结构不成熟。

Qwen/QwenPaw 进入实现前必须满足：

- 三端 login/session 骨架完成。
- 三端 shell 和导航完成。
- 关键页面完成视觉重构。
- deterministic demo 仍然可跑通。
- 端侧页面已经能展示真实 v0.2 数据。

## 4. v0.3 阶段划分

v0.3 分为五个阶段。每个阶段必须有明确产出，不能跳过。

### Phase 0: Research Scout and Visual Direction

目的：系统调研开源模板、组件库、页面模式和可复用 UI，而不是凭空改样式。

产出：

- `docs/research/frontend-template-scout.md`
- `docs/research/frontend-visual-reference-matrix.md`
- `docs/research/frontend-ui-decision.md`

验收：

- 至少调研 12 个开源模板、组件库或 UI kit。
- 至少调研 6 个非代码层面的产品视觉参考。
- 每个候选必须记录：
  - 链接
  - license
  - 技术栈
  - 是否支持 React/Vite
  - 可复用页面或组件
  - 视觉优点
  - 与本项目不匹配之处
  - 是否建议采用
- 必须给出 2 到 3 个候选方向，并说明推荐方向。
- 未完成 Phase 0 前，不允许进入页面重构。

### Phase 1: Demo Identity and Role Shell

目的：建立 demo 级登录和身份分端。

产出：

- `/login`
- 主办方端 shell
- 商户端 shell
- 用户/游客端 shell
- demo session 存储和 route guard

验收：

- 用户打开根路径时进入登录或角色选择页。
- 选择主办方身份后进入主办方端。
- 选择商户身份后进入商户端。
- 选择用户/游客身份后进入用户端。
- 三端导航、布局和可见操作不同。
- 未登录访问受保护页面时回到 `/login`。
- 游客公开 H5 可以保留免登录入口，但不能套主办方 shell。

### Phase 2: Visual System and Layout Redesign

目的：确定并应用 v0.3 视觉系统。

产出：

- 设计 token：颜色、字号、间距、卡片、表格、状态标签、移动端控件。
- 三端布局规范。
- 核心组件重构：
  - activity summary
  - approval queue
  - incident card
  - agent trace
  - route point card
  - merchant task card
  - public notice
  - metric panel

验收：

- 页面不再像默认后台模板。
- 主办方端像运营产品，而不是宣传页或 CRUD 后台。
- 商户端像轻量工作台，优先服务移动或平板场景。
- 用户/游客端像真实活动入口，具有文旅场景感。
- 不使用单一色相铺满整个产品。
- 不使用营销 hero 作为产品首页。

### Phase 3: Role-Specific Page Reconstruction

目的：按身份重构页面，而不是复用一个后台壳。

产出：

- 主办方端：
  - dashboard
  - event workspace
  - exception center
  - review center
- 商户端：
  - merchant dashboard
  - task detail
  - runtime status report
  - notification feed
- 用户/游客端：
  - activity home
  - route timeline
  - route point detail
  - public notice

验收：

- 主办方端可以完成创建/生成/审批/恢复/复盘的演示路径。
- 商户端可以查看任务并上报库存、排队和可接待状态。
- 用户端可以查看路线、站点故事、任务和通知。
- 商户端和用户端不得作为主办方页面内嵌预览存在。

### Phase 4: Deterministic Demo Integration and Verification

目的：保证 v0.3 视觉和分端重构没有破坏 v0.2 状态闭环。

产出：

- 前端 role/session tests。
- 前端 route guard tests。
- 前端 build。
- 后端 pytest 回归。
- 本地浏览器 walkthrough checklist。

验收：

- `npm run test` 通过。
- `npm run build` 通过。
- `python -m pytest -q` 通过。
- 真实本地演示路径可以从登录开始跑完：
  - login as organizer
  - seed/generate v1
  - approve v1
  - login as merchant
  - report sold out
  - login as organizer
  - approve recovery
  - open user/tourist experience
  - see current plan version and notice update
  - open review center

### Phase 5: Qwen/QwenPaw Readiness Review

目的：判断是否进入模型工作流接入，不在 v0.3 默认实现。

产出：

- `docs/proposal/qwen-integration-readiness.md`

验收：

- 明确哪些点适合 Qwen/QwenPaw：
  - 文化叙事生成
  - Agent step 解释
  - recovery proposal 解释
  - public notice 改写
  - review report 表达
- 明确哪些点仍然不能交给模型直接决定：
  - 库存状态
  - 路线约束校验
  - 审批状态
  - 发布状态
  - 预算计算
- 明确 deterministic fallback 仍然存在。

## 5. Phase 0 调研任务细化

调研任务必须单独执行，不能用几句“参考某模板”带过。

### Phase 0 执行规则

Phase 0 是一个独立 research checkpoint。执行时必须联网查看候选项目、官方文档、GitHub repo、license 和 demo 页面，不能只基于记忆写结论。

每个候选至少记录：

```text
Name
URL
Repository URL
License
Last meaningful update
Framework and dependencies
Reusable pages/components
Visual fit score: 1-5
Implementation fit score: 1-5
Role fit: organizer / merchant / user
Risks
Decision: adopt / reference only / reject
Reason
```

调研文档必须附：

- 原始链接。
- 截图或截图路径，至少覆盖入选候选。
- 明确评分，不允许只写“好看”“不错”“可以参考”。
- 至少 3 个 rejected candidates，说明为什么不用。
- 最终推荐只能有 1 个主方案，最多 1 个备选方案。

Phase 0 完成前，任何前端视觉重构、组件库替换、登录页设计都不得开始。

### R0: 调研评价标准

先写清楚候选模板怎么评价。

评价维度：

- 视觉成熟度
- 是否摆脱普通后台感
- 是否适合文旅/城市活动场景
- 是否支持多端 shell
- 是否支持移动端游客体验
- React/Vite 兼容性
- Ant Design 兼容或替换成本
- license 风险
- 组件复用比例
- 本项目改造成本

产出：

```text
docs/research/frontend-research-criteria.md
```

### R1: 开源模板和 UI kit 扫描

目标：找可复用的结构、组件和视觉系统。

候选来源包括但不限于：

- Ant Design Pro 或 ProComponents 生态
- Arco Design Pro
- Semi Design templates
- TDesign templates
- shadcn/ui dashboard examples
- Tremor dashboard examples
- Tailwind dashboard kits
- Mantine templates
- React admin templates
- mobile-first H5/event templates

注意：这些只是搜索方向，不是已确认推荐。每个候选必须在调研时重新核对 license、维护状态和适配成本。

产出：

```text
docs/research/frontend-template-scout.md
```

最低内容：

- 候选列表不少于 12 个。
- 每个候选至少一段评价。
- 明确哪些只是视觉参考，哪些可以复用组件或代码。
- 明确 rejected candidates 和拒绝原因。

### R2: 产品视觉参考调研

目标：不要只看后台模板，还要看真实产品怎么处理活动、路线、商户和用户体验。

参考类型：

- 活动运营平台

- 城市导览或文化路线产品
- 商户任务/订单工作台

最低内容：

- 每个参考说明可借鉴点。
- 不照抄品牌、图片或商业设计，只抽象布局和交互模式。
- 明确哪些参考适合主办方端、商户端、用户端。

## 6. v0.3 信息架构

### 6.1 登录入口

新增 `/login`，作为 demo 身份入口。

登录不是正式安全系统，而是演示级身份选择：

```text
主办方演示账号
商户演示账号：m001
用户/游客演示账号
```

可选实现：

- 三个身份按钮。
- 简单账号列表。
- session 写入 `localStorage`。

不要求：

- 密码加密。
- JWT。
- OAuth。
- 多组织权限。
- 真实注册。

### 6.2 主办方端

默认入口：

```text
/organizer/dashboard
```

主办方端页面：

```text
/organizer/dashboard
/organizer/events
/organizer/events/:eventId
/organizer/events/:eventId/exceptions
/organizer/events/:eventId/review
```

主办方 shell：

- 左侧或顶部产品导航。
- 活动对象为中心。
- 信息密度较高。
- 有待审批、异常、版本和 Agent evidence。
- 可进入商户和用户端，但只能作为外部入口链接，不内嵌其页面。

主办方可执行动作：

- 初始化 demo 活动。
- 生成 v1。
- 审批 v1。
- 查看 AgentTrace。
- 查看异常队列。
- 生成 RecoveryProposal。
- 审批恢复。
- 查看 v1 -> v2 diff。
- 查看复盘。

### 6.3 商户端

默认入口：

```text
/merchant/dashboard
```

商户端页面：

```text
/merchant/dashboard
/merchant/events/:eventId/tasks
/merchant/events/:eventId/status
/merchant/notifications
```

商户 shell：

- 移动或平板优先。
- 不显示主办方导航。
- 不显示复杂 Agent trace。
- 聚焦“今天我要做什么”和“我要上报什么状态”。

商户可执行动作：

- 查看今日任务。
- 查看准备清单。
- 上报库存正常。
- 上报库存紧张。
- 上报库存售罄。
- 上报排队过载。
- 上报暂停接待。
- 查看主办方确认后的新通知。

### 6.4 用户/游客端

默认入口：

```text
/user/events/:eventId
```

保留公开兼容入口：

```text
/public/events/:eventId
```

用户端页面：

```text
/user/events/:eventId
/user/events/:eventId/route
/user/events/:eventId/points/:pointId
/user/events/:eventId/notices
```

用户 shell：

- 移动端优先。
- 不显示主办方导航。
- 不显示商户后台信息。
- 不显示后台术语，例如 `PlanVersion`、`RecoveryProposal`、`Incident`。
- 可以显示“当前路线版本”但用游客语言，例如“路线已更新”。

用户可执行动作：

- 查看活动概览。
- 查看路线点。
- 查看站点故事。
- 查看打卡任务。
- 查看活动通知。
- 查看路线调整原因。

### 6.5 复盘归属

复盘中心属于主办方端，不再作为完全独立第四端。

推荐入口：

```text
/organizer/events/:eventId/review
```

当前 v0.2 的 `/review/:eventId` 可以保留为兼容重定向或快捷入口，但 v0.3 产品语义上应归入主办方端。

## 7. 权限和 Route Guard

v0.3 使用 demo 权限，不做真实安全。

### 7.1 Session 模型

最小 session：

```ts
type DemoRole = "organizer" | "merchant" | "tourist";

interface DemoSession {
  role: DemoRole;
  actor_id: string;
  display_name: string;
  merchant_id?: string;
}
```

存储：

```text
localStorage["zhiyin.demo.session"]
```

### 7.2 Route Guard 规则

规则：

- `/login` 对所有人可见。
- `/organizer/*` 需要 `role = organizer`。
- `/merchant/*` 需要 `role = merchant`。
- `/user/*` 需要 `role = tourist`。
- `/public/events/*` 可免登录。
- 角色不匹配时跳回 `/login`，并显示“请选择对应身份”。

### 7.3 Demo 切换身份

为了演示效率，每个端可以保留“切换身份”入口，但不能让三端共享同一个后台导航。

## 8. 视觉方向约束

### 8.1 总体风格

目标风格：

> 精致、可信、具有澳门旧区文旅气质的活动协同产品。

避免：

- 默认后台模板感。
- 大面积单一蓝灰或紫色渐变。
- 营销 hero 首页。
- 所有内容都放在浮动卡片里。
- 卡片套卡片。
- 空洞宣传话术。

可以使用：

- 克制的中性色底。
- 小面积高识别度强调色。
- 路线、时间线、状态、通知等业务视觉元素。
- 澳门旧区文化纹理或实景图片，但必须服务内容，不做纯装饰。
- 8px 或更小的卡片圆角，除非调研选定模板有明确一致规范。

### 8.2 主办方端风格

主办方端应该像“运营指挥台”，不是普通 admin。

重点：

- 活动状态。
- 待审批。
- 异常风险。
- 方案版本。
- Agent evidence。
- 复盘指标。

页面密度：

- 桌面优先。
- 信息密度中高。
- 清晰分区，不做大面积空白展示。

### 8.3 商户端风格

商户端应该像“今日任务工作台”。

重点：

- 今日任务。
- 准备事项。
- 上报状态。
- 主办方通知。

页面密度：

- 移动或平板优先。
- 操作按钮清楚。
- 状态上报控件大且直接。

### 8.4 用户/游客端风格

用户端应该像真实活动 H5。

重点：

- 活动路线。
- 站点故事。
- 打卡任务。
- 临时通知。
- 商户互动。

页面密度：

- 移动优先。
- 视觉更有场景感。
- 不显示后台术语。
- 不像管理系统。

## 9. 组件边界

v0.3 需要把组件按端和共享层重新组织。

推荐结构：

```text
apps/web/src/
  auth/
    session.ts
    routeGuards.ts
  shells/
    OrganizerShell.tsx
    MerchantShell.tsx
    UserShell.tsx
  pages/
    login/
    organizer/
    merchant/
    user/
  components/
    common/
    organizer/
    merchant/
    user/
  design/
    tokens.ts
    statusColors.ts
```

共享组件只放真正跨端通用的东西，例如：

- status tag
- loading state
- empty state
- date/time formatter

不要把主办方的大型组件拿去商户端和用户端复用，导致三端又长得一样。

## 10. 测试要求

v0.3 必须新增或更新测试，不允许只靠手动打开页面。

### 10.1 Auth and Routing Tests

必须覆盖：

- 未登录访问 `/organizer/dashboard` 跳到 `/login`。
- 主办方登录后可以进入 `/organizer/dashboard`。
- 商户登录后访问 `/organizer/dashboard` 被拒绝。
- 商户登录后可以进入 `/merchant/dashboard`。
- 用户登录后可以进入 `/user/events/demo-night-tour`。
- `/public/events/demo-night-tour` 可免登录访问。

### 10.2 Shell Separation Tests

必须覆盖：

- 用户端不出现主办方导航。
- 商户端不出现主办方异常中心导航。
- 主办方首页不直接渲染商户上报按钮。
- 主办方首页不直接渲染用户路线 H5。

### 10.3 Flow Tests

必须覆盖：

- 主办方端生成并审批 v1。
- 商户端库存售罄上报触发 Incident。
- 主办方端审批 recovery 后生成 v2。
- 用户端看到更新后的通知或版本。

### 10.4 Browser-Level Review

建议增加 Playwright 或等价浏览器 smoke。若暂不引入 Playwright，也必须保留人工 walkthrough checklist。

至少检查：

- desktop 主办方端。
- tablet 商户端。
- mobile 用户端。
- 页面没有明显遮挡、重叠、溢出。
- 用户端不像后台。
- 商户端不像主办方子页。

## 11. 验收标准

v0.3 完成时必须满足：

1. Phase 0 调研文档完成，并经过用户确认。
2. `/login` 存在，能选择主办方、商户、用户/游客身份。
3. 三端有独立 shell，而不是共享同一个后台导航。
4. 主办方端默认首页是运营工作台。
5. 商户端默认首页是今日任务和状态上报。
6. 用户端默认首页是活动路线和故事入口。
7. 用户端不出现后台术语和后台导航。
8. 商户端不出现主办方审批和 Agent trace 主导航。
9. 主办方端可以进入活动工作区、异常中心和复盘。
10. 商户端状态上报仍能触发后端 Incident。
11. 用户端仍使用 public projection 或 user projection 数据。
12. v0.2 deterministic demo loop 仍可运行，不依赖 `DASHSCOPE_API_KEY`。
13. 前端测试通过。
14. 前端 build 通过。
15. 后端 pytest 通过。
16. 文档记录模板调研、视觉决策、实现结果和验证结果。

## 12. Qwen/QwenPaw 后置条件

在以下条件满足前，不进入 Qwen/QwenPaw 实现：

- v0.3 三端 shell 完成。
- v0.3 视觉方向完成。
- 登录和 route guard 完成。
- deterministic flow 测试通过。
- 用户确认当前产品形态不再像普通后台 demo。

满足后再写独立 spec：

```text
docs/proposal/qwen-agent-workflow-integration.md
```

该 spec 必须明确：

- 哪些 Agent step 使用 Qwen。
- 哪些输出需要 JSON schema 校验。
- 失败时如何回退 deterministic fallback。
- 不允许模型直接改变审批、发布、库存、预算等状态。

## 13. 建议实施顺序

后续 implementation plan 应按以下顺序写：

1. Phase 0 research tasks only。
2. 用户确认 research decision。
3. Login/session/route guard tests。
4. Demo auth and role shell implementation。
5. Visual token and layout foundation。
6. Organizer shell and dashboard redesign。
7. Merchant shell and dashboard redesign。
8. User/tourist shell and H5 redesign。
9. Flow integration tests。
10. Browser walkthrough and polish。
11. Documentation and handoff。
12. Qwen/QwenPaw readiness review。

注意：第 1 到 2 步是硬门槛。不能跳过调研直接重画页面。

## 14. 当前结论

v0.3 的核心不是“把页面做漂亮一点”，而是把产品从比赛 demo 形态推进到可信的多端协同产品形态。

一句话目标：

> 让评委看到的不是一个统一后台页面，而是主办方、商户、游客三类角色在同一个旧区活动中协同工作的真实产品雏形。

下一步应先执行 Phase 0 research，不写代码，不改页面。
