# Zhiyin Haojiang v1.2 Event Page And Merchant Edge Agent Implementation Plan

> **For agentic workers:** Use `superpowers:executing-plans` or `superpowers:subagent-driven-development` before implementation. Use `superpowers:test-driven-development` for every behavior change and `superpowers:verification-before-completion` before reporting completion.

**Spec:** `docs/proposal/v1.2-event-page-merchant-edge-agent-spec.md`

**Goal:** Upgrade the current v1.1 demo from "route + public H5 + agent evidence" into an event-page-centered operation loop with merchant-side edge agent packages, simulated in-shop touchpoints, coupon/redemption interactions, operation suggestions, review evidence, and a repaired optional Qwen live-draft path.

**Execution shape:** Implement Task 1-7 first as backend P0, then pause for a checkpoint if requested. Continue Task 8-10 for frontend exposure, visual smoke, and handoff.

**Non-goals:** No real POS/payment/coupon issuance, no real merchant integration, no hardware, no visitor account system, no real map API, no real traffic prediction, no QwenPaw implementation claim, no Qwen-required demo path.

---

## 0. Scope Lock

The v1.2 work must keep the v1.1 deterministic closed loop intact:

1. Seed demo event.
2. Generate and approve `PlanVersion` v1.
3. Merchant runtime update can trigger `Incident`.
4. Recovery proposal approval creates `PlanVersion` v2 and public updates.
5. Review report remains metric-backed.
6. Qwen/DashScope remains optional and never authoritative.

The new v1.2 loop adds:

1. Published `EventPage` as the public-facing event container.
2. Per-merchant `MerchantInteractionPackage` objects.
3. Simulated `InShopTouchpoint`, `CouponRule`, `CouponRedemption`, and `TouchpointInteraction` records.
4. `OperationSuggestion` objects generated from merchant state, incident state, and interaction metrics.
5. Merchant-level review evidence and extension tasks.
6. Qwen candidate normalization and prompt-payload sanitization so the live smoke can succeed without weakening mutation guards.

## 2. Compatibility Rules

Keep these constraints throughout implementation:

1. Preserve existing API routes unless a test proves a route is broken.
2. Add new v1.2 fields and endpoints in a backward-compatible way.
3. Existing public H5 must still work when no `EventPage` has been generated.
4. Existing merchant workbench must still return runtime state and tasks.
5. Public and merchant UI copy must not expose backend terms such as raw schema names, model provider names, or internal approval fields.
6. New touchpoint interactions must be anonymous. Do not add visitor registration, visitor login, visitor profile, phone number, email, or identity linkage.
7. Qwen output can draft text or structured suggestions only. Qwen cannot approve, publish, create plan versions, mutate inventory, redeem coupons, or write authoritative state.

### Task 6: Generate Operation Suggestions And Enrich Review Evidence

**Purpose:** Make the agent capability visible in operational decisions, not only in copy drafts.

**Files:**

- `apps/api/app/services/operation_suggestions.py`
- `apps/api/app/agents/runtime.py`
- `apps/api/app/main.py`
- `apps/api/app/services/review_report.py` or existing review service file
- `apps/api/tests/test_v12_event_page_merchant_edge.py`

**Suggestion generation inputs:**

1. Current `MerchantRuntimeState`.
2. Open `Incident` records.
3. Touchpoint metrics.
4. Coupon claim/redemption rates.
5. Approved `PlanVersion`.

**Deterministic rules:**

1. If a merchant has inventory `0` or `is_accepting_visitors == False`, generate `merchant_capacity`.
2. If queue minutes exceed the existing threshold, generate `route_adjustment` or `public_notice`.
3. If claim rate is high and redemption rate is low, generate `coupon_rebalance`.
4. If a merchant has high scan count but low redemption, generate `extension_task`.

**Service API:**

