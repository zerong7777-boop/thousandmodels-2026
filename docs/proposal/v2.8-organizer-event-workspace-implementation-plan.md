# v2.8 Organizer Event Workspace Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build the organizer UI layer that lets users create, select, and operate on non-demo events created by the v2.7 API.

**Architecture:** Keep the implementation frontend-first. Add a focused create-event panel to the existing organizer events page, route all selected-event links through the current `event_id`, and make the workspace header load/display event summary context. Preserve the existing demo path.

**Tech Stack:** React, TypeScript, Vitest, Testing Library, existing local UI components, existing `api.createEvent`, `api.getEvents`, `api.getEvent`, and organizer routes.

Implement these exact helper semantics in `EventCreatePanel.tsx`:

```ts
export function parseListField(value: string): string[] {
  return value
    .split(/[\n,]/)
    .map((item) => item.trim())
    .filter(Boolean);
}
```

Use a local state shape matching `EventCreateRequest`, with text fields for list-like values before parsing.

- [ ] **Step 2: Implement local validation**

- `title`, `area`, `date`, `time_window`, and `event_goal` must be non-empty;
- `budget_mop` must parse to a positive number;
- `target_audience`, `theme_preferences`, `constraints`, and `priority_rules` must each parse to at least one item.

On validation failure, show one concise form-level message and do not call `api.createEvent`.

- [ ] **Step 3: Submit to api.createEvent**

- [ ] **Step 1: Add selected event id extraction**

Add helper:

```ts
function eventIdFromPath(pathname: string): string {
  const parts = pathname.split("/").filter(Boolean);
  if (parts[0] === "organizer" && parts[1] === "events" && parts[2]) {
    return parts[2];
  }
  if (parts[0] === "review" && parts[1]) {
    return parts[1];
  }
  return "demo-night-tour";
}
```
