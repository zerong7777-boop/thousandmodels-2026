# v2.9 Event Operations Setup Pack Spec

Status: implemented locally on `feat/v29-event-operations-setup-pack`.

## Goal

v2.9 turns a newly created event into an operationally configurable event before route planning. Organizers should be able to choose participating merchants for a selected event, mark whether those merchants are ready, see planning readiness in the workspace, and generate a route plan only from the selected event merchant scope.

This phase closes the product gap between "an event exists" and "an event is ready to plan".

## Background

v2.7 added real non-demo event lifecycle APIs.
v2.8 exposed event creation and selected-event workspace navigation in the organizer UI.

The current remaining gap is that non-demo `generate-plan` still uses the shared merchant catalog through `STORE.list_merchants()`. That means a newly created event can appear ready for planning even though the organizer has not selected which merchants are participating in that event.

## Scope

v2.9 should deliver an Event Operations Setup Pack:

1. Event-level merchant roster data model and store helpers.
2. Organizer APIs to read and update a selected event's merchant roster.
3. Non-demo planning guard: no ready roster, no plan generation.
4. Route planning uses only ready participating merchants for the selected event.
5. Workspace merchant setup panel for selected events.
6. Readiness summary visible before planning.
7. Demo path compatibility for `demo-night-tour`.

## Non-Goals

- No public merchant onboarding.
- No merchant self-registration.
- No external CRM, POS, payment, hardware, map, weather, traffic, or visitor identity integration.
- No multi-tenant organization model.
- No invitation email/SMS workflow.
- No capacity forecasting engine.
- No broad UI redesign.
- No QwenPaw production orchestration.

## Product Requirements

### Event Merchant Roster

- `REQ-ROSTER-001`: Each event can have a merchant roster independent from the global merchant catalog.
- `REQ-ROSTER-002`: A roster entry must include `event_id`, `merchant_id`, participation status, readiness status, optional role hint, optional notes, `created_at`, and `updated_at`.
- `REQ-ROSTER-003`: Participation status values must be `invited`, `confirmed`, or `declined`.
- `REQ-ROSTER-004`: Readiness status values must be `missing`, `needs_setup`, or `ready`.
- `REQ-ROSTER-005`: Roster APIs must reject unknown `event_id` and unknown `merchant_id`.
- `REQ-ROSTER-006`: Replacing the roster must be deterministic: duplicate merchant IDs are de-duplicated, current entries for removed merchants are removed, and existing status/notes are preserved for retained merchants.

### Readiness Summary

- `REQ-READY-001`: The backend must return a compact setup summary for a selected event.
- `REQ-READY-002`: The summary must include total participant count, ready count, needs-setup count, missing count, declined count, and `ready_for_planning`.
- `REQ-READY-003`: `ready_for_planning` is true only when at least one merchant is selected and every selected merchant is `confirmed` and `ready`.
- `REQ-READY-004`: The workspace must show this readiness state before planning.

### Planning Guard And Scope

- `REQ-PLAN-001`: Non-demo `POST /api/events/{event_id}/generate-plan` must reject plan generation when the selected event has no ready roster.
- `REQ-PLAN-002`: The rejection must be a visible client error with a concise message such as `event merchant setup required`.
- `REQ-PLAN-003`: Non-demo route planning must pass only confirmed and ready event roster merchants into planning, pack generation, tasks, and planning evidence.
- `REQ-PLAN-004`: Generated `PlanVersion.merchant_assignments` and merchant tasks must not include merchants outside the selected event's ready roster.
- `REQ-PLAN-005`: Event page drafting and merchant edge package generation should use the current plan's merchant scope, not the entire global catalog.
- `REQ-PLAN-006`: `demo-night-tour` remains backward compatible and may keep using the seeded demo catalog when no event roster exists.

### Organizer Workspace UI

- `REQ-UI-001`: `OrganizerEventWorkspacePage` must show a merchant setup panel for the selected event.
- `REQ-UI-002`: The panel must load the global merchant catalog and the selected event's roster.
- `REQ-UI-003`: Organizers can add or remove merchants from the selected event roster using the existing catalog.
- `REQ-UI-004`: Organizers can mark roster merchants as `ready` or `needs_setup`.
- `REQ-UI-005`: The workspace header or metric area must show whether the event is ready for planning.
- `REQ-UI-006`: For a non-demo event that is not ready for planning, the build-plan action must not silently proceed; it should be disabled or show a clear error.
- `REQ-UI-007`: The UI must remain dense and operational, with no landing-page or hero redesign.

### Internationalization

- `REQ-I18N-001`: New visible text must have explicit entries in `en`, `zh-Hans`, and `zh-Hant`.

### Demo Boundary

- `REQ-DEMO-001`: Existing demo seed and release gates must remain deterministic.
- `REQ-DEMO-002`: Existing demo browser/API release gates must not require manual merchant setup.
- `REQ-DEMO-003`: v2.9 tests may create non-demo events and configure their roster explicitly.

## API Shape

### Existing

- `GET /api/merchants`
- `GET /api/events/{event_id}`
- `POST /api/events/{event_id}/generate-plan`

### New

- `GET /api/events/{event_id}/merchant-roster`
  - Returns `EventMerchantSetupSummary`.

- `PUT /api/events/{event_id}/merchant-roster`
  - Request: `{"merchant_ids": ["m001", "m002"]}`
  - Replaces the selected event roster while preserving retained entry status and notes.
  - Returns `EventMerchantSetupSummary`.

- `PATCH /api/events/{event_id}/merchant-roster/{merchant_id}`
  - Request can update `participation_status`, `readiness_status`, `role_hint`, and `notes`.
  - Returns `EventMerchantSetupSummary`.

## Data Model

`EventMerchantParticipant`:

- `event_id: str`
- `merchant_id: str`
- `participation_status: Literal["invited", "confirmed", "declined"]`
- `readiness_status: Literal["missing", "needs_setup", "ready"]`
- `role_hint: str | None`
- `notes: str`
- `created_at: str`
- `updated_at: str`

`EventMerchantSetupSummary`:

- `event_id: str`
- `participants: list[EventMerchantParticipant]`
- `total_count: int`
- `ready_count: int`
- `needs_setup_count: int`
- `missing_count: int`
- `declined_count: int`
- `ready_for_planning: bool`

## Error Handling

- Missing event: `404 event not found`.
- Unknown merchant: `404 merchant not found`.
- Empty non-demo roster at planning time: `400 event merchant setup required`.
- Roster exists but not all selected merchants are ready: `400 event merchant setup required`.
- Invalid status payload: normal FastAPI/Pydantic `422`.

## Test Requirements

- Backend tests for roster create/list/replace/update.
- Backend tests that unknown events and merchants are rejected.
- Backend tests that non-demo plan generation is blocked until the roster is ready.
- Backend tests that generated plan versions and merchant tasks use only selected ready merchants.
- Backend tests that `demo-night-tour` still works without manual roster setup.
- Frontend tests for workspace merchant setup panel, readiness state, and selected-event planning guard.
- Frontend i18n tests must continue passing.
- Existing v2.7/v2.8 tests must continue passing.

## Acceptance Criteria

v2.9 is complete when:

1. A non-demo event can be created, assigned participating merchants, marked ready, and planned.
2. A non-demo event without a ready roster cannot generate a route plan.
3. The generated plan and merchant tasks are scoped to the selected ready roster.
4. The organizer workspace shows merchant setup and planning readiness.
5. The deterministic demo path still works.
6. Backend focused tests, frontend focused tests, full relevant regressions, build, repo hygiene, and whitespace checks pass.

## Follow-Up Product Packages

After v2.9, the next product package should be `v3.0-merchant-network-pack`, focused on richer merchant profiles, contact/ownership data, operating windows, capability tags, and admin management. v2.9 should not attempt to solve those broader network-management requirements.
