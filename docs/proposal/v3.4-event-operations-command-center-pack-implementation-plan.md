# v3.4 Event Operations Command Center Pack Implementation Plan

Status: implemented on 2026-06-30.

> **For agentic workers:** Use `module-implementer` for scoped code changes and `verification-before-completion` before claiming completion. Use TDD for backend and frontend behavior.

**Goal:** Add a read-only event operations command center that summarizes launch readiness, blockers, warnings, next actions, and event action evidence for organizers.

**Architecture:** Reuse existing persisted records. Add one backend service to compute operations summary from `EventSummary`, event merchant setup, `PlanVersion`, `EventPage`, `MerchantInteractionPackage`, `Incident`, `PublicNotice`, and `AuditLog`. Expose a single organizer endpoint and render it in a new organizer workspace panel.

**Tech Stack:** FastAPI, Pydantic, SQLite-backed `MVPStore`, pytest, React/Vite, Vitest, local i18n dictionaries.

- [x] **Step 5: Add API endpoint**

```python
@app.get("/api/events/{event_id}/operations-summary")
def event_operations_summary(event_id: str, user: AuthUserRecord = Depends(require_organizer)):
    try:
        return build_event_operations_summary(STORE, event_id)
    except LookupError as exc:
        raise HTTPException(status_code=404, detail=str(exc)) from exc
```

## Implementation Notes

- Keep the summary read-only.
- Do not infer readiness from UI state; backend summary is the source of truth.
- Keep failed checks operator-readable and evidence-linked.
- Keep demo compatibility even if demo lacks explicit merchant setup records.
- Do not introduce external provider credentials or network calls in v3.4.
