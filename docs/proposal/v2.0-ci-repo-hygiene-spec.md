# v2.0 CI And Repository Hygiene Spec

Date: 2026-06-27

Status: accepted for implementation.

## Purpose

v2.0 turns the v1.9 commercial-readiness audit's repository-level P0 gaps into enforceable checks. The goal is not to harden product behavior. The goal is to make every future PR or push answer four basic questions automatically:

1. Do API tests pass?
2. Do web tests and production build pass?
3. Did the change introduce obvious secret or local-path leakage?
4. Did the change accidentally include generated screenshot artifacts or dependency folders?

## Scope

### In Scope

- Add a root GitHub Actions workflow under `.github/workflows/`.
- Add a small dependency-free repository hygiene script under `scripts/`.
- Make the hygiene script runnable both locally and in CI.
- Update docs and evidence so future agents know how to reproduce the checks.
- Keep the v1.9 historical PNG byproducts out of the v2.0 commit.

### Out Of Scope

- No product feature changes.
- No QwenPaw productization.
- No production auth/session/CSRF hardening.
- No database migration or persistence rewrite.
- No deployment provider setup.
- No observability stack.
- No real merchant, POS, payment, hardware, map, weather, traffic, or identity integration.

## Required Behavior

### CI Workflow

The workflow must run on pull requests and pushes to `main` and `v1-qwen-controlled-draft`.

It must contain three jobs:

| Job | Required commands |
| --- | --- |
| API | `python -m pip install -e ".[dev]"`, then `python -m pytest -q` in `apps/api`. |
| Web | `npm ci`, `npm run test`, then `npm run build` in `apps/web`. |
| Hygiene | `python scripts/repo_hygiene.py --base "$BASE_SHA"` from the repository root. |

The workflow should use Ubuntu runners, Python 3.11, and Node 20.

### Hygiene Script

The script must be dependency-free and use only Python standard library modules.

It must support these checks:

| Check | Meaning |
| --- | --- |
| `secrets` | Scan tracked text files for high-confidence API key, bearer token, and provider-key assignment patterns. |
| `local-paths` | Scan tracked text files for local absolute paths such as user home directories or local workspace roots. |
| `node-modules` | Fail if any tracked path contains a `node_modules` segment. |
| `generated-artifacts` | Fail if changed files relative to the provided base include generated screenshot PNG evidence under `docs/research/assets/`. |

The script must:

- Default to all checks.
- Accept `--check` repeatedly for focused runs.
- Accept `--base` for changed-file comparison.
- Avoid scanning binary files as text.
- Print a concise pass/fail summary.
- Exit `0` only when all selected checks pass.
- Exit non-zero with actionable file paths when a check fails.

### PNG Policy

v2.0 does not decide whether the 14 historical screenshot PNG diffs should be committed or restored. Those are tracked binary files that were modified by earlier verification runs. Because changing them would either create binary review noise or destroy local evidence, v2.0 must not stage them.

The new hygiene policy is:

- Future generated PNG changes are blocked by the CI changed-file guard unless a separate evidence-refresh task explicitly owns them.
- The current 14 PNG diffs remain visible in `git status` until the user chooses "restore" or "commit as evidence refresh" in a separate task.

## Acceptance Criteria

- `.github/workflows/ci.yml` exists.
- `scripts/repo_hygiene.py` exists and is executable by `python`.
- Local API pytest passes.
- Local web Vitest passes.
- Local web build passes.
- Local hygiene script passes for the committed v2.0 scope.
- Docs record that GitHub Actions will only be proven after push.
- The v2.0 commit contains no historical PNG diffs.
