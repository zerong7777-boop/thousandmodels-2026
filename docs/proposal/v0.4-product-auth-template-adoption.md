# Zhiyin Haojiang v0.4 Product Auth and Template Adoption Spec

Date: 2026-06-10
Project root: `<PROJECT_ROOT>`
Status: design spec for next phase

## 1. Purpose

v0.4 corrects the main weakness exposed after v0.3 Task 6-9: the current frontend has role routes and shells, but the login is still a demo identity switcher and the visual system still looks like a lightly modified management backend.

The next phase must build a credible product foundation:

- Product-grade organizer, merchant, and tourist endpoints.
- Backend-backed demo accounts stored in SQLite.
- Login through FastAPI auth APIs, not localStorage-only identity switching.
- Real template adoption from open-source frontend templates or blocks, not only visual reference.
- Preserved deterministic demo loop for event orchestration.

This spec replaces the v0.3 localStorage demo-auth direction as the main path. During implementation, the old localStorage session may exist only in tests or migration shims. v0.4 is not accepted until product routes use backend auth as the source of truth.

## 2. Current Baseline

The project currently has:

- FastAPI backend in `apps/api`.
- SQLite-backed store through `apps/api/app/store.py`.
- React 19 + Vite + TypeScript frontend in `apps/web`.
- Ant Design 6 and lucide-react installed.
- v0.2 deterministic backend loop:
  - seed demo event
  - generate `PlanVersion v1` with `AgentTrace`
  - approve v1
  - merchant runtime update
  - create `Incident`
  - create and approve `RecoveryProposal`
  - generate `PlanVersion v2`
  - public event projection
  - review report
- v0.3 Task 6-9 frontend foundation:
  - `/login`
  - `OrganizerShell`
  - `MerchantShell`
  - `UserShell`
  - route guards
  - localStorage demo session

The baseline is useful for routing and deterministic flow, but not sufficient as a product.

## 3. Non-Negotiable Direction Changes

### 3.1 Login Must Stop Being a Toy

The v0.4 login must be a product login screen backed by the API and database.

Allowed:

- Username/password form.
- Demo account shortcuts that auto-fill credentials.
- Server-created session cookie or token.
- Logout through API.
- `/auth/me` style session check.

Not allowed as the main path:

- Clicking a role card and directly writing `localStorage`.
- Pretending route guard equals authentication.
- Treating frontend-only state as the source of truth.

### 3.2 All Roles Use Demo Accounts For Now

There is no public self-registration in v0.4.

The backend seed must create demo accounts:

| Role | Username | Password | Binding |
| --- | --- | --- | --- |
| Organizer | `organizer.demo` | `demo1234` | demo organizer |
| Merchant | `merchant.m001.demo` | `demo1234` | merchant `m001` |
| Tourist | `tourist.demo` | `demo1234` | demo tourist |

The UI may show these accounts through a "Use demo account" control, but the final login action must still call the backend auth API.

### 3.3 Registration Is Deferred, Not Forgotten

Do not implement open self-registration in v0.4.

The product should reserve room for later registration:

- Tourist self-registration can be added later.
- Merchant onboarding can become an application and approval flow later.
- Organizer accounts should remain operator-provisioned.

If a `/register` route exists in v0.4, it should be a disabled or informational route saying demo accounts are assigned by the system. It must not create arbitrary users.

### 3.4 Template Reuse Must Be Concrete

The frontend must stop treating templates as vague inspiration.

v0.4 must include a template adoption spike before implementation:

- Clone or otherwise inspect runnable candidate templates.
- Verify license and dependency cost.
- Run or build the selected candidate locally. If the selected candidate cannot be run or built, reject it or document a replacement candidate before implementation.
- Screenshot desktop and mobile views.
- List exact reusable assets, files, components, or layout patterns.
- Decide which template code will be copied/adapted.

Recommended primary direction:

```text
satnaing/shadcn-admin
+ official shadcn/ui blocks
+ existing Ant Design only as migration bridge
```

Backup candidate:

```text
TailAdmin React
```

Reference-only:

```text
Tremor for review metrics
Mantine if shadcn/Tailwind adoption is rejected later
```

The implementation plan must not say "reference layout" when actual reusable code is available.

### 3.5 Product Routes Must Not Be Demo Operator Consoles

The deterministic demo loop must remain runnable, but product routes must not expose demo-operator controls as normal user actions.

Not allowed inside organizer, merchant, tourist, or public product routes:

- "Seed demo event" as a primary product action.
- "Generate v1 plan" phrased as a demo control instead of a business action.
- "Simulate merchant sold out" inside organizer exception pages.
- "Open merchant workbench" from organizer pages as a cross-role shortcut.
- "Open tourist H5" embedded as a preview inside organizer pages.

If the demo still needs one-click orchestration, put it in a clearly separated dev/demo operator surface, test fixture, or script. The product surface should show business actions and current state, not test-driver controls.

## 5. Auth Architecture

### 5.1 Backend Data Model

Add first-class SQLite tables. Do not store auth users or sessions as generic JSON records in the existing `records` table.

`users`:

| Column | Type | Notes |
| --- | --- | --- |
| `user_id` | TEXT PRIMARY KEY | stable id, e.g. `usr_org_demo` |
| `username` | TEXT UNIQUE NOT NULL | login name |
| `password_hash` | TEXT NOT NULL | PBKDF2-HMAC hash with per-user salt |
| `role` | TEXT NOT NULL | `organizer`, `merchant`, `tourist` |
| `display_name` | TEXT NOT NULL | UI display name |
| `merchant_id` | TEXT NULL | set for merchant users |
| `status` | TEXT NOT NULL | `active`, `disabled` |
| `created_at` | TEXT NOT NULL | ISO timestamp |

`sessions`:

| Column | Type | Notes |
| --- | --- | --- |
| `session_id` | TEXT PRIMARY KEY | stable id, not sent to the browser |
| `token_hash` | TEXT UNIQUE NOT NULL | hash of the opaque browser cookie token |
| `user_id` | TEXT NOT NULL | references `users.user_id` |
| `created_at` | TEXT NOT NULL | ISO timestamp |
| `expires_at` | TEXT NOT NULL | ISO timestamp |
| `revoked_at` | TEXT NULL | set on logout |

Required table behavior:

- `users.username` must be unique.
- `sessions.user_id` must reference `users.user_id`.
- `sessions.token_hash` must be indexed or unique.
- Demo account seed must be idempotent.
- `clear_demo()` must not accidentally delete auth users or active sessions unless the implementation plan explicitly defines a test-only reset path.

Password hashing may use Python standard library `hashlib.pbkdf2_hmac` to avoid adding dependencies in the first pass. The stored password hash must include algorithm, iteration count, salt, and hash value. Each user must get a per-user random salt. Password comparison must use constant-time comparison. Login failure must return a generic invalid-credentials error and must not reveal whether the username exists.

### 5.2 Backend Auth Endpoints

Add endpoints:

```text
POST /api/auth/login
POST /api/auth/logout
GET  /api/auth/me
```

`POST /api/auth/login` request:

```json
{
  "username": "organizer.demo",
  "password": "demo1234"
}
```

Success response:

```json
{
  "user": {
    "user_id": "usr_org_demo",
    "username": "organizer.demo",
    "role": "organizer",
    "display_name": "Organizer Demo",
    "merchant_id": null
  }
}
```

Session transport:

- Preferred for v0.4: HTTP-only cookie named `zhiyin_session`.
- Cookie attributes for local demo: `HttpOnly`, `SameSite=Lax`, `Path=/`.
- Expiration: 8 hours for demo.

`GET /api/auth/me` returns the current user or `401`.

`POST /api/auth/logout` revokes the session and clears the cookie.

Demo account seeding must not be a normal public frontend path. Prefer startup seed, test fixture, or CLI/script seed. If an HTTP endpoint is implemented for local development, it must be disabled outside `APP_ENV=local`, must require a development secret, and the frontend must not call it.

### 5.3 Frontend Auth State

Frontend auth must treat `/api/auth/me` as the source of truth.

Allowed frontend storage:

- In-memory React state for current user.
- Optional non-sensitive localStorage cache for last selected username.

Not allowed:

- Storing role as authoritative localStorage session.
- Storing password.
- Storing server session token in localStorage if using cookie sessions.

### 5.4 Route Guards

Route ownership:

| Route | Required role | Notes |
| --- | --- | --- |
| `/login` | public | redirects authenticated users to role home |
| `/organizer/*` | organizer | protected |
| `/review/*` | organizer | compatibility route |
| `/merchant/*` | merchant | protected |
| `/user/*` | tourist | protected |
| `/public/events/*` | public | no login required |

Unauthenticated protected route behavior:

- Redirect to `/login`.
- Preserve intended route in `next` query parameter.
- `next` must be a same-origin relative path. Reject external URLs, protocol-relative URLs, and paths not allowed for the final user role.

Wrong-role behavior:

- Redirect to that user's default role home.
- Show a short "This account cannot access that workspace" notice.

### 5.5 Backend Authorization Boundary

Frontend route guards are not enough. Backend APIs must enforce role and owner boundaries.

Default rule:

- Every non-public business API must require `current_user`.
- Public APIs are limited to:
  - `GET /api/health`
  - `POST /api/auth/login`
  - `POST /api/auth/logout`
  - `GET /api/auth/me`
  - `GET /api/public/events/{event_id}`

Role rules:

| API area | Organizer | Merchant | Tourist | Public |
| --- | --- | --- | --- | --- |
| events list and event summary | read/write demo event operations | no | no | no |
| generate plan and approve plan | yes | no | no | no |
| plan versions and diffs | yes | no | no | no |
| agent traces | yes | no | no | no |
| incidents | yes | no | no | no |
| recovery proposals | yes | no | no | no |
| audit logs | yes | no | no | no |
| merchant workbench | organizer may inspect by event | own merchant only | no | no |
| merchant runtime update | no direct product action | own merchant only | no | no |
| public event projection | yes | yes, read only if needed | yes | read only |
| review report | yes | no | no | no |

Merchant ownership rules:

- Merchant identity comes from the authenticated session, not from a freely editable frontend route.
- If a route or API contains `merchant_id`, it must match `current_user.merchant_id` for merchant users.
- A merchant user attempting to read or write another merchant's workbench or runtime state must receive `403`.

Session rules:

- Browser cookie stores an opaque random token with at least 32 bytes of entropy.
- Database stores only the token hash, never the raw token.
- Every authenticated request checks token hash, `expires_at`, `revoked_at`, and user `status`.
- Disabled users cannot login, and existing sessions for disabled users must be rejected by `/api/auth/me`.
- Logout revokes the session and clears the cookie with matching `Path` and `SameSite` attributes.

Mutation request boundary:

- CORS allowed origins must come from configuration.
- State-changing requests must validate `Origin` or `Referer`, or require a lightweight CSRF header. v0.4 does not need a full enterprise CSRF framework, but it must not leave cookie-authenticated mutation endpoints completely unchecked.

Audit rules:

- Audit actor fields must come from `current_user.user_id` and `current_user.role`.
- Hard-coded actors such as `demo-organizer` are not accepted for authenticated v0.4 product flows.

## 7. Frontend Experience Requirements

### 7.1 Login Page

The login page must look like a product entry screen, not a role switcher.

Required elements:

- Product name: `Zhiyin Haojiang`.
- Short subtitle describing old-district event orchestration.
- Username field.
- Password field.
- Login button.
- Demo account quick-fill controls for organizer, merchant, tourist.
- Clear note that this is demo access for the competition prototype.

Login actions:

- Quick-fill only fills credentials.
- Final login calls `POST /api/auth/login`.
- After login, call or update `/api/auth/me` state.
- Redirect to role default route.
- The page must not contain the text `Choose demo identity`, `Demo-only local session`, or `Switch identity`.

### 7.2 Organizer Shell

The organizer shell should be adapted from the selected product shell template.

It should include:

- Event switcher or current event label.
- Operations navigation.
- Approval and incident badges.
- User menu with logout.
- Workspace title and contextual action area.

The organizer shell is an operations product shell. It must not provide direct role-switch shortcuts to merchant or tourist workspaces as primary navigation.

### 7.3 Merchant Shell

The merchant shell should not reuse the full organizer navigation.

It should include:

- Merchant name.
- Today's event context.
- Task/status/notice tabs or compact navigation.
- Large status action area.
- Logout.

The merchant shell must be tablet/mobile friendly. At widths around `1024x768` and below, it must use compact task/status/notice navigation and large touch-friendly status controls, not a desktop back-office sidebar.

### 7.4 Tourist Shell

The tourist shell should be mobile-first.

It should include:

- Event title.
- Route progress or route status.
- Notices.
- Bottom navigation or compact top navigation.
- Logout for logged-in tourist route.

The public route can use the same visual components but must not require auth.

The tourist route must not display `Tourist H5`, `Route version`, `PlanVersion`, `Incident`, `RecoveryProposal`, or `AgentTrace`.

### 7.5 Role Contract Matrix

| Role surface | Allowed reads | Allowed writes | Must not show | Must not do |
| --- | --- | --- | --- | --- |
| Organizer product | events, plan versions, evidence, incidents, recovery proposals, review reports, merchant status summaries | generate plan, approve plan, approve recovery, generate review | merchant-only status controls, embedded tourist H5 | directly mutate merchant inventory or queue as a product action |
| Merchant product | own merchant profile, own tasks, own runtime state, organizer notices | own runtime status update | organizer approvals, AgentTrace, RecoveryProposal, internal incident queue | access another merchant's workbench or act as organizer |
| Tourist product | visitor-safe event projection, route points, notices, own route progress if implemented | visitor task/progress actions if implemented | PlanVersion, Incident, RecoveryProposal, AgentTrace, merchant operational internals | access merchant or organizer APIs |
| Public event | visitor-safe event projection only | none | login-only personal state, internal backend terms | access protected APIs |

### 7.6 Route-Specific Page Requirements

Core routes must not all point to one generic component with only `activeKey` changes.

Minimum requirements:

- `/merchant/events/demo-night-tour/tasks`, `/merchant/events/demo-night-tour/status`, and `/merchant/notifications` must have route-specific content, empty states, and tests.
- `/user/events/demo-night-tour` and `/public/events/demo-night-tour` may share low-level route cards, but the logged-in user route must have a user shell and at least one authenticated-user affordance or state placeholder. The public route must remain visitor-safe and unauthenticated.
- `/organizer/events/demo-night-tour/exceptions` must not contain a normal product button that simulates merchant inventory. Incident triggers should come from merchant runtime updates or a separated demo operator tool.

## 9. Tests

Add tests for:

- Demo account seed creates organizer, merchant, and tourist users.
- Passwords are hashed and raw password is not stored.
- Login succeeds with demo credentials.
- Login fails with wrong password.
- `/api/auth/me` returns current user with valid session.
