# v3.2 Route Assembly Quality Pack Spec

Status: implemented.

Implemented: 2026-06-29.

## Goal

v3.2 makes generated route plans use route-point evidence instead of slicing the first six catalog points. The planner should score candidate route points against the event brief and selected merchants, assemble a deterministic route sequence, and warn organizers when route coverage is thin.

This phase keeps the current local catalog, deterministic planning path, v2.9 merchant roster gate, v3.0 merchant eligibility gate, v3.1 merchant-fit evidence, and human approval boundary intact.

## Background

v2.7 introduced non-demo event lifecycle APIs.
v2.8 exposed organizer event creation and selected-event workspace navigation.
v2.9 added event-level merchant roster setup and selected-merchant planning scope.
v3.0 added organizer-managed merchant profiles and eligibility checks.
v3.1 made non-demo merchant selection fit-ranked and explainable.

The remaining planning gap is route assembly. `generate_plan_version()` still builds `route_points` from `route_points[:6]`, then filters `linked_merchants` to selected merchants. This can leave selected merchants without route linkage, include weakly related route points, and gives organizers no route-quality evidence.

## Scope

v3.2 delivers a Route Assembly Quality Pack:

1. Deterministic route-point scoring for generated plan versions.
2. Route assembly that covers selected merchants where linked route points exist.
3. Explainable route-point roles and rationales in `PlanVersion`.
4. Route warnings for missing merchant linkage, thin coverage, weak route fit, high crowd risk, and rainy-day constraints.
5. Planning Agent route-step evidence that records route assembly output.
6. Organizer workspace visibility for route assembly rationale and warnings.
7. Regression coverage that v2.9/v3.0/v3.1 gates and the deterministic `demo-night-tour` path remain compatible.

## Non-Goals

- No external map, geocoding, walking-distance, routing, weather, traffic, POS, payment, CRM, ERP, visitor identity, or hardware integrations.
- No route geometry optimization.
- No live Qwen, QwenPaw, DashScope, or model-based planning.
- No automatic plan approval or public publication.
- No public merchant onboarding.
- No event archive/delete/clone workflow.
- No merchant contract, settlement, compliance, or accounting workflow.

## Product Requirements

### Route-Point Scoring

- `REQ-ROUTE-001`: The backend must compute deterministic route-fit evidence for route points in the local catalog.
- `REQ-ROUTE-002`: Route scoring must consider:
  - event area;
  - event time window;
  - event target audience, goal, theme preferences, constraints, and priority rules;
  - route point type, story, visitor task, indoor/outdoor status, estimated stay minutes, rainy-day score, crowd risk, current status, and linked merchants;
  - selected plan merchants from v3.1 merchant-fit output.
- `REQ-ROUTE-003`: Route scoring must produce:
  - `point_id`;
  - `score`;
  - `fit_level` as `strong`, `medium`, or `weak`;
  - `role` as a deterministic route role;
  - `linked_selected_merchants`;
  - `matched_signals`;
  - `warnings`;
  - `rationale`.
- `REQ-ROUTE-004`: Paused route points must not be selected unless no active route points exist, in which case the plan must warn the organizer.
- `REQ-ROUTE-005`: Route scoring must be deterministic and locally testable.

### Route Assembly

- `REQ-ASSEMBLY-001`: `PlanVersion.route_points` must be selected through route assembly, not raw `route_points[:6]`.
- `REQ-ASSEMBLY-002`: The assembled route must prefer route points linked to selected plan merchants.
- `REQ-ASSEMBLY-003`: The assembled route must include at most six route points.
- `REQ-ASSEMBLY-004`: The assembled route should include one suitable route point per selected merchant where the catalog has linkage.
- `REQ-ASSEMBLY-005`: The assembled route should fill remaining slots with the strongest route-fit points.
- `REQ-ASSEMBLY-006`: The final sequence must remain stable and deterministic. In the absence of real geometry, selected points should be ordered by local catalog order after scoring chooses the set.
- `REQ-ASSEMBLY-007`: `linked_merchants` must remain scoped to selected plan merchants.
- `REQ-ASSEMBLY-008`: Existing public event projection, event-page draft, merchant-edge packages, operation suggestions, and review flows must continue to read `PlanVersion.route_points`.

### Plan Output

- `REQ-PLAN-001`: `PlanVersion` must expose route assembly evidence with backward-compatible defaults.
- `REQ-PLAN-002`: `PlanVersion` must expose route warnings separately from merchant-fit warnings.
- `REQ-PLAN-003`: Existing stored plan versions without route assembly fields must continue to validate.

### Agent Evidence

- `REQ-AGENT-001`: Planning Agent route step must include route assembly evidence.
- `REQ-AGENT-002`: The route step must explain why route points were selected and how merchant coverage was handled.
- `REQ-AGENT-003`: Agent evidence must remain deterministic and must not claim live map, weather, traffic, or model reasoning.

### Organizer UI

- `REQ-UI-001`: Organizer workspace must show route assembly warnings when they exist.
- `REQ-UI-002`: Organizer workspace must show route-point role, fit level, score, rationale, matched signals, and warnings for the current plan.
- `REQ-UI-003`: UI copy must stay operations-focused and dense.
- `REQ-UI-004`: New visible copy must use the existing i18n dictionary layer.

### Compatibility

- `REQ-COMPAT-001`: `demo-night-tour` remains deterministic and does not require manual roster setup.
- `REQ-COMPAT-002`: v2.9 roster readiness remains the hard planning gate for non-demo events.
- `REQ-COMPAT-003`: v3.0 merchant eligibility remains the hard ineligible gate before fit scoring.
- `REQ-COMPAT-004`: v3.1 merchant-fit ranking remains authoritative for selected merchant assignment order.
- `REQ-COMPAT-005`: Existing route point consumers keep receiving `RoutePoint` objects.

## Proposed Data Model

### RouteFitResult

- `point_id: str`
- `score: int`
- `fit_level: Literal["strong", "medium", "weak"]`
- `role: Literal["start", "story", "merchant_stop", "rest", "backup", "finish"]`
- `linked_selected_merchants: list[str]`
- `matched_signals: list[str]`
- `warnings: list[str]`
- `rationale: str`

### PlanVersion Additions

- `route_fit: list[RouteFitResult] = []`
- `route_warnings: list[str] = []`

Defaults preserve compatibility for existing stored plan versions.

## Deterministic Route Scoring Rules

The exact numbers may be tuned during implementation, but the scoring contract must stay stable and understandable:

1. Start each active route point at 40 points.
2. Add points for linked selected merchants.
3. Add points for event-token matches against route point name, type, story, and visitor task.
4. Add points for rainy-day score when the event mentions rain, indoor, covered, backup, family, or rest constraints.
5. Add points for indoor points when the event constraints or priority rules mention indoor, covered, rain, backup, or rest.
6. Add points when estimated stay is inside an operational range for a guided route, with penalties for very long stays.
7. Subtract points for high crowd risk, paused status, or no selected merchant linkage.
8. Mark `strong` at 75 or above, `medium` from 55 to 74, and `weak` below 55.
9. Produce warnings for no selected merchant linkage, weak fit, high crowd risk, paused status, missing rainy-day backup, and selected merchants without any linked route point.

## Deterministic Assembly Rules

1. Score every candidate route point.
2. For each selected merchant in `PlanVersion.merchant_assignments` order, choose the highest-scoring unused route point linked to that merchant.
3. Add the strongest unused route points until the route has at most six points.
4. Preserve local catalog order for the selected point set to avoid fake geometry claims.
5. Filter each route point's `linked_merchants` to selected plan merchants.
6. Attach all route fit evidence and warnings to the plan.

This is route-quality assembly, not walking-distance optimization.

## API Behavior

No new public endpoint is required.

Existing enhanced behavior:

- `POST /api/events/{event_id}/generate-plan`
  - Builds merchant scope through v2.9/v3.0 gates.
  - Computes v3.1 merchant-fit assignments.
  - Computes v3.2 route-point fit evidence.
  - Stores selected route points, route fit evidence, route warnings, merchant fit evidence, and planner warnings on `current_plan`.
  - Returns the enhanced `current_plan`.

- `GET /api/events/{event_id}/plan-versions`
  - Returns stored plan versions with route assembly evidence when available.

## Error Handling

- If route catalog is empty, existing local catalog seeding remains the fallback in `POST /api/events/{event_id}/generate-plan`.
- If no active route points exist, use the best available non-active points and include a route warning.
- If selected merchants have no linked route point, planning may proceed but must include route warnings.
- If all route points are weak but available, planning may proceed but must include route warnings.
- If old plan records lack v3.2 fields, response validation must use defaults and not fail.

## Test Requirements

- Backend unit tests for deterministic route scoring:
  - merchant-linked route points outrank unlinked weak route points;
  - rainy/covered event constraints favor indoor and high rainy-day score points;
  - high crowd-risk and unlinked points produce warnings.
- Backend assembly tests:
  - generated plans cover linked selected merchants where possible;
  - selected route points are not just raw catalog order when stronger linked points appear later;
  - `linked_merchants` remains scoped to selected plan merchants;
  - selected merchants without route linkage produce route warnings.
- Backend Agent evidence tests:
  - planning route step includes `route_fit` and `route_warnings`.
- Backend regressions:
  - v2.9 roster gate still blocks incomplete setup;
  - v3.0 ineligible merchants remain blocked;
  - v3.1 merchant assignment fit order remains intact;
  - `demo-night-tour` can still generate a plan without manual roster setup.
- Frontend tests:
  - organizer workspace renders route assembly warnings and rationales.
- Full verification:
  - backend pytest;
  - frontend Vitest;
  - frontend build;
  - repo hygiene;
  - whitespace check.

## Acceptance Criteria

v3.2 is complete when:

1. Generated plan route points are selected through deterministic route assembly.
2. Route assembly prioritizes selected merchant coverage where route linkage exists.
3. Route points remain scoped to selected plan merchants.
4. Generated plan versions include route fit evidence and route warnings.
5. Planning Agent route step includes route assembly evidence.
6. Organizer workspace exposes route assembly rationale and warnings.
7. v2.9, v3.0, v3.1, and deterministic demo regressions pass.
8. Full backend/frontend/build/hygiene/whitespace verification passes.
