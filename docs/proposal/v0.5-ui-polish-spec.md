# Zhiyin Haojiang v0.5 UI Polish Spec

Date: 2026-06-11
Project root: `<PROJECT_ROOT>`
Status: design spec for next phase

## 1. Purpose

v0.4 solved the main structural problem: the project now has backend-backed demo auth, role-separated routes, deterministic event orchestration, and screenshot-backed organizer, merchant, tourist, and public surfaces.

v0.5 should not add more backend workflow, model integration, or new product scope. Its purpose is to make the current product skeleton visually credible enough for a competition demo:

- The organizer side should feel like an event operations product, not a sparse admin dashboard.
- The merchant side should feel like a mobile/tablet task app, not a backend form.
- The tourist/public side should feel like a real cultural route H5, not a backend preview card list.
- The login page should feel like a product entry point, not only a demo account utility.
- The UI should show complete states, not only the happy path.

The output of v0.5 must be screenshot-driven. The phase is accepted only if the refreshed Playwright screenshots demonstrate a visible improvement over the current v0.4 screenshots.

## 3. Current UI Problems

### 3.1 Cross-Surface Problems

The v0.4 UI is structurally separated but still too plain. It uses clean cards and role shells, but the result is closer to a functional wireframe than a product MVP.

Problems to fix:

- Too much empty space with too little operational context.
- Cards look generic and interchangeable across roles.
- Status labels are visible but weak; they do not drive attention or decisions.
- Many pages lack a clear primary workflow path.
- Loading, empty, error, success, pending, and completed states are not designed as first-class states.
- The interface has little Macau/cultural-route specificity beyond text content.
- The visual hierarchy is not strong enough for a judge to understand the product story within seconds.

### 3.2 Organizer Problems

Observed from `02-organizer-dashboard.png`, `03-organizer-workspace.png`, `04-organizer-exceptions.png`, and `05-organizer-review.png`:

- The dashboard is too sparse for an operations product.
- It shows metrics but not enough next actions, risk context, or timeline.
- The event workspace has the right objects but reads like a card dump.
- The exception center has the right flow but lacks impact scope, proposal comparison, and confirmation state.
- The review page is functional but does not feel metric-backed enough.
- Sidebar and header are clean but not distinctive.

### 3.3 Merchant Problems

Observed from `06-merchant-status-mobile.png`:

- The merchant page is usable but still form-like.
- It asks the merchant to understand inventory/queue fields instead of presenting direct operational choices.
- The "Report sold out" action works but is visually just another button.
- The post-submit result is too small and easy to miss.
- The current status area does not feel like a live event workbench.

### 3.4 Tourist/Public Problems

Observed from `07-tourist-route-mobile.png` and `08-public-h5-mobile.png`:

- The route is presented as plain cards, not as a cultural walk.
- The public H5 lacks event atmosphere and shareable visual identity.
- There is no route progress rhythm beyond "Stop 1 / Stop 2".
- Visitor notices are clear but not designed like live route guidance.
- The current route version/update state is technically visible but not emotionally or behaviorally strong.

### 3.5 Login Problems

Observed from `01-login.png`:

- Login is clean, but still reads like a demo utility.
- Demo account cards take a lot of space without adding product trust.
- The product purpose is too abstract; it should tell users which workspace they are entering.
- The visual treatment is not aligned with the role-specific product surfaces.

## 4. v0.5 Design Principles

### 4.1 Product, Not Admin

Organizer can be dense and operational, but it must not become a generic admin dashboard. Every visible block should answer one of these product questions:

- What event is running?
- What changed?
- What needs approval?
- What is at risk?
- What has been communicated to merchants and visitors?
- What evidence supports the decision?

### 4.2 Role-Specific, Not Shared Skin

The same primitives may be reused, but each role must feel different:

- Organizer: operations console, decision queue, evidence, approvals.
- Merchant: today's task app, large status actions, notices.
- Tourist/public: mobile cultural route, live guidance, route story.

### 4.3 Screenshot-First Acceptance

Unit tests prove behavior. Screenshots prove product credibility. v0.5 must compare before/after screenshots and record visible improvements in `docs/ai/VERIFY.md`.

### 4.4 Deterministic Demo Remains The Required Path

No UI polish may make model APIs, real merchants, real maps, hardware, or Qwen/QwenPaw required for the local demo.

### 4.5 Complete States Matter

The product must show credible states for:

```text
loading
empty
error
draft
pending approval
approved
incident open
proposal ready
recovery confirmed
plan v2 published
merchant update submitted
public notice published
```

## 5. Scope

### 5.1 In Scope

v0.5 includes:

- Screenshot-driven visual audit of current v0.4 screens.
- Focused template/block adoption pass based on the approved shadcn/ui-inspired direction.
- Stronger role-specific layout and visual hierarchy.
- More complete UI states.
- Mobile-first polish for merchant and tourist/public surfaces.
- Better product copy where current labels feel internal or generic.
- Playwright screenshot refresh and manual screenshot review.
- Frontend tests for visual/role boundaries where practical.
- Documentation updates in `docs/ai`.

### 5.2 Out Of Scope

v0.5 must not include:

- Qwen/QwenPaw required demo path.
- Model API integration as a required workflow.
- Backend orchestration redesign.
- New database domain model unless required for frontend state display and explicitly approved.
- Real merchant onboarding.
- Open public registration.
- Hardware, AI glasses, IoT, robotics, or real traffic prediction.
- Payment/POS integrations.
- Real map API.
- Marketing landing page.
- Full SaaS tenant/user-management system.

## 6. Visual Direction

### 6.1 Primary Direction

Continue with:

```text
shadcn/ui-inspired product interface
+ owned Tailwind primitives
+ role-specific product shells
+ Ant Design only as migration bridge
```

The next pass should make this direction visible in the product, not only present in file structure.

### 6.2 Template Adoption Upgrade

v0.4 adopted a shadcn/ui-inspired local component direction, but the result is still closer to a clean wireframe than a product interface. v0.5 must move from primitive-level reuse to page-level pattern adoption.

This does not mean importing a full admin product or applying a generic template skin. It means selecting concrete blocks, layout patterns, interaction patterns, and visual hierarchy examples that map to the exact Zhiyin Haojiang pages.

Before implementation, create:

```text
docs/research/v0.5-ui-template-adoption.md
```

This document may reference `docs/research/v0.5-ui-polish-reference.md`, but it must stand on its own as the implementation-facing template adoption decision.

Minimum template/reference categories:

| Category | Required mapping target |
| --- | --- |
| Organizer operations dashboard | Dashboard layout, metric strip, attention queue, activity timeline |
| Organizer exception/recovery workflow | Incident queue, selected detail, before/after diff, approval panel |
| Organizer review metrics | KPI blocks, chart/report panels, recommendation sections |
| Merchant mobile/tablet workbench | Task cards, large quick actions, status controls, mobile navigation |
| Tourist/public event H5 | Event header, route timeline, stop story cards, notice feed, route progress |
| Login/product entry | Credential form, compact demo account selector, role entry treatment |

Candidate pools to re-check, not pre-approve:

| Candidate pool | Likely use |
| --- | --- |
| shadcn/ui blocks | Primary page/block direction for owned Tailwind components |
| satnaing/shadcn-admin | Organizer dashboard and shell references, only after license and maintenance check |
| TailAdmin React | Backup reference for operational density and dashboard composition |
| Tremor-style metrics | Organizer review and KPI/report references |
| Flowbite or Untitled UI-style mobile blocks | Merchant mobile controls and task cards, only if license allows the intended use |
| Luma, Wanderlog, Roadtrippers, itinerary/event H5 patterns | Reference-only inspiration for visitor route rhythm; do not copy proprietary UI |

For every selected candidate or reference, record:

```text
URL
repo or official source
license and usage notes
last checked date
maintenance signal
demo URL or screenshot path
screenshot note describing the reusable visual pattern
local target route
local target file/component
copy/adapt/reference-only decision
which visible v0.4 screenshot problem it solves
implementation risk
adopt/reference/reject conclusion
```

At least 3 rejected candidates must be listed with concrete rejection reasons, such as unclear license, too generic admin tone, poor mobile behavior, incompatible dependency weight, or visual mismatch with cultural-route product positioning.

### 6.3 Visual Tone

The UI should feel:

- calm and operational for organizer
- direct and touch-friendly for merchant
- warmer and more cultural for tourist/public
- modern and clean without looking like a generic admin template

Avoid:

- one-note teal/green across the whole app
- generic admin dashboard composition
- decorative orbs, vague gradients, or marketing hero sections
- oversized hero typography inside operational tools
- nested card-in-card layouts
- text-heavy instructional panels explaining how to use the app

### 6.4 Template Adoption Gates

No v0.5 UI implementation should start until the template adoption document has:

1. Compared at least 10 concrete template/block candidates or visual references.
2. Mapped at least 8 concrete blocks or patterns to local routes and files.
3. Selected 1 primary page-level direction and at most 1 backup direction.
4. Identified which parts are copied, adapted, or used as reference-only.
5. Listed at least 3 rejected candidates and why they are unsuitable.
6. Included screenshots, local screenshot paths, or precise screenshot notes.
7. Explained how each selected pattern fixes a visible problem in the v0.4 screenshots.
8. Confirmed license and dependency risks before copying any code.

If a candidate's license is unclear, it may only be used as visual reference until the license is resolved.

### 6.5 Design Review Rubric

Every v0.5 before/after screenshot review should score these dimensions from 1 to 5:

| Dimension | What to check |
| --- | --- |
| Product credibility | Does the page look like a real product, not a competition display page? |
| Role differentiation | Does organizer, merchant, and visitor UI feel meaningfully different? |
| Information density | Is the page dense enough for its role without becoming cluttered? |
| Visual hierarchy | Can the user identify status, risk, and next action within seconds? |
| Mobile ergonomics | Are touch targets, spacing, and reading flow credible on mobile? |
| Cultural/event atmosphere | Do visitor-facing pages feel like a Macau cultural route, not backend data? |
| Low backend leakage | Are internal terms hidden from merchant, tourist, and public surfaces? |

The review is not complete if it only says "looks better". It must name what changed, which reference informed it, and what remaining weakness is visible.

### 6.6 Not Enough To Pass

These changes do not count as a successful v0.5 polish pass by themselves:

- Color tweaks only.
- Replacing plain cards with slightly nicer cards.
- Using a template only for sidebar/header shell while page content stays unchanged.
- Adding decorative hero gradients or ornamental backgrounds.
- Making all roles share the same dashboard/card composition.
- Keeping merchant status as a form-first admin page.
- Keeping tourist/public H5 as a backend preview card list.
- Importing a whole admin template without product-specific adaptation.

## 7. Page-Level Requirements

### 7.0 Implementation Priority

v0.5 should prioritize the pages that most strongly affect product credibility.

| Priority | Pages | Reason |
| --- | --- | --- |
| P0 | `/login`, `/organizer/dashboard`, `/organizer/events/demo-night-tour/exceptions`, `/merchant/events/demo-night-tour/status`, `/public/events/demo-night-tour` | These pages define the first impression, operational decision loop, merchant live action, and visitor-facing product quality. |
| P1 | `/organizer/events/demo-night-tour`, `/organizer/events/demo-night-tour/review`, `/merchant/dashboard`, `/merchant/events/demo-night-tour/tasks`, `/merchant/notifications`, `/user/events/demo-night-tour/route`, `/user/events/demo-night-tour/notices` | These pages broaden consistency after the high-impact surfaces are visibly improved. |

P0 pages must receive direct page-level template or reference mapping. P1 pages may reuse the refined components and patterns from P0, but they must still avoid looking like unchanged v0.4 card lists.

### 7.1 Login Page

Route:

```text
/login
```

Goal:

Make login feel like a product entry point with demo access, not a role switcher.

Required changes:

- Use a clearer product entry layout with product identity, short product purpose, and login form.
- Keep username/password fields and demo quick-fill.
- Demo account quick-fill should be visually secondary to the actual sign-in action.
- Show three demo accounts as compact selectable chips or role cards, not oversized generic cards.
- Explain the three roles through concise labels:
  - Organizer workspace
  - Merchant workbench
  - Visitor route
- Do not imply open registration.

Acceptance:

- User can still use all three demo accounts.
- Login remains backend-backed.
- Page does not show `Choose demo identity`, `Switch identity`, or localStorage language.
- Screenshot looks like a product screen, not a test utility.

### 7.2 Organizer Dashboard

Route:

```text
/organizer/dashboard
```

Goal:

Make this the operations home for the current event.

Required layout:

- Event status header with date, time window, area, current plan, and public release state.
- "Needs attention" queue with pending approvals and incident status.
- Operational metric strip with current plan, live exceptions, merchant readiness, public notice state.
- Timeline or activity feed showing recent system/merchant/organizer actions.
- Primary CTA to enter event workspace.
- Secondary CTA to exception center if an incident exists.

Must not:

- Show merchant-only update controls.
- Show tourist H5 as an embedded preview.
- Show generic admin labels that do not map to event operations.

Acceptance:

- A judge can understand within 5 seconds that this is an event operations product.
- Empty dashboard space is reduced.
- The page has a clear next action.

### 7.3 Organizer Event Workspace

Route:

```text
/organizer/events/demo-night-tour
```

Goal:

Make planning and approval feel like a controlled workflow, not disconnected cards.

Required layout:

- Workflow stepper or stage strip:
  - briefing ready
  - route plan generated
  - plan approved
  - merchant tasks active
  - public route published
- Plan version panel with v1/v2 status and diff.
- Evidence trail as a timeline or expandable step list.
- Merchant task readiness section with task status and responsible merchant.
- Route points section with story point, linked merchants, and current status.
- Approval panel that clearly distinguishes "draft", "approved", and "superseded" states.

Acceptance:

- `AgentTrace` remains hidden behind product language such as "Evidence trail".
- Route plan versioning and merchant task readiness are visually connected.
- Main CTA state changes make sense before and after approval.

### 7.4 Organizer Exception Center

Route:

```text
/organizer/events/demo-night-tour/exceptions
```

Goal:

Make incident recovery look like a decision product.

Required layout:

- Incident queue with severity, source, affected merchant, affected route point, and age.
- Selected incident detail panel.
- Recovery suggestion panel with:
  - recommended change summary
  - affected visitors/merchant scope
  - before/after route impact
  - public notice preview
- v1 -> v2 diff block with clear visual comparison.
- Confirmation panel that shows what will be published after approval.

Must not:

- Use raw backend labels as primary UI labels.
- Provide a normal product button to simulate merchant sold out.

Acceptance:

- The page makes it clear why a human organizer must approve recovery.
- The v2 plan generation is visible as a result of confirmation, not just another notice.

### 7.5 Organizer Review Center

Route:

```text
/organizer/events/demo-night-tour/review
```

Goal:

Make the review report feel metric-backed.

Required layout:

- KPI summary with H5 visits, check-ins, response time, queue metric, budget used.
- Incident and recovery timeline.
- What worked / what failed / next recommendations sections.
- Evidence references to approvals and public notice publishing.

Acceptance:

- Review is not just summary text.
- At least three concrete metrics are visible.
- Recommendations are visually separated from raw results.

### 7.6 Merchant Dashboard

Route:

```text
/merchant/dashboard
```

Goal:

Make this the merchant's "today" screen.

Required layout:

- Merchant name and event context.
- Next required action.
- Today's assigned task.
- Current visitor intake state.
- Organizer notice preview.
- Large action link to report live status.

Acceptance:

- Merchant can understand what to do next without reading backend-style fields.
- Page works at mobile and tablet widths.

### 7.7 Merchant Tasks

Route:

```text
/merchant/events/demo-night-tour/tasks
```

Goal:

Make merchant tasks actionable.

Required layout:

- Task card with time slot, preparation items, visitor task, fallback instruction.
- Task status badge.
- "Before visitors arrive" checklist style if data exists or can be derived.

Acceptance:

- Task page is clearly different from status page.
- It does not show organizer evidence or approval internals.

### 7.8 Merchant Status

Route:

```text
/merchant/events/demo-night-tour/status
```

Goal:

Turn the current form into a live status control.

Required layout:

- Large current status block:
  - accepting visitors / paused
  - stock normal / low / sold out
  - queue normal / busy / overloaded
- Large touch-friendly quick actions:
  - Accept visitors
  - Pause visitors
  - Report low stock
  - Report sold out
  - Queue busy
- Advanced detail form can remain below quick actions.
- Submitted state must be visible and persistent enough to notice.
- If sold out triggers incident, show "Organizer review requested" prominently.

Acceptance:

- The merchant can report a sold-out state in one obvious action.
- The page does not feel like an admin form.

### 7.9 Merchant Notifications

Route:

```text
/merchant/notifications
```

Goal:

Show organizer messages in a credible message feed.

Required layout:

- Notice list with timestamp/status where available.
- Empty state if no notice exists.
- Recovery-related notice should be visually distinct.

Acceptance:

- Notifications are route-specific and not a placeholder.

### 7.10 Tourist Logged-In Route

Routes:

```text
/user/events/demo-night-tour
/user/events/demo-night-tour/route
/user/events/demo-night-tour/points/rp001
/user/events/demo-night-tour/notices
```

Goal:

Make logged-in tourist routes feel like a mobile route companion.

Required layout:

- Event identity and current route status.
- Route progress strip or stop sequence.
- Next stop card.
- Story content for each route point.
- Visitor task callout.
- Live notice area.
- Bottom navigation remains, but should feel more native/mobile.

Must not show:

```text
PlanVersion
Incident
RecoveryProposal
AgentTrace
Tourist H5
Route version
```

Acceptance:

- Logged-in tourist route is visually distinct from public route.
- It feels like a route guide, not a backend preview.

### 7.11 Public H5

Route:

```text
/public/events/demo-night-tour
```

Goal:

Make this the shareable visitor-facing route page.

Required layout:

- Event header with route identity and concise public status.
- Live update block for public notices.
- Route stop sequence with richer visual hierarchy.
- Story and task content.
- Current route change should be worded for visitors, not operators.

Acceptance:

- Public route remains unauthenticated.
- It works cleanly at `390x844`.
- It does not expose internal backend terms.
- It looks like a cultural event page, not an admin preview.

## 10. State Requirements

Each major page must include at least the states it can realistically hit in the deterministic demo.

Required minimum:

| Page | Required states |
| --- | --- |
| Login | idle, submitting, invalid credentials |
| Organizer dashboard | event active, live exception present, no exception |
| Event workspace | no plan, plan draft, plan approved, v2 after recovery |
| Exception center | no exception, exception open, proposal ready, recovery approved |
| Merchant status | normal, submitting, submitted, incident requested |
| Tourist/public route | loading, route active, notice present, no notice |
| Review | report missing, report available |

States can be deterministic/mock-backed, but they must be visible and designed.

## 13. Stop Gates

Stop and ask for review if:

- The implementation requires a new major UI framework.
- The chosen reference/template license is unclear.
- The template adoption document does not map references to exact local files.
- A visual change would remove role separation.
- A visual change requires backend core workflow changes.
- A proposed change only modifies color, spacing, or shell layout while leaving the page workflow unchanged.
- Mobile screenshots show horizontal overflow or incoherent overlap.
- Public/tourist pages leak internal backend terms.
- Login breaks for any demo account.
