# v2.1 Auth, Session, And Demo Boundary Hardening Spec

Date: 2026-06-27

Status: proposed.

## Purpose

v2.1 separates the current local demo authentication behavior from any future staging or production behavior. The project currently has backend-backed login, role guards, session cookies, and CSRF checks, but the defaults are intentionally demo-oriented:

- demo users are seeded on startup;
- demo credentials are visible in the frontend login UI;
- mutation requests use a fixed `X-Zhiyin-CSRF: demo` header;
- session cookies are `httponly` and `samesite=lax`, but not environment-aware;
- startup does not refuse unsafe production configuration.

v2.1 does not introduce public registration, multi-tenant identity, password reset, OAuth, or a real user administration console. It creates a defensible boundary so demo behavior cannot be mistaken for production behavior.

## Readiness Target

v2.1 moves authentication from "demo-convenient" to "environment-gated beta baseline".

It must preserve local demo ergonomics while making unsafe production startup impossible.

## In Scope

- Add typed backend application/security settings.
- Define `APP_ENV` values: `local`, `demo`, `staging`, `production`.
- Define explicit `DEMO_MODE`.
- Seed demo accounts only when demo mode is enabled.
- Make session cookie attributes environment-aware.
- Replace unconditional fixed demo CSRF acceptance with an environment-gated policy.
- Add startup validation that refuses production mode with demo CSRF, demo accounts, missing secret material, or insecure origins.
- Add frontend configuration for demo quick-fill visibility and mutation CSRF header behavior.
- Add tests for local/demo compatibility and production refusal cases.
- Update `.env.example`, README, and docs/ai verification notes.

## Out Of Scope

- Public sign-up.
- Password reset.
- OAuth or SSO.
- Role management UI.
- Tenant isolation.
- Database migration work beyond the settings needed for auth tests.
- Deployment provider configuration.

## Required Backend Behavior

### Environment Settings

Backend code must expose one settings object loaded from environment variables:

| Variable | Allowed Values | Local Default | Production Requirement |
| --- | --- | --- | --- |
| `APP_ENV` | `local`, `demo`, `staging`, `production` | `local` | must be `production` in production deployments |
| `DEMO_MODE` | `true`, `false` | `true` for local | must be `false` |
| `APP_SECRET_KEY` | non-empty string | optional for local | required |
| `SESSION_COOKIE_SECURE` | `true`, `false`, `auto` | `auto` | must resolve to `true` |
| `SESSION_SAMESITE` | `lax`, `strict`, `none` | `lax` | `lax` or `strict`; `none` requires secure cookies |
| `SESSION_TTL_HOURS` | positive integer | `8` | positive integer |
| `CSRF_MODE` | `demo`, `double-submit` | `demo` when demo mode is enabled | must be `double-submit` |
| `ALLOWED_ORIGINS` | comma-separated origins | localhost origins | must be explicit HTTPS origins |

### Demo Account Seeding

`seed_demo_accounts(STORE)` may only run when `settings.demo_mode` is true.

If `APP_ENV=production` and demo mode is true, startup must fail before serving requests.

### Session Cookies

Cookie creation and clearing must use settings, not hard-coded attributes.

Production cookie rules:

- `httponly=true`
- `secure=true`
- `samesite=lax` or `strict`
- positive `max_age`
- explicit path `/`

Local demo may continue using non-secure localhost cookies.

### CSRF Policy

Local/demo mode may continue accepting `X-Zhiyin-CSRF: demo`.

Staging/production must reject the fixed demo header and require a real CSRF token strategy. v2.1 uses a double-submit baseline:

- `GET /api/auth/csrf` returns a CSRF token payload;
- the endpoint also sets a non-HttpOnly CSRF cookie;
- mutation requests must send a matching `X-Zhiyin-CSRF` header;
- tokens are signed with `APP_SECRET_KEY`;
- invalid, missing, expired, or mismatched tokens return `403`.

### Origin Policy

`is_allowed_local_dev_origin()` remains local/demo-only. Staging/production must use `ALLOWED_ORIGINS`.

### Public Endpoints

Public event endpoints remain unauthenticated, but their mutation endpoints must still pass CSRF/origin checks. v2.1 does not add visitor identity.

## Required Frontend Behavior

- `VITE_DEMO_MODE=true` keeps quick-fill demo account buttons visible.
- `VITE_DEMO_MODE=false` hides demo quick-fill controls.
- Frontend mutation requests no longer hard-code `X-Zhiyin-CSRF: demo` as the only path.
- A shared frontend CSRF helper retrieves `/api/auth/csrf` when not in demo mode and attaches the returned token.
- Existing local tests should continue to use demo mode by default.

## Tests

Backend tests must cover:

- local default settings keep demo flow working;
- demo accounts are seeded only in demo mode;
- production startup rejects `DEMO_MODE=true`;
- production startup rejects missing `APP_SECRET_KEY`;
- production startup rejects demo CSRF mode;
- session cookie attributes resolve correctly for local and production;
- demo CSRF header is accepted only in demo mode;
- double-submit CSRF accepts valid token/cookie pairs and rejects invalid pairs.

Frontend tests must cover:

- demo quick-fill visible in demo mode;
- demo quick-fill hidden when `VITE_DEMO_MODE=false`;
- mutation helper sends demo header in demo mode;
- mutation helper requests and sends a CSRF token when demo mode is false.

## Acceptance Criteria

- Local demo login still works.
- API full pytest passes.
- Web tests and build pass.
- CI remains green.
- Production-like env tests fail fast when unsafe configuration is supplied.
- Documentation states that v2.1 is a beta baseline, not complete production identity.

## Commercial Readiness Impact

v2.1 closes the v1.9 `CR-P0-003` gap for demo-vs-production auth boundary. It does not claim full production identity, tenant isolation, or compliance readiness.
