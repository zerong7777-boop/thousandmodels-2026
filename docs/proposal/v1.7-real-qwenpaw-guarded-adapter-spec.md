# Zhiyin Haojiang v1.7 Real QwenPaw Guarded Adapter Spec

Date: 2026-06-22
Project root: `<PROJECT_ROOT>`
Status: design spec for next phase

## 1. Purpose

v1.6 proved that a real local QwenPaw runtime can be reached from this workspace:

```text
isolated local QwenPaw service
-> active model opencode/deepseek-v4-flash-free
-> POST /api/agent/process
-> optional X-Agent-Id: QwenPaw_QA_Agent_0.2
-> sanitized non-empty response evidence
```

That is useful, but it is still only provider reachability. The next credibility gap is:

```text
The project has not yet proven that real QwenPaw output is structured enough
to be accepted by the existing advisory-only safety boundary.
```

v1.7 should close that gap with a guarded adapter checkpoint. The phase should not put real QwenPaw into the production organizer workflow. It should make the manual smoke stricter:

```text
real local QwenPaw call
+ bounded response extraction
+ advisory field qualification
+ authority-claim rejection
+ sanitized field-level evidence
+ explicit unqualified outcome when fields are missing
+ unchanged deterministic product path
```

The product claim after v1.7 should be:

```text
The project can optionally call a real local QwenPaw runtime and decide whether
the returned advisory is structurally qualified for the shadow adapter boundary.
```

The product claim must not be:

```text
QwenPaw now drives the live business workflow or produces authoritative operations.
```

## 2. Current baseline

Relevant completed state:

- v1.4 implements the accepted product-facing QwenPaw-style path through a fake shadow adapter.
- v1.5 adds a guarded localhost-only real QwenPaw smoke script.
- v1.6 configures a local active model and records `live_success` through `QwenPaw_QA_Agent_0.2`.
- `apps/api/scripts/live_qwenpaw_smoke.py` already enforces `RUN_LIVE_QWENPAW_SMOKE=1`, localhost-only URLs, bounded reads, `trust_env=False`, secret/path redaction, optional `QWENPAW_AGENT_ID`, and provider-error classification.
- `docs/ai/NEXT.md` recommends stricter success criteria before wiring real QwenPaw into product flow.

Current limitation:

```text
outcome=live_success
```

currently means:

```text
HTTP 200 + non-empty sanitized response preview + no detected provider error
```

It does not mean:

```text
the response contains recovery_rationale, visitor_safe_notice_draft, and safety_notes
```

v1.7 should make that distinction machine-readable.

## 3. Scope decision

v1.7 is:

- a guarded adapter qualification phase
- a response-shape parsing phase
- a field-level sanitized evidence phase
- an authority-boundary rejection phase
- a smoke script and test hardening phase
- a documentation and handoff phase

v1.7 is not:

- a production QwenPaw orchestration migration
- a default FastAPI runtime dependency
- a frontend product feature
- a public/merchant/tourist Agent evidence feature
- a HiClaw or QwenPaw Team deployment
- a remote QwenPaw service integration
- a real merchant, POS, payment, hardware, map, weather, traffic, or visitor identity integration phase

## 4. Recommended approach

Extend the existing script:

```text
apps/api/scripts/live_qwenpaw_smoke.py
```

Do not introduce a new parallel smoke script. The existing script already owns the required guards and evidence writer.

Add a qualification layer:

```text
bounded QwenPaw response stream
-> extract full bounded response text
-> classify provider/runtime failures
-> qualify advisory fields
-> write sanitized field-level evidence
-> return strict outcome
```

The v1.7 script should distinguish:

| Signal | Meaning |
| --- | --- |
| `provider_reachable=true` | Local QwenPaw returned HTTP 200 with usable non-empty response text. |
| `advisory_status=qualified` | Required advisory fields were detected and passed safety checks. |
| `advisory_status=unqualified` | Provider responded, but the advisory shape or safety checks failed. |
| `outcome=advisory_qualified` | Provider was reachable and advisory was qualified. |
| `outcome=advisory_unqualified` | Provider was reachable but advisory was not qualified. |
| `outcome=blocked` | Missing precondition, localhost guard, auth/config, provider error, timeout, or service unavailable. |
| `outcome=failed` | Script or evidence writer hit an unexpected implementation error. |

This replaces "non-empty response equals success" with a stricter adapter checkpoint.

## 5. Advisory qualification contract

### 5.1 Required fields

The QwenPaw response is qualified only when it contains all required fields:

```text
recovery_rationale
visitor_safe_notice_draft
safety_notes
```

Each field must have non-empty sanitized content.

### 5.2 Accepted shapes

The qualifier should accept two response shapes.

JSON-like shape:

```json
{
  "recovery_rationale": "Merchant m001 is sold out, so visitor flow should be rerouted.",
  "visitor_safe_notice_draft": "Some items at this stop may be unavailable. Please follow on-site guidance.",
  "safety_notes": "Advisory only. Organizer approval is required before publication."
}
```

Markdown heading shape:

```markdown
### recovery_rationale

Merchant `m001` is sold out, so the organizer should reduce visitor exposure to the stop.

### visitor_safe_notice_draft

Some items at this stop may be unavailable. Please follow on-site guidance.

### safety_notes

This is advisory only. No plan, notice, merchant runtime, coupon, or redemption state has been changed.
```

The qualifier may ignore preamble text before the first required field. It must not store the full raw transcript.

### 5.3 Rejected shapes

The response is unqualified if:

- any required field is missing;
- any required field is empty after redaction;
- the response is only preamble, self-reflection, tool-call chatter, or setup text;
- the response claims that it approved, published, applied, mutated, created, redeemed, or otherwise changed authoritative state;
- the response contains detected secrets, Authorization material, cookies, or local absolute paths after redaction;
- QwenPaw returns an SSE or JSON error payload even if HTTP status is 200.

## 7. Safety and redaction

v1.7 must preserve all v1.6 evidence safety rules:

- no raw response stream in committed artifacts
- no API keys
- no bearer tokens
- no Authorization headers
- no cookies
- no local absolute paths
- no QwenPaw secret/config paths
- no unbounded prompt or response transcript

Add one more safety check:

```text
qualified advisory excerpts must pass the same redaction path as previews
```

Authority-claim scan should reject statements that imply completed mutation:

```text
approved the plan
published the notice
applied the suggestion
mutated the state
updated merchant runtime
created a coupon
claimed a coupon
redeemed a coupon
changed inventory
```

The scan should be conservative. If the model says "do not approve" or "no state was changed", that should remain allowed. If the model says it already changed state, the advisory is unqualified.

## 8. Runtime and exit-code behavior

The script should become stricter:

| Outcome | Exit code | Meaning |
| --- | --- | --- |
| `advisory_qualified` | 0 | Local QwenPaw provider was reachable and advisory fields passed qualification. |
| `advisory_unqualified` | 1 | Provider was reachable, but adapter qualification failed. |
| `blocked` | 1 | Local precondition or provider configuration is missing. |
| `failed` | 1 | Unexpected implementation failure. |

This is intentionally stricter than v1.6. A non-empty response is no longer enough to pass the manual checkpoint.

## 13. Acceptance criteria

v1.7 is complete when:

1. `docs/proposal/v1.7-real-qwenpaw-guarded-adapter-spec.md` exists.
2. `docs/proposal/v1.7-real-qwenpaw-guarded-adapter-implementation-plan.md` exists.
3. The smoke script separates provider reachability from advisory qualification.
4. The script returns `advisory_qualified` only when all required advisory fields are present.
5. The script returns `advisory_unqualified` for missing fields, preamble-only output, or unsafe authority claims.
6. Provider/runtime failures still return `blocked`, not `advisory_unqualified`.
7. Evidence records field-presence booleans and sanitized field excerpts.
8. Evidence remains bounded and sanitized.
9. `main()` exits 0 only for `advisory_qualified`.
10. v1.4 fake QwenPaw shadow orchestration remains the accepted product path.
11. Full backend tests pass serially.
12. Final scans find no real key, bearer token, password, local absolute path, or internal term leakage to merchant/public/tourist pages.

## 14. Stop gates

Stop and ask for review if:

- the only way to get qualified output is to relax advisory-only safety wording;
- QwenPaw output must approve, publish, apply, or mutate authoritative state;
- evidence would need to store raw response streams or unbounded transcripts;
- qualification requires remote/public QwenPaw service access;
- implementation requires changing merchant, tourist, or public pages;
- implementation requires changing the v1.4 fake adapter product path;
- implementation requires HiClaw, Docker, Kubernetes, or QwenPaw Team deployment.

## 15. Summary

v1.7 should turn the real QwenPaw smoke from reachability evidence into adapter-quality evidence.

```text
localhost-only real QwenPaw call
+ provider reachability signal
+ required advisory field qualification
+ unsafe authority-claim rejection
+ sanitized field-level evidence
+ strict exit code
+ unchanged deterministic product path
```

That is the smallest useful step before deciding whether any real QwenPaw output should feed a future guarded shadow adapter.
