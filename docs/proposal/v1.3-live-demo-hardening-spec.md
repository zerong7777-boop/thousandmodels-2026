# Zhiyin Haojiang v1.3 Live Demo Hardening Spec

## 1. Purpose

v1.2 completed the event-page and merchant-edge loop, but the strongest verification is still split across backend pytest, frontend unit tests, and mocked Playwright visual smoke. That is enough to prove implementation correctness, but it is not yet enough for a competition demo where the project must look reproducible under a real local run.

v1.3 should turn the current v1.2 implementation into a stable, evidence-backed local demo:

```text
FastAPI backend
+ Vite frontend
+ real browser login
+ real API state transitions
+ screenshot-backed demo evidence
+ operator script
+ optional Qwen live-smoke classification
```

This phase is a hardening and evidence phase, not a broad feature phase.

## 2. Current Baseline

As of the v1.2 handoff:

- v1.2 backend contracts, services, frontend surfaces, tests, and screenshots exist in the working tree.
- `docs/ai/STATUS.md` records v1.2 Task 8-10 as complete.
- `docs/ai/VERIFY.md` records full v1.2 verification as passed.
- The v1.2 Playwright smoke uses deterministic mocked API data for product-flow and screenshot evidence.
- Backend authority is covered by FastAPI pytest, not by a live backend plus browser E2E.
- The current working tree contains the v1.2 changes and should be frozen before starting v1.3 implementation.
- The current Qwen live-smoke record proves the optional DashScope path can produce model-call evidence, but the recorded outcome is `live_completed_without_success_model_call`, not `live_success`.
- QwenPaw workflow orchestration is not implemented and must not be claimed.

## 3. Problem Statement

The project can currently be shown through polished screenshots and tests, but a judge or teammate may still ask:

1. Does the browser really talk to the FastAPI backend locally?
2. Can the demo be reset to a known state before presenting?
3. Does role login work in a real browser session, not only mocked tests?
4. Does merchant status reporting really create organizer-side incidents and suggestions?
5. Do public scan, coupon claim, and redemption interactions really feed the review report?
6. Is Qwen actually live, or only an optional architecture lane?

v1.3 must answer those questions with concrete commands, screenshots, and sanitized artifacts.

## 4. Positioning Decision

v1.3 is:

- a real-local-demo hardening phase
- a reproducibility phase
- a competition-evidence phase
- a demo-script phase
- a Qwen live-smoke classification phase

v1.3 is not:

- a QwenPaw implementation
- a new product-surface redesign
- a real merchant integration
- a POS, payment, map, weather, traffic, or hardware phase
- a model-training or fine-tuning phase
- a phase that makes Qwen or DashScope required for the normal demo

The recommended story after v1.3 is:

> "The deterministic multi-role operations loop runs locally end to end through the real backend. Qwen is an optional controlled draft lane; when live Qwen succeeds we show sanitized model evidence, and when it fails or is unsafe we show the fallback record without weakening operational control."

## 5. Scope

### 5.1 In Scope

1. Freeze the v1.2 baseline before starting new changes.
2. Add a deterministic demo reset command for local operators.
3. Add a live FastAPI + Vite Playwright smoke that uses the real backend.
4. Exercise real browser login for organizer and merchant roles.
5. Exercise the real public H5 route without login.
6. Run the event-page and merchant-edge flow through real API state:
   - seed demo
   - generate and approve plan v1
   - draft and publish event page
   - generate merchant edge packages
   - merchant reports sold out
   - organizer sees incident
   - organizer generates operation suggestions
   - public page records scan, coupon claim, and redemption
   - review report shows touchpoint and merchant outcome metrics
7. Capture v1.3 screenshots from the live local run.
8. Write `docs/research/v1.3-live-demo-smoke.md`.
9. Write `docs/research/v1.3-operator-demo-script.md`.
10. Re-run optional Qwen live smoke only from process environment credentials if credentials are available locally.
11. Update `docs/ai/STATUS.md`, `docs/ai/VERIFY.md`, and `docs/ai/NEXT.md` after execution.

### 5.2 Out Of Scope

1. QwenPaw workflow orchestration.
2. Required model API path.
3. Model-generated authoritative `PlanVersion`.
4. Model approval, publication, inventory mutation, coupon authority, or route mutation.
5. Real merchant onboarding.
6. Hardware, POS, payment, QR payment, or coupon settlement integrations.
7. Real map, weather, traffic, or crowd-estimation APIs.
8. Registration, full SaaS permission model, or production auth.
9. Broad frontend visual redesign.
10. Training or fine-tuning models.

## 6. Target Demo Loop

The v1.3 live demo smoke should prove this loop through the real app:

```text
Reset demo state
-> organizer login
-> generate plan v1
-> approve plan v1
-> draft event page
-> publish event page
-> generate merchant edge packages
-> merchant login
-> merchant reports sold out
-> organizer exception center sees incident
-> organizer generates operation suggestion
-> public event page shows current published event page
-> visitor scan / claim / redeem interactions are recorded
-> organizer review center shows touchpoint metrics and merchant outcomes
```

The smoke can use backend API calls for setup operations that are not the focus of browser presentation, but it must include real browser route visits and real authenticated sessions for the role surfaces.

## 7. Technical Design

### 7.1 v1.2 Baseline Freeze

Before v1.3 implementation begins, the existing v1.2 working tree should be either committed or deliberately split into a clean baseline. The goal is to prevent v1.3 hardening from being mixed with the already-completed v1.2 feature work.

The baseline freeze should include:

- v1.2 backend service files
- v1.2 frontend surfaces and API types
- v1.2 tests
- v1.2 proposal and smoke docs
- v1.2 screenshot assets
- updated `docs/ai` handoff files

No v1.3 source changes should be made before this decision is explicit.

### 7.2 Demo Reset Command

v1.3 should add a local reset script under `apps/api/scripts/`.

Required behavior:

- clear `demo-night-tour` event state
- clear demo auth sessions when the store supports it
- ensure auth schema exists
- seed demo accounts
- seed the demo event
- print JSON with `status`, `event_id`, and demo account usernames

The reset command must not require DashScope or Qwen credentials.

### 7.3 Live Playwright Config

The existing Playwright config starts Vite only and is designed for mocked route smoke. v1.3 should add a separate live config instead of changing the historical visual-smoke behavior.

Required behavior:

- start FastAPI on `127.0.0.1:8000`
- start Vite on a separate fixed port such as `5179`
- set `VITE_API_BASE_URL=http://127.0.0.1:8000` for the Vite server
- keep the default draft backend deterministic
- run only v1.3 live smoke tests through this config

The existing mocked Playwright suite should remain available and should not become dependent on a backend process.

### 7.4 Live Smoke Test

The live smoke should be a focused Playwright test under `apps/web/tests/e2e/`.

It must verify:

- organizer login uses the real login form and backend session
- organizer workspace displays the published event page and merchant package state
- merchant login uses the real login form and backend session
- merchant sold-out action creates an incident through the backend
- organizer exception center displays the incident and operation suggestion
- public H5 is accessible without login
- public scan, claim, and redeem actions succeed through the backend
- review center displays touchpoint metrics and merchant outcomes after public interactions
- public H5 does not expose raw backend/model terms

Screenshots should be saved under:

```text
docs/research/assets/v1.3-live-demo-smoke/
```

### 7.5 Qwen Live Smoke Classification

v1.3 should not make Qwen required. It should classify the optional live Qwen state honestly:

- `live_success`: at least one Qwen model call returns `response_status == "success"` and `fallback_used == false`
- `live_completed_without_success_model_call`: provider calls occurred but all useful outputs fell back or failed schema/safety checks
- `blocked`: the local process lacks required env, network, quota, or provider access

The existing guarded script `apps/api/scripts/live_qwen_smoke.py` should be reused unless implementation discovers a concrete bug. The script must continue to avoid storing keys, Authorization headers, and raw provider payloads.

### 7.6 Evidence Documents

v1.3 should produce two research documents:

```text
docs/research/v1.3-live-demo-smoke.md
docs/research/v1.3-operator-demo-script.md
```

The smoke document should record:

- commands
- exit codes
- screenshots
- live route coverage
- backend and frontend verification
- Qwen smoke outcome if run
- fallback classification if Qwen does not produce live success
- secret and local-path scan results

The operator script should record:

- demo accounts
- pre-demo reset command
- 3-5 minute browser walkthrough
- expected screen evidence for each step
- claims that are allowed
- claims that must not be made

## 8. Testing Requirements

### 8.1 Backend

Run:

```powershell
python -m pytest -q
```

The full backend suite must pass serially. Do not run backend pytest in parallel against the default SQLite store.

### 8.2 Frontend Unit And Build

Run from `apps/web`:

```powershell
npm.cmd run test
npm.cmd run build
```

Both must pass.

### 8.3 Mocked Playwright Regression

Run from `apps/web`:

```powershell
npm.cmd exec playwright test
```

The existing mocked visual/product-flow suite should continue to pass with the same expected historical skips.

### 8.4 Live Playwright Smoke

Run from `apps/web`:

```powershell
npm.cmd exec playwright test tests/e2e/v13-live-demo-smoke.spec.ts --config playwright.live.config.ts
```

This command must start both FastAPI and Vite and use the real backend API.

### 8.5 Boundary Scans

Run from project root:

```powershell
rg -n "sk-[A-Za-z0-9._-]{20,}|Bearer\s+sk-[A-Za-z0-9._-]{20,}|DASHSCOPE_API_KEY\s*=\s*sk-" apps docs README.md .gitignore
rg -n "([A-Z]:\\Users\\[^\\]+|[A-Z]:\\rz\\|[A-Z]:/rz/)" docs apps README.md
rg -n "AgentDraft|AgentRun|PlanVersion|RecoveryProposal|qwen|dashscope|schema_failed|approval_status" apps\web\src\pages\public apps\web\src\pages\merchant apps\web\src\i18n\dictionaries
```

The first two strict scans should find no real secret or local absolute path. The public/merchant source scan should not expose raw model/backend terms in visitor or merchant surfaces.

## 9. Acceptance Criteria

v1.3 is complete when:

1. v1.2 baseline is frozen before v1.3 changes.
2. A reset command exists and is tested.
3. A separate live Playwright config exists.
4. A live Playwright smoke uses FastAPI + Vite with `VITE_API_BASE_URL`.
5. The live smoke covers organizer, merchant, and public H5 surfaces.
6. The live smoke records scan, claim, redemption, incident, operation suggestion, and review metrics through the real backend.
7. v1.3 screenshots exist under `docs/research/assets/v1.3-live-demo-smoke/`.
8. `docs/research/v1.3-live-demo-smoke.md` records commands, results, screenshots, and boundaries.
9. `docs/research/v1.3-operator-demo-script.md` gives a runnable 3-5 minute demo script.
10. Optional Qwen live smoke is either re-run and recorded, or explicitly classified as blocked due to missing local process credentials.
11. Full backend tests, frontend tests, frontend build, mocked Playwright, live Playwright smoke, secret scan, local-path scan, and public-boundary scan are recorded.
12. `docs/ai/STATUS.md`, `docs/ai/VERIFY.md`, and `docs/ai/NEXT.md` are updated with actual results.
13. No QwenPaw implementation claim is introduced.
14. No model output becomes authoritative operational state.

## 10. Stop Gates

Stop and report instead of forcing implementation if:

- v1.2 baseline cannot be cleanly frozen
- live Playwright requires storing secrets or local paths in repo files
- the backend cannot be started locally without changing production-facing code
- the live smoke would require weakening auth, CSRF, schema, or unsafe-mutation guards
- Qwen success would require making model output authoritative
- public or merchant pages need to expose raw backend/model terms for the demo to pass
- the work drifts into QwenPaw orchestration or real external integrations
