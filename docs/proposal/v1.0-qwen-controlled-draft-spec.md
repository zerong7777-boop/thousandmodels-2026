# Zhiyin Haojiang v1.0 Qwen Controlled Draft Spec

Date: 2026-06-12
Project root: `<PROJECT_ROOT>`
Status: design spec for next phase

## 1. Purpose

v0.8 made the backend Agent layer concrete. It records `AgentRun`, enriched `AgentStep`, `AgentToolCall`, `AgentDraft`, `AgentModelCall`, and `AgentEvaluation` evidence for deterministic planning, recovery, and review.

v0.9 made that evidence visible inside organizer-facing product pages. Judges can now see which specialist Agents ran, which deterministic tools were called, which drafts were produced, and where human approval remains mandatory.

The next weakness is clear:

```text
The Agent runtime is auditable and product-visible, but the draft generation path is still deterministic.
```

v1.0 should add optional Qwen-assisted draft generation for controlled `AgentDraft` objects only. It should make the Agent layer more competition-legible without weakening the deterministic demo, human approval boundary, or public safety gates.

The target stance:

```text
Qwen may draft, explain, rewrite, and summarize.
Qwen must not approve, publish, mutate inventory, create route versions, or bypass organizer confirmation.
```

## 3. Problem Statement

The project can now show Agent evidence, but the model-assisted part is not implemented in a judge-visible, verifiable way.

If v1.0 jumps directly to broad QwenPaw claims or model-generated plans, it risks three failures:

1. Reliability failure:
   local demo depends on network/model availability.
2. Safety failure:
   model output appears to mutate authoritative operational state.
3. Evidence failure:
   the project claims Qwen/QwenPaw involvement without clear `AgentModelCall`, validation, fallback, and UI evidence.

v1.0 should instead implement the narrowest useful model capability:

```text
Qwen-assisted controlled drafts
+ strict schema validation
+ deterministic fallback
+ AgentModelCall evidence
+ organizer-only visibility
```

This improves Agent depth while preserving the strongest current asset: a deterministic end-to-end demo that always works.

## 4. Positioning Decision

v1.0 is:

- a backend-first controlled draft enrichment phase
- an optional model-assisted Agent capability phase
- a validation and fallback evidence phase
- a small organizer evidence UI extension if backend P0 succeeds

v1.0 is not:

- a QwenPaw workflow implementation
- a model-generated route planning rewrite
- a chat assistant feature
- a public-facing model feature
- a broad UI redesign
- a real merchant, hardware, POS, payment, map, weather, or traffic integration
- a replacement for deterministic planning and recovery services

The v1.0 product claim should be:

```text
The system can optionally use Qwen to enrich controlled organizer drafts.
Every model output is schema-checked, safety-checked, recorded, and fallbacks to deterministic output.
Operational state still changes only through existing backend workflow and organizer approval.
```

## 5. Scope

### 5.1 In Scope

v1.0 includes:

- a controlled draft generation abstraction
- Qwen-backed draft generation behind explicit configuration
- deterministic draft fallback for every Qwen path
- strict JSON parsing and Pydantic validation for model output
- `AgentModelCall` persistence for skipped, success, invalid JSON, schema failure, and provider error states
- `AgentRun` fallback/status updates when Qwen is requested but unavailable or invalid
- public-copy forbidden-term evaluation for model-generated public notices
- optional organizer evidence UI display for model provider/status/fallback
- backend tests for success, no-key fallback, provider error fallback, invalid JSON fallback, schema failure fallback, public forbidden-term rejection, and no unsafe mutation
- frontend tests only if organizer UI is changed
- docs/ai handoff updates after verification

### 5.2 Target Draft Types

P0 draft types:

```text
recovery_explanation
public_notice
review_summary
```

Rationale:

- They are text-heavy and benefit from model assistance.
- They are already modeled as `AgentDraft`.
- They do not need to become authoritative state.
- They support the strongest demo sequence: incident -> recovery -> public notice -> review.

P1 draft type:

```text
route_story
```

Rationale:

- It may improve cultural narrative quality.
- It is lower priority because the current demo judgment gap is Agent capability evidence, not story copy.

Out-of-scope draft type for v1.0:

```text
plan_candidate
```

Reason:

- Model-generated plan candidates are closer to authoritative route planning.
- They create more validation, safety, and product complexity.
- They should wait until the controlled text draft path is proven.

### 5.3 Out Of Scope

v1.0 must not include:

- mandatory `DASHSCOPE_API_KEY`
- required network access for local demo
- QwenPaw workflow orchestration
- autonomous plan approval
- autonomous recovery approval
- autonomous public notice publication
- autonomous merchant runtime mutation
- model-generated `PlanVersion` v2
- model-generated `MerchantRuntimeState`
- registration/account expansion
- real merchant onboarding
- vector database or external RAG
- real map/weather/traffic/POS/payment integration
- broad page redesign

## 6. Configuration Model

v1.0 should introduce a separate draft backend configuration instead of overloading the existing `plan_event` adapter.

Recommended environment variables:

```text
AGENT_DRAFT_BACKEND=deterministic | qwen
DASHSCOPE_API_KEY=<optional>
QWEN_MODEL=qwen-plus
QWEN_TIMEOUT_SECONDS=30
```

Default:

```text
AGENT_DRAFT_BACKEND=deterministic
```

Behavior:

| Configuration | Behavior |
| --- | --- |
| unset | deterministic drafts only |
| `AGENT_DRAFT_BACKEND=deterministic` | deterministic drafts only |
| `AGENT_DRAFT_BACKEND=qwen` without `DASHSCOPE_API_KEY` | deterministic fallback, record skipped model call if a Qwen run was requested |
| `AGENT_DRAFT_BACKEND=qwen` with key and valid response | Qwen draft accepted as `AgentDraft(status=draft)`, not authoritative state |
| `AGENT_DRAFT_BACKEND=qwen` with invalid/error response | deterministic fallback, record failure reason |

Do not make `AGENT_BACKEND=qwen` the v1.0 control path. That variable currently relates to the older planning adapter. v1.0 should avoid confusing "plan generation backend" with "controlled draft backend".

## 7. Target Architecture

### 7.1 Draft Generator Interface

Introduce a small backend abstraction:

```text
DraftGenerator
  generate_recovery_explanation(context) -> DraftGenerationResult
  generate_public_notice(context) -> DraftGenerationResult
  generate_review_summary(context) -> DraftGenerationResult
```

Two implementations:

```text
DeterministicDraftGenerator
QwenDraftGenerator
```

`DeterministicDraftGenerator` may wrap existing functions:

```text
build_recovery_explanation_draft
build_public_notice_draft
build_review_summary_draft
```

`QwenDraftGenerator` must call Qwen only to produce draft candidate content and structured payload. The backend remains responsible for IDs, timestamps, source run id, status, validation, persistence, and fallback.

### 7.2 DraftGenerationResult

Recommended internal result object:

```text
draft: AgentDraft
model_call: AgentModelCall
evaluation: AgentEvaluation | None
fallback_used: bool
fallback_reason: str | None
```

If Qwen is not requested, `model_call` may be omitted or recorded as `provider=deterministic`, depending on implementation simplicity. If Qwen is requested and skipped/failed, `AgentModelCall` should be recorded so the organizer can see why fallback occurred.

### 7.3 Qwen Candidate Schemas

Do not ask Qwen to return full `AgentDraft` objects. That would let model output invent IDs, statuses, timestamps, or source references.

Ask Qwen to return draft candidate payloads:

```text
QwenDraftCandidate
  content: str
  locale: zh-Hans | zh-Hant | en | mixed
  structured_payload: dict
  evidence_refs: list[str]
  safety_notes: list[str]
```

Backend wraps candidate into `AgentDraft`:

```text
draft_id: generated by backend
event_id: trusted backend value
source_run_id: trusted backend value
draft_type: trusted backend value
status: "draft"
created_at: backend time
```

### 7.4 Prompt Templates

Prompt templates should be explicit and versioned:

```text
qwen_recovery_explanation_v1
qwen_public_notice_v1
qwen_review_summary_v1
```

Prompt templates must instruct:

- return JSON only
- no markdown fences
- do not include backend schema names in public copy
- do not approve, publish, mutate inventory, or create plan versions
- include only draft content and evidence references
- preserve organizer approval boundary

Store prompt template ids in `AgentModelCall.prompt_template_id`.

Do not persist raw prompts by default. Persist:

- template id
- input refs
- provider
- model
- response status
- parsed output if safe
- error summary

## 8. Validation Pipeline

Every Qwen draft candidate must pass the same ordered pipeline:

```text
provider response
-> JSON parse
-> Pydantic candidate schema
-> draft-type-specific validation
-> public forbidden-term check when draft_type=public_notice
-> unsafe mutation field check
-> wrap into AgentDraft
-> persist AgentModelCall, AgentDraft, AgentEvaluation
```

### 8.1 JSON Parse Failure

If JSON parsing fails:

```text
AgentModelCall.response_status = invalid_json
AgentModelCall.fallback_used = true
AgentRun.status = fallback_completed
AgentRun.fallback_used = true
fallback to deterministic draft
```

### 8.2 Schema Failure

If candidate validation fails:

```text
AgentModelCall.response_status = schema_failed
fallback to deterministic draft
record validation reason in error_summary
```

### 8.3 Provider Error

If DashScope request fails, times out, or returns non-2xx:

```text
AgentModelCall.response_status = provider_error
fallback to deterministic draft
record compact error_summary
```

Do not expose raw provider error bodies if they may contain sensitive content.

### 8.4 No Key Or Disabled Backend

If Qwen is requested but no `DASHSCOPE_API_KEY` is present:

```text
AgentModelCall.response_status = skipped
fallback to deterministic draft
fallback_reason = missing_provider_key
```

Use a normalized internal reason such as:

```text
missing_provider_key
```

Do not ask users for the key inside the product UI.

### 8.5 Unsafe Mutation Check

Reject model output that contains fields or instructions suggesting direct mutation:

```text
approval_status
publish_status
approved_by
approved_at
inventory_status
queue_status
available_for_visitors
plan_patch
merchant_task_patch
public_notice.publish_status
create_plan_version
approve_recovery
publish_notice
```

This check can start as a recursive key scan plus content scan. It does not need to be a complex policy engine in v1.0.

### 8.6 Public Copy Forbidden Terms

For `public_notice`, reuse or extend `apps/api/app/agents/guards.py`.

Forbidden public terms must include at least:

```text
PlanVersion
RecoveryProposal
Incident
AgentTrace
AgentRun
AgentDraft
AgentToolCall
AgentModelCall
backend
schema
fallback
Qwen
QwenPaw
deterministic
runtime state
runtime_state
```

If forbidden terms appear:

```text
AgentEvaluation.forbidden_public_terms_present = true
AgentEvaluation.public_copy_ready = false
fallback to deterministic public notice draft
```

## 9. Runtime Integration

### 9.1 Incident Recovery

Current deterministic flow:

```text
merchant runtime update
-> Incident
-> AgentRuntime.run_incident_recovery_preview
-> RecoveryProposal draft payload
-> recovery explanation AgentDraft
-> public notice AgentDraft
-> AgentEvaluation
```

v1.0 target:

```text
merchant runtime update
-> Incident
-> AgentRun(trigger=incident_recovery, mode=qwen_draft if requested)
-> deterministic recovery proposal tool call
-> Qwen or deterministic recovery_explanation draft
-> Qwen or deterministic public_notice draft
-> validation and fallback
-> pending evidence only
```

Important boundary:

```text
The model does not create or approve RecoveryProposal.
The existing recovery service remains the source of the proposal payload.
```

### 9.2 Review Generation

Current deterministic flow:

```text
ReviewAgent reads metrics/incidents/notices/proposals
-> build_review_summary_draft
```

v1.0 target:

```text
ReviewAgent reads evidence refs
-> Qwen or deterministic review_summary draft
-> schema validation
-> fallback if needed
-> ReviewReport remains metric-backed deterministic output
```

Important boundary:

```text
The model draft may improve prose and recommendations.
It must not fabricate metrics or claims not present in input refs.
```

### 9.3 Planning Generation

v1.0 should not enable Qwen plan generation.

Allowed:

```text
optional route_story draft if P0 is complete
```

Not allowed:

```text
Qwen creates PlanVersion
Qwen changes route point order authoritatively
Qwen changes merchant task assignment authoritatively
```

## 13. Testing Requirements

### 13.1 Backend P0 Tests

Add backend tests for:

1. `AGENT_DRAFT_BACKEND` defaults to deterministic.
2. `AGENT_DRAFT_BACKEND=qwen` without `DASHSCOPE_API_KEY` falls back deterministically.
3. missing key records skipped `AgentModelCall` when Qwen draft mode is requested.
4. mocked Qwen success creates `AgentDraft` with `source_run_id`, `status=draft`, and `AgentModelCall(response_status=success)`.
5. invalid JSON falls back and records `response_status=invalid_json`.
6. schema failure falls back and records `response_status=schema_failed`.
7. provider error falls back and records `response_status=provider_error`.
8. public notice containing forbidden terms is rejected or replaced by deterministic fallback.
9. model output with unsafe mutation fields is rejected.
10. recovery approval still creates `PlanVersion` v2 only through existing approval endpoint.
11. public notice is not published until organizer approval.
12. review summary draft cannot claim metrics absent from input refs.
13. full `python -m pytest -q` remains green without `DASHSCOPE_API_KEY`.

### 13.2 Frontend Tests If UI Changes

If v1.0 adds model evidence UI:

1. organizer exception center shows model draft status when present.
2. organizer review center shows model draft status when present.
3. deterministic runs still show "No model call recorded" or equivalent.
4. fallback runs show fallback status only to organizer.
5. merchant/tourist/public source gates still reject model/backend terms.
6. Playwright v0.9/v1.0 smoke still passes.

### 13.3 Mocking Requirements

Do not call live DashScope in automated tests.

Use one of:

```text
injected provider function
httpx mock transport
provider interface stub
monkeypatch on Qwen client call
```

Tests should be deterministic and offline.

## 16. Stop Gates

Stop and ask for review if:

- implementation would make `DASHSCOPE_API_KEY` mandatory
- a live model call is needed for automated tests
- Qwen output needs to create or approve `PlanVersion`
- Qwen output needs to publish `PublicNotice`
- Qwen output needs to mutate `MerchantRuntimeState`
- schema parsing is unreliable
- public notice safety checks cannot block internal terms
- fallback behavior cannot be recorded in `AgentRun` or `AgentModelCall`
- model evidence would leak into merchant/tourist/public pages
- implementation requires a large backend workflow rewrite
- scope drifts into QwenPaw, real merchants, hardware, maps, payment, POS, weather, or traffic prediction

## 17. Recommended Implementation Phases

### Phase A: Tests And Interfaces

Write failing backend tests for draft backend selection, missing-key fallback, success parsing, invalid JSON, schema failure, provider error, forbidden public terms, and unsafe mutation fields.

Define the draft generator interface and candidate schemas.

### Phase B: Deterministic Wrapper

Wrap existing deterministic draft functions behind the new interface. This should preserve current behavior before Qwen is added.

### Phase C: Qwen Provider Adapter

Implement a provider adapter that can be mocked in tests. It should return raw response text to the validation pipeline and never directly produce authoritative state.

### Phase D: Validation And Fallback Pipeline

Implement parse, schema validation, draft-type validation, public forbidden-term checks, unsafe mutation checks, fallback, and `AgentModelCall` recording.

### Phase E: Runtime Integration

Integrate the draft generator into incident recovery and review generation. Keep proposal creation, approval, public notice publication, and plan versioning unchanged.

### Phase F: Organizer Evidence UI

Only if backend P0 is stable, expose model call status and fallback evidence in organizer Agent evidence panels.

### Phase G: Verification And Handoff

Run backend tests, frontend tests if changed, build, Playwright if changed, boundary scans, and update `docs/ai/STATUS.md`, `docs/ai/VERIFY.md`, and `docs/ai/NEXT.md`.

## 19. Summary

v1.0 should make model-assisted Agent capability real without making the system fragile.

The target is:

```text
deterministic demo remains the source of truth
+ optional Qwen draft enrichment
+ strict schema validation
+ public-copy and mutation guards
+ deterministic fallback
+ AgentModelCall evidence
+ organizer-only model evidence
+ no raw model/backend terms on merchant/tourist/public pages
```

This is the right next step because v0.8/v0.9 already prove Agent evidence and organizer visibility. v1.0 should now prove the project can safely use Qwen in the narrow place where a model is most useful: controlled drafts that humans review before any operational state changes.
