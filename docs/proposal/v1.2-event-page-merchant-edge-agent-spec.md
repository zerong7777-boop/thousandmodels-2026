# Zhiyin Haojiang v1.2 Event Page And Merchant Edge Agent Spec

Date: 2026-06-15
Project root: `<PROJECT_ROOT>`
Status: design spec for next phase

## 1. Purpose

v1.1 made the optional Qwen lane safer and more explainable, but the product still has a positioning risk:

```text
Judges may read the current public H5 and tourist route as a visitor travel product,
instead of an activity-level AI operations system for organizers and merchants.
```

The external review document supplied on 2026-06-15 makes a useful correction. The strongest competition positioning is not "a tourist app for old-district routes." The stronger positioning is:

```text
an AI activity coordination and operations layer
connecting organizers, merchants, Event Page touchpoints, in-shop touchpoints,
merchant runtime state, operational adjustment, and review evidence.
```

v1.2 should therefore convert the current public H5 and merchant status flow into a more explicit operations loop:

```text
Organizer creates and approves an EventPlan
-> Central Agent publishes an Event Page
-> Merchant Edge Agent creates per-shop activity packages
-> Event Page and in-shop QR/coupon/redemption touchpoints collect anonymous interaction data
-> Merchant runtime state and touchpoint data trigger operational suggestions
-> Organizer approves exposure/route/notice adjustments
-> Event Page updates safely
-> Review report includes merchant-level outcomes and extension tasks
```

The goal is to make the Agent capability more credible without starting a broad rewrite.

## 3. Problem Statement

The project already has an end-to-end loop, but the loop is still centered on:

```text
plan -> merchant status update -> incident -> recovery -> public H5 -> review
```

That is good, but it does not fully express the external review document's core idea:

```text
Central Agent manages the activity.
Merchant Edge Agent manages each shop's in-shop execution and interaction data.
Event Page is the public activity touchpoint, not a generic tourist app.
```

If v1.2 continues adding generic UI polish or more Qwen evidence, the project may still look like:

```text
a polished event management demo plus model draft cards
```

v1.2 should instead make the activity operations model concrete:

- Event Page becomes a first-class business object and public projection.
- Merchant Edge Agent becomes a first-class per-merchant capability.
- QR, coupon, redemption, and in-shop touchpoints become simulated but structured data sources.
- The Agent layer can use those sources to produce operational suggestions.
- The review report can cite anonymous interaction and redemption data, not just static metrics.

## 4. Positioning Decision

v1.2 is:

- an Event Page product-modeling phase
- a Merchant Edge Agent capability phase
- a simulated QR/coupon/redemption loop phase
- an anonymous interaction data phase
- an Agent evidence hardening phase
- a small product copy/IA correction phase

v1.2 is not:

- a broad UI redesign
- a new tourist app phase
- a real merchant onboarding phase
- a real QR payment or POS integration
- a real coupon platform integration
- a real map, traffic, weather, or hardware integration
- a QwenPaw implementation
- a model-autonomous operations phase
- a mandatory live Qwen phase

The v1.2 product claim should be:

```text
The system coordinates an old-district activity through an organizer-controlled Event Page
and per-merchant Edge Agent packages.
Shop-level QR/coupon/redemption signals and merchant state updates feed the Agent runtime,
which proposes adjustments while organizer approval remains mandatory.
```

## 5. Scope

### 5.1 In Scope

v1.2 includes:

- `EventPage` as an explicit backend object or projection
- Event Page draft, publish status, version, route, merchant list, offer summary, and notice projection
- public H5 copy and route naming aligned to "Event Page" rather than "tourist app"
- `MerchantInteractionPackage` for each participating merchant
- `InShopTouchpoint` records for QR, coupon, redemption, status report, and in-shop display
- simulated QR/coupon/redemption identifiers and URLs
- simulated anonymous interaction events:
  - Event Page view
  - activity QR scan
  - in-shop QR scan
  - coupon claim
  - coupon redemption
  - task/stamp completion if low cost
- merchant runtime updates still triggering incidents
- Merchant Edge Agent run/evidence for activity package generation and interaction summary
- operational suggestions that can use:
  - sold-out status
  - queue status
  - availability
  - coupon claim count
  - redemption count
  - Event Page view/scan count
- organizer approval boundary for exposure, notice, and route changes
- review report enrichment with merchant-level performance and follow-up tasks
- tests for the new backend loop and public boundary
- docs/ai handoff updates after implementation

### 5.2 Out Of Scope

v1.2 must not include:

- real POS integration
- real payment integration
- real coupon issuance or clearing
- real QR code image generation as a hard dependency
- real visitor accounts
- visitor identity tracking
- visitor preference memory
- real merchant registration workflow expansion
- real merchant private financial data upload
- real map/geolocation API
- real weather or traffic prediction
- vector database or external RAG
- QwenPaw workflow orchestration
- mandatory `DASHSCOPE_API_KEY`
- model-generated authoritative Event Page publication
- model-generated direct coupon issuance
- model-generated direct merchant runtime mutation

### 5.3 Existing Tourist/User Routes

v1.2 should not necessarily delete current tourist/user routes immediately. For compatibility, they may remain as:

```text
logged-in participant route preview
or legacy user-facing route surface
```

However, the main competition story and public route should be:

```text
Event Page + in-shop touchpoints
```

not:

```text
tourist app with personalized recommendations
```

If UI changes are made, public-facing labels should avoid "personalized tourist route" language.

## 7. Agent Architecture Changes

### 7.1 Central Planning Agent

Existing planning Agent behavior should be reframed as:

```text
CentralPlanningAgent
```

Responsibilities:

- transform `EventBrief` into `EventPlan`
- define activity rules and global KPI
- prepare Event Page draft inputs
- coordinate merchant roles and task framework
- preserve organizer approval before publication

No model output may publish Event Page directly.

### 7.2 Event Page Agent

Name:

```text
EventPageAgent
```

Responsibilities:

- generate Event Page draft from approved plan
- assemble activity title, intro, route summary, merchant cards, offer summary, and notice copy
- create or refresh main activity QR token
- produce visitor-safe public projection
- record `AgentDraft(type=event_page_copy)` or equivalent if needed
- run public-copy forbidden-term checks

Boundary:

```text
EventPageAgent may draft.
Organizer/backend publication workflow publishes.
```

### 7.3 Merchant Edge Agent

Name:

```text
MerchantEdgeAgent
```

Responsibilities:

- generate per-merchant activity package
- generate shop story and poster copy
- create simulated in-shop QR/coupon/redemption/status touchpoints
- summarize shop readiness and constraints
- read merchant runtime state
- summarize anonymous shop interaction metrics
- recommend whether the shop can receive more flow or should pause exposure

MVP implementation stance:

```text
One deterministic MerchantEdgeAgent module can handle all demo merchants.
It should create separate AgentRun or AgentStep evidence per merchant package.
It does not need a separate long-running process per shop.
```

### 7.4 Offer Operations Agent

Name:

```text
OfferOperationsAgent
```

Responsibilities:

- create coupon rules within activity-level constraints
- simulate claim and redemption metrics
- detect over-claiming, low redemption, or exhausted offers
- suggest offer display or merchant exposure adjustments

Boundary:

```text
No real coupon issuance, settlement, payment, or POS integration.
```

### 7.5 Field Operations Agent

Name:

```text
FieldOperationsAgent
```

Responsibilities:

- read merchant runtime state, interaction counts, and redemption counts
- compare overloaded merchants with merchants that still have capacity
- produce operation suggestions:
  - shift Event Page merchant ordering
  - pause a sold-out offer
  - update public notice
  - route visitors to an alternative merchant
  - create a merchant follow-up task

Boundary:

```text
FieldOperationsAgent does not commit route versions, publish notices, or mutate merchant state.
```

### 7.6 Review And Extension Agent

Name:

```text
ReviewExtensionAgent
```

Responsibilities:

- generate activity-level review from:
  - Event Page views/scans
  - in-shop scans
  - coupon claims/redemptions
  - merchant runtime updates
  - incidents and approvals
  - public notices
- generate merchant-level performance summaries
- propose 7 to 14 day extension tasks

MVP boundary:

```text
Extension tasks are recommendations only.
They do not create real campaigns or external posts.
```

## 8. Target Demo Loop

v1.2 should support this judge-facing loop:

```text
1. Organizer reviews generated plan v1.
2. Organizer approves plan v1.
3. Central/Event Page Agent creates Event Page draft.
4. Organizer publishes Event Page.
5. Merchant Edge Agent generates packages for selected merchants.
6. Merchant pages show per-shop package, QR token, coupon, redemption/status touchpoints.
7. Demo operator or merchant action simulates:
   - Event Page views/scans
   - in-shop QR scans
   - coupon claims/redemptions
   - merchant sold-out or overloaded state
8. Field Operations Agent creates suggestion or recovery evidence.
9. Organizer approves adjustment.
10. Event Page public projection updates with visitor-safe notice/order/route hint.
11. Review center shows:
   - activity metrics
   - merchant-level outcomes
   - Agent evidence
   - 7 to 14 day extension tasks.
```

This loop should be deterministic and reproducible.

Stop and ask for review if:

- implementation would require real POS, payment, coupon, or settlement integration

- implementation would require real merchant private financial data
- Event Page publication would bypass organizer approval

Write failing tests for Event Page projection, Merchant Edge packages, touchpoints, coupon/redemption simulation, anonymous metrics, operation suggestions, and public-copy boundaries.

Create or formalize Event Page draft/published projection from approved plan and notices. Keep public H5 endpoint compatible where possible.

Run backend tests, frontend tests/build/Playwright if changed, boundary scans, screenshot review, and update docs/ai.

The target is:

```text
Event Page as the public activity touchpoint
+ Merchant Edge Agent packages for each shop
+ simulated QR/coupon/redemption/status touchpoints
+ anonymous interaction metrics
+ operation suggestions from runtime and touchpoint data
+ organizer approval before public updates
```
