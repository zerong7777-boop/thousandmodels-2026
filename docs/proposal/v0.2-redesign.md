# 智引濠江 v0.2 Redesign Spec

版本：v0.2 design draft  
日期：2026-06-09  
任务：`20260518-thousandmodels-ai-competition`  
项目根目录：`<PROJECT_ROOT>`  
状态：后续实现设计约束，不直接代表当前已实现能力

## 1. 目的

这份 spec 用来修正 v0.1 的核心问题：当前版本能跑通固定 demo loop，但整体仍像展示网页和规则模板，不像一个真实产品，也不足以证明多 Agent 运营协同能力。

v0.2 的目标不是继续堆页面，而是把项目改造成：

> 面向旧区文旅活动主办方的活动运营协同产品，内部由可观察的多 Agent 工作流支撑策划、执行、异常恢复和复盘。

v0.1 保留为 technical scaffold / clickable prototype。v0.2 要补的是产品形态、对象模型、状态闭环和 Agent 证据链。

## 2. 产品重新定位

### 2.1 从展示页转为运营产品

当前页面给人的第一印象是“为比赛专门做的展示站”。v0.2 要改成“每天可以被活动运营人员打开使用的工作台”。

核心变化：

| 维度 | v0.1 问题 | v0.2 方向 |
| --- | --- | --- |
| 首页 | 像项目展示页 | 运营指挥台，直接显示活动、风险、待审批 |
| 活动 | 固定示例流程 | 可管理的活动对象和活动状态 |
| Agent | 最终结果可见，过程较弱 | Agent 步骤、工具输入、工具输出和人工确认可见 |
| 异常 | 按钮触发通知 | 状态变化触发 plan version、商户包、游客端同步更新 |
| 商户 | 辅助展示页 | 能上报状态并影响主办方决策 |
| 游客 | 后台数据预览 | 活动 H5，展示真实路线、故事、任务和临时变更 |
| 复盘 | 总结文本 | 基于日志、指标和事件时间线的运营复盘 |

### 2.2 主要用户

主用户是活动主办方或运营人员。系统首先要服务他们的日常工作：

- 创建和维护旧区活动。
- 审核 AI 生成的活动方案。
- 分派商户任务。
- 处理库存、排队、天气等异常。
- 决定哪些变更可以对游客发布。
- 在活动结束后沉淀复盘。

次用户是商户和游客。商户端负责状态上报和执行包查看；游客端负责接收路线、文化内容、任务和临时通知。

### 2.3 业务问题

本项目不是旅游推荐 App，也不是活动文案生成器。真实问题应定义为：

> 澳门旧区活动的难点不是“推荐去哪玩”，而是主办方要在有限预算、商户承载、天气变化、库存波动、游客体验和旧区秩序之间持续协调。

没有系统时，主办方通常依赖表格、聊天群、人工通知和临时判断。v0.2 要证明 Agent 可以把这个过程变成结构化、可追踪、可恢复的工作流。

## 3. 设计原则

1. Workbench first：首屏就是工作台，不做营销 hero 或功能介绍页。
2. Objects first：前端围绕活动、方案版本、商户、任务、事件、通知、复盘等对象组织，而不是围绕按钮组织。
3. State transitions explicit：所有关键动作都改变可见状态，不能只追加文案。
4. Human approval gates：任何影响游客端和商户执行的变更都需要主办方确认。
5. Agent trace visible：Agent 的输入、工具调用、输出和人工确认要能被评委看到。
6. Deterministic demo remains：不依赖 `DASHSCOPE_API_KEY` 跑通闭环，模型接入放在闭环稳定之后。
7. Product density over showcase：界面要像运营系统，信息密度适中，减少装饰性大卡片和宣传话术。

## 4. v0.2 信息架构

### 4.1 Organizer Home

主办方进入系统后看到运营总览，而不是项目介绍。

必须包含：

- 活动列表：草稿、待确认、执行中、已结束。
- 今日执行：当前活动状态、游客端发布状态、商户响应状态。
- 待审批：方案确认、恢复动作确认、公开通知确认。
- 风险告警：库存、排队、天气、路线拥堵、居民影响。
- 快捷动作：创建活动、查看异常中心、进入复盘中心。

### 4.2 Activity Workspace

单个活动的核心工作区。它不是一张方案展示页，而是活动对象的操作面板。

必须包含：

- 活动 brief：时间、区域、目标客群、预算、约束。
- 方案版本：`v1`、`v2` 等版本状态和 diff。
- 路线与时间表：结构化路线点、停留时间、室内外属性、备用点。
- 商户任务板：每个商户的角色、时段、准备事项、状态。
- Agent workflow：展示多个 Agent step、工具调用、结构化输出。
- 发布控制：哪些内容已发布到游客端，哪些仍需审批。

### 4.3 Exception Center

异常中心用于处理运行期变化，不能只是 dashboard 上的按钮区。

必须包含：

- 异常队列：库存不足、排队过载、强降雨、商户不可接待。
- 影响范围：影响哪些路线点、商户任务、游客通知和预算。
- Recovery proposal：Agent 生成的候选恢复动作。
- 变更预览：确认后将如何修改 plan version、商户包和游客 H5。
- 审批记录：谁确认了什么，何时对外发布。

### 4.4 Merchant Workbench

商户端要形成闭环，而不是只看执行包。

必须包含：

- 当前活动任务：角色、导流时段、准备物料、优惠或互动。
- 状态上报：库存、排队、可接待状态、临时备注。
- 系统通知：主办方确认后的执行包变更。
- 历史状态：最近一次上报时间和上报来源。

### 4.5 Tourist H5

游客端不是后台预览页。它应该像一个真实活动入口。

必须包含：

- 活动标题和当前状态。
- 路线点卡片：站点故事、推荐停留时间、关联商户、室内外属性。
- 打卡任务：游客能理解的任务文本。
- 商户优惠或互动。
- 临时通知：恢复动作确认后同步显示。
- 路线变更原因：用游客能理解的语言说明，而不是暴露后台字段。

### 4.6 Review Center

复盘中心要展示运营证据，而不是只生成一段总结。

必须包含：

- 活动时间线。
- Agent 动作和人工审批时间线。
- 游客 H5 访问、打卡、路线完成等 mock 指标。
- 商户负载、库存异常、排队状态等 mock 指标。
- 预算计划与消耗。
- 下次活动建议，且建议必须引用指标或事件。

## 5. v0.2 Demo Scenario

v0.2 固定演示仍然只做一场活动，避免范围膨胀。

场景：

1. 主办方创建“福隆新街周末旧区夜游”，目标是带动 8 个旧区商户。
2. Agent 读取活动 brief、商户档案、天气 mock、路线点位和预算约束。
3. 多 Agent 生成 `EventPlan v1`，并留下 trace。
4. 主办方确认 `v1`，系统生成商户执行包和游客 H5。
5. 活动运行中，老字号糕点店上报库存售罄。
6. 系统生成库存异常，并由 Recovery Agent 提出 `v2` 候选修改。
7. 主办方查看影响范围和 v1 -> v2 diff，确认恢复动作。
8. 商户执行包更新，游客 H5 更新路线、通知或任务。
9. 随后天气 mock 触发强降雨，Agent 切换室内备用路线。
10. 活动结束后生成复盘报告，报告引用访问、打卡、异常耗时、商户负载和预算指标。

演示要展示“商户 -> Agent -> 主办方 -> 游客”的闭环，而不是“主办方按按钮 -> 页面变文案”。

## 7. Agent Workflow

v0.2 要呈现多 Agent 编排，即使底层仍是 deterministic fallback，也要按真实编排结构输出 trace。

最小 Agent 集合：

| Agent | 输入 | 输出 | 价值 |
| --- | --- | --- | --- |
| OrchestratorAgent | EventBrief | 任务拆解和最终合并 | 证明不是单模板生成 |
| CulturalNarrativeAgent | 片区点位、商户故事 | 主题、站点故事、游客文案 | 贴合澳门旧区文化 |
| RoutePlannerAgent | 点位、时长、室内外、风险 | 路线和备用路线 | 支撑雨天恢复 |
| MerchantMatcherAgent | 商户档案、活动目标、容量 | 商户角色和导流时段 | 支撑商户协同 |
| RiskAgent | 商户状态、天气、路线、预算 | 风险清单和触发条件 | 支撑执行期监控 |
| RecoveryAgent | Incident、当前 PlanVersion | RecoveryProposal | 支撑动态重规划 |
| ReviewAgent | 日志、指标、事件 | ReviewReport | 支撑复盘沉淀 |

Qwen 或 QwenPaw 后续可以接入文化叙事、文案生成、风险解释和复盘表达；预算、库存、路线约束必须先由规则工具校验，避免模型直接决定运营动作。

## 8. Runtime State 与版本机制

v0.2 的恢复逻辑必须体现真实对象变化。

库存异常确认后至少改变：

- `Incident.status`
- `RecoveryProposal.approval_status`
- `PlanVersion.version`
- `MerchantTask.task_status` 或任务内容
- `PublicNotice.publish_status`
- `AgentTrace.human_decision_ref`

雨天异常确认后至少改变：

- 路线点室内外选择。
- 游客端路线卡片。
- 商户导流时段或备用商户。
- 复盘时间线。

页面必须展示 v1 -> v2 diff。diff 不需要做复杂文本比较，最低可显示：

- 替换了哪些路线点。
- 哪些商户任务被取消、改时段或新增。
- 哪些游客通知被发布。
- Agent 给出的原因。
- 主办方确认时间。

## 9. UI 设计约束

v0.2 仍可使用 React/Vite/Ant Design，但页面信息架构必须重做。

### 9.1 整体风格

- 像运营后台，不像宣传页。
- 首屏不放 hero、不放大标题口号、不放功能介绍卡片。
- 使用侧边导航或顶部产品导航，让用户感觉进入了一个工作系统。
- 页面标题应是业务对象，例如“活动运营”“异常中心”“商户工作台”，不是“AI 智能体验”。
- 减少装饰性渐变和大面积单色主题，重点放在状态、列表、时间线和操作。

### 9.2 多使用端边界

v0.2 禁止把主办方、商户、游客、复盘全部堆在一个单页 demo 中。可以共用同一个 React 应用和同一个后端，但用户体验上必须表现为不同使用端或独立工作区。

最低要求：

- 主办方端：面向活动运营人员，包含活动列表、活动工作区、异常中心、审批和复盘入口。
- 商户端：面向单个商户，包含执行包、准备清单、库存/排队/可接待状态上报。
- 游客 H5：面向公众游客，包含活动路线、站点故事、任务、优惠和临时通知。
- 复盘中心：面向主办方，独立展示活动指标、事件时间线和下次优化建议。

实现上可以用同一个前端项目承载这些入口，例如 `/organizer`、`/organizer/events/:id`、`/merchant/:merchantId`、`/public/events/:id`、`/review/:eventId`。但不能用一个页面同时展示所有角色视角，也不能把商户端和游客端做成主办方页面里的静态预览卡片。

### 9.3 主办方界面

推荐布局：

- 左侧：活动列表和过滤状态。
- 中间：当前活动工作区、方案版本、路线、商户任务。
- 右侧：待审批动作、异常影响、Agent trace 摘要。

主办方必须能一眼看到：

- 当前活动状态。
- 当前计划版本。
- 当前风险和异常。
- 哪些动作需要确认。
- 确认后会影响哪些对象。

### 9.4 Agent Evidence Panel

必须增加 Agent 证据面板，用来和普通后台区分。

面板内容：

- Agent step 时间线。
- 每一步的输入引用。
- 工具调用结果摘要。
- 结构化输出。
- 人工确认点。

这部分是比赛展示重点，不能藏在日志页末尾。

### 9.5 商户端

商户端应更像轻量工作台：

- 上方显示今日任务。
- 中间显示准备清单和导流时段。
- 下方是状态上报控件。
- 状态提交后，主办方异常中心应出现对应事件。

### 9.6 游客 H5

游客端应更接近真实活动页面：

- 以路线和站点为主。
- 每个站点有故事、任务、商户和状态。
- 通知必须解释变更原因。
- 不能显示后台术语，例如 `RecoveryAction`、`PlanVersion`。

## 10. v0.2 验收标准

v0.2 完成时必须满足：

1. 首屏是主办方运营工作台，不是展示页。
2. 主办方端、商户端、游客 H5、复盘中心必须是独立入口或独立路由，不能全部堆在一个页面。
3. 商户端和游客 H5 不能只是主办方端里的静态预览，必须能通过各自入口访问真实数据。
4. 能看到活动列表和单个活动 workspace。
5. 能生成 `PlanVersion v1`，并展示多 Agent trace。
6. Agent trace 至少包含 5 个可见 step。
7. 每个 step 至少展示输入、工具结果、结构化输出。
8. 主办方确认 `v1` 后生成商户任务和游客 H5 数据。
9. 商户端库存或排队状态更新能触发 `Incident`。
10. Recovery Agent 能生成 `RecoveryProposal`。
11. 主办方确认恢复后生成 `PlanVersion v2`。
12. 页面能展示 v1 -> v2 diff。
13. 商户执行包根据恢复动作发生可见变化。
14. 游客 H5 根据恢复动作发生可见变化。
15. 复盘报告引用至少 5 个 mock 指标。
16. 所有对外发布动作都有人工确认记录。
17. 本地 demo 不依赖 `DASHSCOPE_API_KEY`。

## 11. 从 v0.1 到 v0.2 的迁移

### 11.1 保留

- FastAPI + Pydantic 后端。
- React + Vite + Ant Design 前端。
- deterministic fallback。
- 基础活动、商户、恢复、复盘对象。
- 主办方确认边界。
- 不接硬件、不接真实商户、不做真实交通预测、不训练模型的范围控制。

### 11.2 重构

- 将单一 deterministic planner 拆成可观察 Agent workflow。
- 将恢复逻辑从“追加通知”改为“生成 proposal + plan patch + 人工确认 + 新版本”。
- 将游客端从后台预览改为真实 H5 数据模型。
- 将商户端从查看页改为可触发状态变化的工作台。
- 将复盘报告从总结文本改为指标驱动报告。
- 将首页从展示页改为活动运营首页。

### 11.3 暂缓

- 真实地图 API。
- 真实天气 API。
- 真实商户系统对接。
- 真实支付、POS、CRM、库存系统。
- Qwen/QwenPaw 强依赖。
- 多片区、多活动并发。
- 手机 App 或硬件设备。

## 12. 后续实施顺序

后续 implementation plan 应按以下顺序拆解：

1. 重写数据对象和测试，先让状态闭环成立。
2. 引入 AgentTrace 和 deterministic multi-agent trace。
3. 实现 PlanVersion 和 v1 -> v2 diff。
4. 实现商户状态上报触发 Incident。
5. 实现 RecoveryProposal 和人工确认。
6. 重做主办方信息架构。
7. 重做商户端和游客 H5。
8. 增加 mock 指标和复盘中心。
9. 最后再评估 Qwen/QwenPaw 接入点。

这个顺序的原因是：先有对象和状态，前端才有产品感；先有 deterministic trace，模型接入才不会变成不可控文本生成。

## 13. 评审叙事

v0.2 对评委要讲清楚三件事：

1. 这是旧区活动运营问题，不是旅游推荐问题。
2. Agent 的价值在于把策划、执行、异常恢复和复盘变成可追踪工作流。
3. 人工确认、版本 diff 和审计记录让 AI 建议不会直接失控发布。

一句话版本：

> 智引濠江把澳门旧区活动从一次性策划文案，转成可执行、可恢复、可复盘的多方协同流程。

## 14. 当前结论

v0.2 不应该追求“页面更炫”或“功能更多”。它要解决的是产品成熟度问题。

如果后续实现只完成一个目标，那就是让评委相信：

> 这个系统不是为比赛临时拼出来的展示页，而是一个可以承载旧区活动运营工作的 Agent 产品雏形。
