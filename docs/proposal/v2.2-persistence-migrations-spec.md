# v2.2 Persistence And Migrations Spec

Date: 2026-06-27

Status: proposed.

## Purpose

v2.2 turns the current local SQLite persistence into a migration-aware data layer. The project currently uses:

- SQLite by default;
- explicit `users` and `sessions` tables;
- a generic `records(collection, item_key, payload)` table for most product objects;
- implicit table creation during `MVPStore` initialization;
- demo reset and seed scripts that assume local state.

That is acceptable for a deterministic MVP, but it is not enough for long-lived beta data. v2.2 introduces schema versioning, migration discipline, store mode boundaries, and documented backup/reset behavior without rewriting every domain model into relational tables.

## Readiness Target

v2.2 moves persistence from "implicit local store" to "migration-managed beta store".

It remains SQLite-compatible for local and staging tests. It does not claim high-concurrency production database readiness.

## In Scope

- Add migration directory under `apps/api/app/migrations/`.
- Add `schema_migrations` table.
- Add a migration runner callable from startup and scripts.
- Replace implicit schema creation with migration-managed creation for current tables.
- Add store metadata health reporting: database path, schema version, pending migrations.
- Add a CLI command or script to run migrations explicitly.
- Add demo reset safeguards so reset is local/demo only.
- Add tests for fresh database migration, idempotent migration, pending migration detection, and demo reset boundary.
- Update `.env.example`, README, and docs/ai.

## Out Of Scope

- Full relational decomposition of all domain objects.
- PostgreSQL production deployment.
- Multi-tenant row-level isolation.
- Backup service integration.
- Data retention policy beyond documenting current local behavior.
- Online migration rollback automation.

## Required Behavior

### Migration Model

Migrations are versioned Python modules or SQL files. v2.2 should prefer Python modules so tests can run without extra dependencies.

Required migration table:

```sql
CREATE TABLE IF NOT EXISTS schema_migrations (
  version TEXT PRIMARY KEY,
  applied_at TEXT NOT NULL
)
```

Initial migrations:

| Version | Responsibility |
| --- | --- |
| `0001_initial_records` | Create `records`. |
| `0002_auth_tables` | Create `users`, `sessions`, and auth indexes. |

### Store Initialization

`MVPStore` must not silently drift schema without recording migration state. On initialization it should:

1. connect to the configured database;
2. ensure `schema_migrations`;
3. apply known migrations by default in local/demo;
4. optionally refuse startup in production if migrations are pending and `AUTO_MIGRATE=false`.

### Health Surface

`GET /api/health` must include:

- app status;
- agent backend;
- schema version;
- pending migration count;
- store kind, such as `sqlite`;
- no local absolute path leakage in production mode.

### Scripts

Add:

- `apps/api/scripts/migrate_store.py`
- optional `apps/api/scripts/store_status.py`

The migration script must be safe to run repeatedly.

### Demo Reset Boundary

`apps/api/scripts/reset_demo_state.py` and any API reset path must refuse to run unless demo/local mode is enabled.

## Tests

Backend tests must cover:

- new empty database migrates to latest schema;
- repeated migration run is idempotent;
- existing `records`, `users`, and `sessions` operations still work;
- pending migrations are detected;
- health endpoint reports schema metadata;
- demo reset refuses when demo mode is disabled.

## Acceptance Criteria

- API full pytest passes.
- CI remains green.
- Migration script runs successfully on a fresh temp database.
- Existing deterministic demo flow still works.
- Documentation clearly states that v2.2 is migration readiness, not production database scaling.

## Commercial Readiness Impact

v2.2 closes most of v1.9 `CR-P0-004`: schema ownership and migration discipline. It does not close production database operations, backups, or tenant isolation.
