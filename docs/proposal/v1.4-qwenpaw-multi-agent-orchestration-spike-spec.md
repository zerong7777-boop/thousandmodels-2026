## 1. Purpose

v1.3 made the demo runnable through a real local FastAPI backend and Vite frontend. The project now has a stable deterministic operations loop:

```text
organizer plan approval
-> event page publication
-> merchant edge package generation
-> merchant runtime signal
-> organizer exception handling
-> public scan / claim / redeem
-> review evidence
```

The next useful step is not another competition delivery package. v1.4 should test whether QwenPaw and multi-agent orchestration add real engineering value to this loop.

This is an intentional branch from the previous `docs/ai/NEXT.md` recommendation to prepare competition-facing material. The user has explicitly deprioritized a v1.4 delivery package and asked to start the QwenPaw / multi-agent orchestration spike first.

The target stance:

```text
QwenPaw may coordinate, delegate, draft, and explain.
QwenPaw must not become the authoritative operations engine in this spike.
```

v1.4 should therefore build a narrow, auditable spike:

```text
optional QwenPaw-style shadow orchestration
+ supervisor / worker delegation evidence
+ permissioned tool contracts
+ deterministic replay or fake/recorded runtime
+ organizer-only evidence
+ unchanged v1.3 deterministic demo path
```

## 3. Problem statement

The project can already show specialist Agent evidence, but the current Agent runtime is still mostly a deterministic, hard-coded sequence. That is acceptable for demo reliability, but it leaves a gap:

```text
The system shows Agent evidence, but it does not yet prove a real orchestration boundary.
```

There are two bad ways to close that gap.

First, v1.4 could overclaim by renaming existing deterministic steps as QwenPaw. That would not prove multi-agent orchestration.

Second, v1.4 could integrate a live external runtime too deeply and let it mutate business state. That would endanger the strongest existing asset: a reliable, reproducible, approval-bound demo.

v1.4 should take the middle path. It should prove the orchestration contract first, using a fake or recorded QwenPaw-style runtime, then leave real QwenPaw provider execution as a guarded optional smoke.

## 4. Positioning decision

v1.4 is:

- a multi-agent orchestration spike
- a QwenPaw adapter contract phase
- a shadow orchestration phase
- a permissioned tool registry phase
- an evidence and replay phase
- an organizer-only visibility phase

v1.4 is not:

- a competition slide deck, recording, or final delivery package phase
- a production QwenPaw migration
- an autonomous operations phase
- a required Qwen, DashScope, or QwenPaw network dependency
- a model-generated authoritative plan, publication, notice, inventory, coupon, or redemption phase
- a real merchant, POS, payment, hardware, map, weather, traffic, or identity integration phase
- a public/merchant Agent debugging UI phase

The v1.4 product claim should be:

```text
The system can run an optional QwenPaw-style shadow orchestration for one merchant incident.
The orchestration delegates work across specialist Agents, records permissioned tool evidence,
and returns advisory drafts without changing authoritative operations.
```

The claim should not be:

```text
QwenPaw now orchestrates the live business workflow.
```

That claim requires a later phase with real provider evidence, latency control, failure handling, security review, and live smoke success.

## 5. Mature patterns to borrow

### 5.1 Recommended: StateGraph-style supervisor and workers

Use a project-local StateGraph-style workflow as the first implementation pattern.

The important ideas to borrow:

- state is explicit and accumulates evidence
- each node has a clear role
- edges determine the next specialist step
- tools are called through a controlled registry
- the final result is structured, not a chat transcript

This matches the current backend better than a broad framework migration. `AgentRuntimeResult` already accumulates runs, steps, tool calls, drafts, model calls, and evaluations. v1.4 should make that accumulation more declarative and reusable.

### 5.2 Recommended: QwenPaw team-leader and worker model

QwenPaw should influence the role model:

```text
CentralOpsLeader
-> MerchantEdgeWorker
-> FieldOpsWorker
-> PublicNoticeWorker
-> ReviewEvidenceWorker
-> SafetyGateWorker
```

The Leader decomposes and aggregates. Workers inspect bounded state and return structured advisory outputs.

This should be implemented first as a local fake/recorded adapter. Real QwenPaw runtime execution remains optional until the contract is stable.

### 5.3 Recommended: permissioned tools and guards

QwenPaw/AgentScope-style tool guard ideas are directly useful. v1.4 should classify tools before any Agent can request them:

| Permission class | Meaning | v1.4 default |
| --- | --- | --- |
| `read_only` | inspect event, plan, incident, touchpoint, metric, and merchant snapshots | allowed |
| `draft_only` | create advisory bundle, draft copy, or evidence note | allowed |
| `approval_required` | approve, publish, recover, mutate public state, or apply operations | denied to shadow runtime |
| `forbidden` | secrets, auth, payment, POS, hardware, private visitor identity, real external integrations | denied |

Every requested tool call should produce a permission decision, even if denied.

### 5.4 Defer: AutoGen/CrewAI-style role chats

AutoGen and CrewAI are useful references for multi-agent teams and flows, but v1.4 should not make free-form role chat the primary evidence. The project needs structured, replayable evidence more than a long conversation transcript.

### 5.5 Defer: full external QwenPaw runtime migration

AgentScope and QwenPaw expose a larger runtime surface: agents, tools, permissions, events, sessions, and sandboxing. That is valuable, but it overlaps with the current FastAPI store, auth, audit, and evidence model. v1.4 should add an adapter seam first, then decide whether a later phase should migrate runtime ownership.

## 6. Scope

v1.4 includes:

- visitor identity collection
- public or merchant display of raw QwenPaw/model/backend evidence

## 7. Target spike scenario

The v1.4 scenario should use the existing v1.3 business loop:

```text
1. Reset demo state.
2. Organizer generates and approves plan v1.
3. Organizer publishes Event Page.
4. Organizer generates merchant edge packages.
5. Merchant reports sold out.
6. Backend creates incident and existing recovery/operation evidence.
7. Organizer triggers QwenPaw shadow orchestration.
8. CentralOpsLeader delegates to specialist workers.
9. Workers inspect read-only snapshots and produce advisory outputs.
10. SafetyGateWorker evaluates permissions and public-copy boundaries.
11. Shadow orchestration records evidence.
12. Organizer still uses existing approval APIs for any real operational change.
13. Public H5 sees only visitor-safe published output.
```

The shadow orchestration output should include:

```text
recovery_rationale
visitor_safe_notice_draft
merchant_impact_note
review_evidence_refs
permission_decisions
human_approval_required=true
authoritative_mutation=false
```

## 8. Target architecture

### 8.1 Workflow contract

Introduce a small internal contract:

```text
WorkflowState
  event_id
  run_id
  trigger
  input_refs
  snapshots
  steps
  tool_calls
  drafts
  model_calls
  evaluations
  advisory_bundle

WorkflowNode
  node_id
  agent_name
  objective
  run(state) -> WorkflowNodeResult

WorkflowSpec
  workflow_id
  trigger
  nodes
  edges
  allowed_tool_permissions

WorkflowExecutor
  execute(spec, state) -> AgentRuntimeResult
```

This is not a general workflow engine. It is a small adapter seam for v1.4.

### 8.2 Tool registry

Introduce a registry that wraps tool calls before they reach Agent evidence:

```text
ToolDefinition
  name
  permission
  input_schema_name
  output_schema_name
  handler

ToolRequest
  tool_name
  requested_by_agent
  input_payload

ToolDecision
  allowed
  permission
  reason
```

The registry should record allowed read-only calls and denied mutation calls. Denied calls must not execute the handler.

Suggested initial tools:

| Tool | Permission | Purpose |
| --- | --- | --- |
| `event.get_current_plan_snapshot` | `read_only` | inspect current approved plan |
| `incident.get_active_incident_snapshot` | `read_only` | inspect incident |
| `merchant.get_runtime_snapshot` | `read_only` | inspect merchant runtime state |
| `touchpoints.get_metric_snapshot` | `read_only` | inspect anonymous touchpoint metrics |
| `operation.build_advisory_bundle` | `draft_only` | create structured advisory bundle |
| `operation.apply_suggestion` | `approval_required` | denied in v1.4 shadow runtime |
| `event.publish_notice` | `approval_required` | denied in v1.4 shadow runtime |
| `merchant.update_runtime_state` | `approval_required` | denied in v1.4 shadow runtime |

### 8.3 QwenPaw adapter

Introduce an adapter interface:

```text
QwenPawWorkflowAdapter
  run_shadow_incident_workflow(context) -> QwenPawWorkflowResult
```

Implementations:

```text
FakeQwenPawWorkflowAdapter
RecordedQwenPawWorkflowAdapter
LiveQwenPawWorkflowAdapter
```

v1.4 acceptance requires the fake or recorded adapter. The live adapter is optional and must be guarded.

The adapter result must be schema-validated before it becomes evidence. It may contain:

```text
leader_decision
worker_outputs
advisory_bundle
permission_requests
safety_notes
```

It must not contain authoritative backend objects such as approved plans, published notices, runtime mutations, or coupon transactions.

### 8.4 Runtime integration

Keep `AgentRuntime` as the public backend entry point. Add a new method:

```text
run_qwenpaw_shadow_orchestration(event_id, incident, context_snapshots) -> AgentRuntimeResult
```

This method should:

- create `AgentRun(mode="qwenpaw_workflow", trigger="incident_recovery")`
- execute the local workflow contract
- use the fake/recorded QwenPaw adapter
- record Leader and Worker steps
- record permissioned tool calls
- record advisory drafts and evaluations
- set `fallback_used=true` only when adapter execution fails and deterministic advisory fallback is used
- never call existing approval or publication mutation services

### 8.5 API integration

Add an organizer-only endpoint:

```text
POST /api/events/{event_id}/qwenpaw-shadow-orchestration/run
```

The endpoint should:

- require organizer auth
- require mutation origin checks
- find the latest active incident or accept a bounded `incident_id`
- run the shadow orchestration
- persist the `AgentRuntimeResult`
- return the persisted run and advisory summary

Suggested response:

```text
{
  "agent_run": AgentRun,
  "advisory_bundle": {...},
  "steps": [AgentStep],
  "permission_decisions": [...]
}
```

The organizer page may show:

## 9. Validation and safety

Every shadow workflow output must pass:

```text
adapter output
-> JSON/schema validation
-> unsafe mutation key scan
-> permission decision check
-> public-copy forbidden-term check for visitor-facing drafts
-> human approval required check
-> wrap into Agent evidence
-> persist evidence
```

Unsafe mutation fields include:

```text
approval_status
publish_status
approved_by
approved_at
inventory_status
queue_status
available_for_visitors
plan_patch
merchant_task_patch
public_notice.publish_status
create_plan_version
approve_recovery
publish_notice
apply_suggestion
coupon_redemption
coupon_claim
claim_coupon
coupon_claim_status
redeem_coupon
redemption_status
payment
pos_transaction
visitor_identity
```

If unsafe mutation is detected:

- record `AgentEvaluation.unsafe_mutation_attempted=true`
- do not persist the unsafe draft content as accepted evidence
- use deterministic advisory fallback
- keep `human_approval_required=true`

## 10. Configuration

Use a separate orchestration configuration to avoid confusion with existing draft generation:

```text
AGENT_ORCHESTRATION_BACKEND=deterministic | qwenpaw_fake | qwenpaw_recorded | qwenpaw_live
```

Default:

```text
AGENT_ORCHESTRATION_BACKEND=deterministic
```

Behavior:

| Configuration | Behavior |
| --- | --- |
| unset | deterministic Agent runtime, no QwenPaw shadow run unless endpoint explicitly uses fake default |
| `qwenpaw_fake` | deterministic fake adapter, accepted v1.4 main path |
| `qwenpaw_recorded` | replay a sanitized recorded adapter result, accepted v1.4 main path |
| `qwenpaw_live` without required credentials/runtime | return blocked or fallback evidence; do not fail default demo |
| `qwenpaw_live` with provider success | optional evidence only, not required for v1.4 acceptance |

Do not reuse `AGENT_DRAFT_BACKEND`; that variable controls Qwen controlled drafts, not workflow orchestration.

## 14. Acceptance criteria

v1.4 spec is complete when:

1. `docs/proposal/v1.4-qwenpaw-multi-agent-orchestration-spike-spec.md` exists.
2. It states v1.4 is not a competition delivery package.
3. It defines QwenPaw as optional shadow orchestration, not authoritative workflow execution.

## 15. Stop gates

Stop and ask for review if:

- implementation would require live QwenPaw or DashScope credentials for the normal demo
- QwenPaw output would need to approve, publish, or mutate authoritative state
- tool permissions cannot block mutation calls
- schema validation cannot reject unsafe adapter output
- public or merchant pages need raw Agent/QwenPaw terms to make the demo pass
- evidence would need to store API keys, raw provider payloads, Authorization headers, or local absolute paths
- implementation drifts into real POS, payment, hardware, map, weather, traffic, or identity integrations
- workflow extraction requires a large rewrite of existing planning, recovery, merchant edge, or review services

## 16. Recommended implementation phases

### Phase A: Contract and tests

Write failing tests for workflow execution, tool permissions, fake adapter output, unsafe output fallback, and non-mutation boundaries.

### Phase B: Workflow executor and tool registry

Implement a small StateGraph-style executor and permissioned tool registry without touching public business APIs.

### Phase C: QwenPaw fake/recorded adapter

Implement the adapter contract and deterministic shadow incident workflow.

### Phase D: API and organizer evidence

Add an organizer-only endpoint and, if useful for demo clarity, a small organizer UI trigger and evidence display.

### Phase E: Evidence and optional live smoke

Record sanitized shadow-run evidence. Run optional live provider smoke only when local credentials/runtime exist.

### Phase F: Verification and handoff

Run full backend, frontend, mocked Playwright, v1.3 live smoke, boundary scans, and update docs/ai.
