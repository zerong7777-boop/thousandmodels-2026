# v3.0 Merchant Network Pack Spec

Status: planned.

## Goal

v3.0 turns the thin local merchant catalog into an organizer-managed merchant network. Organizers should be able to create, edit, inspect, and reuse merchant profiles, then use those profiles to decide whether a merchant is eligible and ready for an event roster before route planning.

This phase makes the product closer to a real operations tool while staying inside the current local store, demo auth, and deterministic planning boundaries.

## Background

v2.7 introduced non-demo event lifecycle APIs.
v2.8 exposed event portfolio, event creation, and selected-event workspace navigation.
v2.9 added event-level merchant roster setup and blocked non-demo route planning until selected merchants are ready.

The remaining gap is that the merchant catalog is still a static seed list. It does not represent merchant ownership, contact data, operating windows, capacity notes, participation constraints, status, or participation history. That makes v2.9's roster gate operationally useful but not yet commercially credible.

## Scope

v3.0 delivers a Merchant Network Pack:

1. Extended merchant profile fields with safe defaults for seeded data.
2. Organizer-only merchant create, update, list, and detail APIs.
3. Merchant participation history derived from event rosters and plans.
4. Merchant eligibility evaluation for selected event rosters.
5. Organizer merchant network page with list, create, edit, detail, and eligibility-facing fields.
6. v2.9 roster panel enrichment using merchant profile status, operating windows, and participation constraints.
7. Deterministic demo compatibility for `demo-night-tour`.

## Non-Goals

- No public merchant self-registration.
- No invitation email, SMS, or merchant onboarding workflow.
- No merchant identity administration beyond the existing demo auth boundary.
- No contracts, settlement, payment, POS, CRM, ERP, hardware, map, traffic, weather, or visitor identity integrations.
- No tenant or organization hierarchy.
- No file uploads, compliance document management, or KYC.
- No production QwenPaw orchestration.
- No broad visual redesign or marketing page.

## Product Requirements

### Merchant Network Profile

- `REQ-MN-001`: A merchant profile must remain backward compatible with existing seeded `MerchantProfile` records.
- `REQ-MN-002`: A merchant profile must include editable network fields:
  - `contact_name`
  - `contact_phone`
  - `address_label`
  - `area`
  - `operating_windows`
  - `capacity_notes`
  - `category_tags`
  - `participation_constraints`
  - `status`
- `REQ-MN-003`: Merchant `status` values must be `active`, `inactive`, or `suspended`.
- `REQ-MN-004`: `operating_windows` must be structured enough for event-window overlap checks.
- `REQ-MN-005`: Empty or legacy profiles must be accepted, but missing network fields should make eligibility less confident rather than crash existing flows.
- `REQ-MN-006`: `merchant_id` must remain stable and unique.
- `REQ-MN-007`: Create and update payloads must reject empty merchant names, invalid capacity levels, invalid scores, invalid status values, and malformed operating windows.

### Merchant Management API

- `REQ-API-001`: Organizer users can list merchants through `GET /api/merchants`.
- `REQ-API-002`: Organizer users can read merchant detail through `GET /api/merchants/{merchant_id}`.
- `REQ-API-003`: Organizer users can create merchants through `POST /api/merchants`.
- `REQ-API-004`: Organizer users can update merchants through `PATCH /api/merchants/{merchant_id}`.
- `REQ-API-005`: Merchant create rejects duplicate `merchant_id`.
- `REQ-API-006`: Merchant update rejects unknown `merchant_id`.
- `REQ-API-007`: Merchant create seeds a default runtime state so existing merchant workbench/runtime paths can operate.
- `REQ-API-008`: Merchant list and detail must not expose secrets, auth hashes, cookies, or model/provider payloads.

### Participation History

- `REQ-HIST-001`: Merchant detail must include participation history derived from stored event merchant roster entries, event summaries, and plan/package state where available.
- `REQ-HIST-002`: History items should include event id, event title, event date, participation status, readiness status, latest plan version if any, and whether an interaction package exists.
- `REQ-HIST-003`: History is derived from existing local records; no new analytics pipeline is required.

### Event Roster Eligibility

- `REQ-ELIG-001`: Event roster summaries must expose merchant eligibility for each selected participant.
- `REQ-ELIG-002`: Eligibility must consider:
  - merchant status;
  - event time window versus merchant operating windows;
  - missing network profile fields;
  - participation constraints.
- `REQ-ELIG-003`: A selected merchant can be marked ready only when the eligibility result is `eligible` or explicitly `needs_review`.
- `REQ-ELIG-004`: A selected merchant with `inactive` or `suspended` status must not be ready for planning.
- `REQ-ELIG-005`: If event and merchant operating windows do not overlap, the merchant must not be ready for planning.
- `REQ-ELIG-006`: `ready_for_planning` remains true only when every selected merchant is confirmed, ready, and not ineligible.
- `REQ-ELIG-007`: Demo compatibility remains: `demo-night-tour` without a manual roster can still use the seeded catalog.

### Organizer Merchant Network UI

- `REQ-UI-001`: Add `/organizer/merchants` as an organizer shell route.
- `REQ-UI-002`: Add a Merchant Network item to organizer navigation.
- `REQ-UI-003`: The page must list merchants with status, area, capacity level, operating window summary, category tags, and contact owner.
- `REQ-UI-004`: Organizers can create a merchant from the page.
- `REQ-UI-005`: Organizers can edit key profile fields from the page.
- `REQ-UI-006`: The page must show participation history for the selected merchant or inline detail.
- `REQ-UI-007`: The page must use the existing dense operations style, not a landing-page layout.

### Event Workspace Integration

- `REQ-WORKSPACE-001`: The v2.9 merchant setup panel must show merchant status, area, operating window, and constraint hints.
- `REQ-WORKSPACE-002`: Ineligible merchants must show a visible reason.
- `REQ-WORKSPACE-003`: The Mark ready action must not silently succeed for ineligible merchants.
- `REQ-WORKSPACE-004`: Existing v2.8/v2.9 workspace tests must continue passing.

### Internationalization

- `REQ-I18N-001`: New visible text must have explicit entries in `en`, `zh-Hans`, and `zh-Hant`.
- `REQ-I18N-002`: Existing i18n source gates must continue passing.

## API Shape

### Existing Enhanced Endpoints

- `GET /api/merchants`
  - Returns enhanced `MerchantProfile` rows.
  - Initializes local catalog if empty, without creating a demo event.

- `GET /api/events/{event_id}/merchant-roster`
  - Returns `EventMerchantSetupSummary` with per-participant eligibility.

- `PATCH /api/events/{event_id}/merchant-roster/{merchant_id}`
  - Rejects ready updates for ineligible merchants.

### New Endpoints

- `GET /api/merchants/{merchant_id}`
  - Returns `MerchantDetail`.

- `POST /api/merchants`
  - Request: `MerchantCreateRequest`.
  - Returns `MerchantDetail`.

- `PATCH /api/merchants/{merchant_id}`
  - Request: `MerchantUpdateRequest`.
  - Returns `MerchantDetail`.

## Data Model

### MerchantOperatingWindow

- `label: str`
- `start_time: str`
- `end_time: str`

`start_time` and `end_time` use `HH:MM` 24-hour format. Overnight ranges are out of scope for v3.0 and must be rejected.

### MerchantProfile Additions

- `contact_name: str = ""`
- `contact_phone: str = ""`
- `address_label: str = ""`
- `area: str = ""`
- `operating_windows: list[MerchantOperatingWindow] = []`
- `capacity_notes: str = ""`
- `category_tags: list[str] = []`
- `participation_constraints: list[str] = []`
- `status: Literal["active", "inactive", "suspended"] = "active"`

Existing fields remain available:

- `merchant_id`
- `name`
- `type`
- `location`
- `opening_hours`
- `capacity_level`
- `signature_products`
- `story`
- `suitable_activity_types`
- `rainy_day_score`
- `night_score`
- `constraints`

### MerchantEligibility

- `merchant_id: str`
- `status: Literal["eligible", "needs_review", "ineligible"]`
- `reasons: list[str]`

### MerchantParticipationHistoryItem

- `event_id: str`
- `event_title: str`
- `event_date: str`
- `participation_status: str`
- `readiness_status: str`
- `latest_plan_version: int`
- `has_interaction_package: bool`

### MerchantDetail

- `merchant: MerchantProfile`
- `participation_history: list[MerchantParticipationHistoryItem]`

## Eligibility Rules

Eligibility must be deterministic and locally testable.

1. If `merchant.status` is `inactive` or `suspended`, eligibility is `ineligible`.
2. If the event time window cannot be parsed, eligibility is `needs_review`.
3. If merchant operating windows are empty and `opening_hours` cannot be parsed into a window, eligibility is `needs_review`.
4. If no merchant operating window overlaps the event time window, eligibility is `ineligible`.
5. If contact or area fields are missing, eligibility is `needs_review`.
6. If participation constraints exist but do not directly block the event, eligibility is `needs_review`.
7. Otherwise eligibility is `eligible`.

For v3.0, `needs_review` merchants may be marked ready by the organizer. `ineligible` merchants may not be marked ready.

## Error Handling

- Duplicate merchant id: `409 merchant already exists`.
- Unknown merchant id: `404 merchant not found`.
- Invalid merchant payload: standard FastAPI/Pydantic `422`.
- Readying an ineligible roster merchant: `400 merchant is not eligible for this event`.
- Unauthorized or wrong role access: existing auth/authorization error behavior.

## Test Requirements

- Backend tests for merchant create, duplicate rejection, update, detail, and default runtime state creation.
- Backend tests for participation history derived from event rosters.
- Backend tests for eligibility: active overlap, inactive/suspended rejection, non-overlap rejection, missing profile `needs_review`.
- Backend tests that ineligible merchants cannot be marked ready for event planning.
- Backend regression tests that v2.9 ready roster and demo compatibility still work.
- Frontend tests for `/organizer/merchants` route, nav item, list, create/edit flow, and merchant detail/history.
- Frontend tests that event roster panel shows eligibility and blocks ineligible ready action.
- i18n source and dictionary tests continue passing.
- Build, repo hygiene, and whitespace checks continue passing.

## Acceptance Criteria

v3.0 is complete when:

1. Organizers can manage merchants through backend APIs and frontend UI.
2. Merchant profiles carry contact, operating, capacity, tag, constraint, and status fields.
3. Merchant detail shows local participation history.
4. Event roster readiness considers merchant network eligibility.
5. Ineligible merchants cannot be silently marked ready.
6. Demo and existing deterministic release paths remain compatible.
7. Focused backend/frontend tests, full relevant regressions, build, repo hygiene, and whitespace checks pass.

## Follow-Up Product Packages

After v3.0, likely follow-ups are:

- `v3.1-event-planning-eligibility-pack`: use merchant network data deeper inside route scoring and plan generation.
- `v3.2-merchant-portal-setup-pack`: allow merchants to review and maintain their own profile within the existing auth boundary.
- `v3.3-integration-readiness-pack`: define a connector abstraction for future CRM/POS/map integrations without implementing those integrations yet.
