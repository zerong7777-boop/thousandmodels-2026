> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Produce a v1.9 commercial-readiness audit pack that converts the current P0/P1 production gaps into evidence-backed risk, remediation, and next-phase documents.

**Architecture:** Keep v1.9 documentation-first. Gather local command evidence and GitHub state, write a readiness matrix JSON artifact, write a Markdown audit report, and update `docs/ai` handoff files. Do not implement production infrastructure, auth rewrites, database migrations, or live QwenPaw product wiring in this phase.

**Tech Stack:** Markdown, JSON evidence artifacts, PowerShell, `git`, pytest, Vitest, Vite build, GitHub connector or `gh` fallback, `rg` scans.

## 0. Scope lock

v1.9 is an audit and evidence pack. It must not change product runtime behavior.

Do not modify:

- `apps/api/app/main.py`
- `apps/api/app/auth.py`
- `apps/api/app/store.py`
- `apps/api/app/seed.py`
- `apps/api/app/agents/*`
- `apps/web/src/*`
- `.env.example`

Do not add production infrastructure in this phase:

- no database migration implementation
- no Docker or cloud deployment implementation
- no real merchant, POS, payment, map, weather, traffic, or identity integration
- no live QwenPaw product wiring
- no CI workflow unless the user explicitly asks to move from audit to remediation

Historical PNG diffs in `docs/research/assets/**` are not v1.9 outputs. Do not
stage them unless the user explicitly asks.

## 2. Implementation decisions

1. The audit uses a four-value status scale: `pass`, `partial`, `gap`,
   `blocked`.
2. Readiness levels are `CR0`, `CR1`, `CR2`, and `CR3`. The current product
   can be classified as `CR0`; v1.9 can recommend work toward `CR1`.
3. v1.9 stores evidence as bounded JSON plus Markdown. It does not store raw
   provider payloads, secrets, local absolute paths, or screenshots.
4. GitHub state is captured from local `git` plus the GitHub connector. If the
   connector is unavailable, `gh` may be used as a fallback after checking
   authentication.
5. The audit matrix records local file evidence for every P0 and P1 gap.
6. Test/build commands are rerun in this phase. If a command cannot run because
   of environment failure, record the exact failure and classify it as an audit
   blocker, not a product failure.
