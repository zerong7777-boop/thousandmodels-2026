# v3.1 Event Planning Eligibility Pack Spec

Status: implemented on 2026-06-29.

## Goal

v3.1 makes non-demo route planning use merchant-network data as planning intelligence, not only as a readiness gate. The planner should rank eligible merchants with deterministic fit evidence, include explainable selection rationales in generated plan artifacts, and warn organizers when a ready merchant is eligible but weak for the event.

This phase keeps the current local store, demo auth, deterministic release path, and human approval boundary intact.

## Background

v2.7 introduced non-demo event lifecycle APIs.
v2.8 exposed organizer event creation and selected-event workspace navigation.
v2.9 added event-level merchant roster setup and blocked planning until selected merchants are ready.
v3.0 added organizer-managed merchant profiles and event roster eligibility.

The remaining planning gap is that `generate_plan_version()` still picks the first scoped merchants and the agent evidence still uses a generic night-score selection tool. That means merchant profile fields such as category tags, operating windows, capacity notes, and participation constraints do not yet influence route planning output.

## Scope

v3.1 delivers an Event Planning Eligibility Pack:

1. Deterministic merchant-fit scoring for non-demo route planning.
2. Planner warnings for eligible but weak-fit merchants.
3. Explainable merchant selection rationales in `PlanVersion`.
4. Merchant-fit evidence in planning Agent steps.
5. Organizer workspace visibility for selected merchant rationale and planner warnings.
6. Regression coverage that the deterministic `demo-night-tour` path and v2.9/v3.0 gates remain compatible.

## Non-Goals

- No live Qwen, QwenPaw, DashScope, or model-based scoring.
- No external weather, map, traffic, POS, payment, CRM, ERP, visitor identity, or hardware integrations.
- No public merchant onboarding.
- No route geometry optimization.
- No machine learning model training.
- No automatic plan approval or public publication.
- No merchant removal, archive, contract, settlement, or compliance workflow.

## Product Requirements

### Merchant-Fit Scoring

- `REQ-FIT-001`: The backend must compute a deterministic merchant-fit result for every merchant in the ready planning scope.
- `REQ-FIT-002`: Fit scoring must consider:
  - merchant eligibility from v3.0;
  - event time-window overlap;
  - merchant category tags;
  - merchant suitable activity types;
  - merchant capacity level and capacity notes;
  - merchant night score and rainy-day score;
  - event target audience, theme preferences, goal, constraints, and priority rules;
  - participation constraints.
- `REQ-FIT-003`: Fit scoring must produce:
  - `merchant_id`;
  - `score`;
  - `fit_level` as `strong`, `medium`, or `weak`;
  - `matched_signals`;
  - `warnings`;
  - `rationale`.
- `REQ-FIT-004`: Ineligible merchants must not be selected for plan assignments.
- `REQ-FIT-005`: Weak but eligible merchants may remain in the roster but must produce visible planner warnings.
- `REQ-FIT-006`: Scoring must be deterministic and locally testable.

### Plan Output

- `REQ-PLAN-001`: `PlanVersion` must expose merchant fit evidence keyed by merchant id or as a list with stable ordering.
- `REQ-PLAN-002`: `PlanVersion.merchant_assignments` must be ordered by planner fit rather than raw catalog order for non-demo planning.
- `REQ-PLAN-003`: `PlanVersion` must expose planner warnings for weak-fit or needs-review merchants.
- `REQ-PLAN-004`: Existing plan consumers must remain backward compatible when old stored plan versions do not include the new fields.
- `REQ-PLAN-005`: Route point `linked_merchants` must continue to be scoped to selected plan merchants.

### Agent Evidence

- `REQ-AGENT-001`: Planning Agent evidence must include merchant-fit scoring output.
- `REQ-AGENT-002`: The merchant matching step must explain why selected merchants fit the event.
- `REQ-AGENT-003`: Agent evidence must remain deterministic and must not claim live model reasoning.

### Organizer UI

- `REQ-UI-001`: Organizer workspace must show selected plan merchant rationale when a current plan exists.
- `REQ-UI-002`: Organizer workspace must show planner warnings when weak-fit merchants are present.
- `REQ-UI-003`: UI copy must stay operations-focused and dense; no landing page.
- `REQ-UI-004`: New visible copy must use the existing i18n dictionary layer.

### Compatibility

- `REQ-COMPAT-001`: `demo-night-tour` remains deterministic and does not require manual roster setup.
- `REQ-COMPAT-001a`: The legacy `demo-night-tour` no-roster fallback preserves catalog-order merchant assignment while still recording fit evidence.
- `REQ-COMPAT-002`: v2.9 roster readiness remains the hard planning gate for non-demo events.
- `REQ-COMPAT-003`: v3.0 merchant eligibility remains the hard ineligible gate before fit scoring.
- `REQ-COMPAT-004`: Existing event-page, merchant-edge package, public interaction, and review flows continue to read plan merchant assignments.

## Proposed Data Model

### MerchantFitResult

- `merchant_id: str`
- `score: int`
- `fit_level: Literal["strong", "medium", "weak"]`
- `matched_signals: list[str]`
- `warnings: list[str]`
- `rationale: str`

### PlanVersion Additions

- `merchant_fit: list[MerchantFitResult] = []`
- `planner_warnings: list[str] = []`

Defaults preserve compatibility for existing stored plan versions.

## Deterministic Scoring Rules

The exact numbers may be tuned during implementation, but the scoring contract must stay stable and understandable:

1. Start each eligible merchant at 50 points.
2. Add points for night score and rainy-day score.
3. Add points for capacity level.
4. Add points when `category_tags`, `type`, `suitable_activity_types`, or `signature_products` match event target audience, theme preferences, event goal, constraints, or priority rules.
5. Add points when operating windows overlap the event time window.
6. Subtract points for missing contact owner, missing area, missing operating windows, or participation constraints.
7. Mark `strong` at 75 or above, `medium` from 55 to 74, and `weak` below 55.
8. Produce warnings for weak fit, needs-review eligibility, missing profile data, tight operating-window overlap, and participation constraints.

The implementation should keep these rules in a focused service module so the scoring can be tested without starting FastAPI.

## API Behavior

No new public endpoint is required.

Existing enhanced behavior:

- `POST /api/events/{event_id}/generate-plan`
  - Builds the non-demo merchant scope through v2.9/v3.0 gates.
  - Computes merchant-fit results.
  - Orders selected merchants by fit score.
  - Stores fit evidence and planner warnings on the `current_plan`.
  - Returns the enhanced `current_plan`.

- `GET /api/events/{event_id}/plan-versions`
  - Returns stored plan versions with fit evidence when available.

## Error Handling

- If roster setup is incomplete, existing `400 event merchant setup required` behavior remains.
- If all ready merchants are ineligible, planning remains blocked by v3.0 readiness summary.
- If all ready merchants are weak but eligible, planning may proceed but must include planner warnings.
- If old plan records lack v3.1 fields, response validation must use defaults and not fail.

## Test Requirements

- Backend unit tests for deterministic merchant-fit scoring:
  - strong fit from matching tags/activity/capacity/time;
  - weak fit warnings from missing profile fields and constraints;
  - ineligible merchants excluded.
- Backend API tests that non-demo plan generation orders assignments by fit score.
- Backend API tests that generated plan versions include merchant rationales and warnings.
- Backend regression tests for v2.9/v3.0 readiness and demo compatibility.
- Frontend tests that organizer workspace renders selected merchant rationale and planner warnings.
- Frontend route/i18n/shell regressions continue passing.
- Build, repo hygiene, and whitespace checks continue passing.

## Acceptance Criteria

v3.1 is complete when:

1. Non-demo planning ranks selected ready merchants by deterministic fit score.
2. Generated plan versions include merchant-fit evidence and planner warnings.
3. Organizer workspace exposes fit rationale and warnings.
4. Ineligible merchants remain blocked before planning.
5. Weak but eligible merchants can be planned with visible warnings.
6. Demo and existing release-gate paths remain compatible.
7. Focused backend/frontend tests, full relevant regressions, build, repo hygiene, and whitespace checks pass.
