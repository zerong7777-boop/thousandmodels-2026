# v0.7 Demo Narrative

Date: 2026-06-12
Status: draft for competition demo preparation

## One-Sentence Positioning

Zhiyin Haojiang is an AI-assisted operations system for Macau old-district cultural-tourism events, proving a closed loop from planning to merchant execution, live recovery, visitor update, and review.

## 30-Second Problem Frame

Macau old districts have cultural stories, long-standing shops, and walkable streets, but turning those resources into repeatable cultural-tourism events is operationally hard. Organizers need to plan routes, coordinate merchants, control risks, update visitors, and review results. Small merchants need a low-friction way to participate without installing complex systems. Zhiyin Haojiang addresses this coordination problem rather than building another tourist recommendation app.

## Core Demo Story

A weekend night-walk event is planned across Fulong New Street, Senado/New Road, and the Inner Harbour. The organizer wants to connect old shops, cultural stories, visitor check-ins, and merchant capacity. During execution, one merchant sells out. The system detects the incident, proposes a recovery plan, waits for organizer approval, updates the public visitor route, adjusts merchant tasks, and generates a metric-backed review report.

## 3-5 Minute Demo Script

| Time | Screen | Speaker message |
| --- | --- | --- |
| 0:00-0:30 | Problem framing | This project is not a tourist recommendation app. It targets the harder operations problem behind old-district cultural-tourism events: planning, merchant coordination, live recovery, visitor updates, and review. |
| 0:30-1:00 | `/login` and `/organizer/dashboard` | The same event is served through role-specific product surfaces. The organizer enters an operations workspace rather than a generic demo page. |
| 1:00-1:45 | `/organizer/events/demo-night-tour` | The system represents the event as structured route, merchant task, budget, risk, and evidence objects. The important point is not free-form text generation; it is a controllable plan that can be approved and versioned. |
| 1:45-2:20 | `/merchant/events/demo-night-tour/status` | A small merchant can report live status with one obvious action. This low-friction update enters the event loop without requiring POS, hardware, or a complex merchant system. |
| 2:20-3:10 | `/organizer/events/demo-night-tour/exceptions` | The merchant update triggers an incident. The system proposes recovery, shows impact, previews the visitor notice, and waits for organizer confirmation. It does not auto-publish risky changes. |
| 3:10-3:45 | `/public/events/demo-night-tour` | The visitor sees only useful route guidance and live updates. Internal backend concepts such as plan versioning and recovery proposals are hidden from the public H5. |
| 3:45-4:30 | `/organizer/events/demo-night-tour/review` | The event closes with metrics, incident history, approvals, and recommendations. The result is structured operational memory, not a chat transcript. |
| 4:30-5:00 | Closing | The current MVP proves the deterministic closed loop. Qwen/QwenPaw can later enrich advisory drafting and structured planning, but the demo does not depend on model API availability. |

## 8-10 Minute Extended Script

2. Show the multilingual login screen and explain the three roles.
3. Enter organizer dashboard and explain the event operations home.
4. Enter event workspace and show route plan, merchant tasks, evidence trail, and approval state.
5. Explain that the current planning path is deterministic for repeatable local demonstration.
6. Switch to merchant status and trigger sold-out reporting.
7. Return to exception center and show incident, recovery suggestion, impact scope, public notice preview, and confirmation.
8. Open public H5 and show the visitor-facing update.
9. Open review center and explain metrics, incident traceability, and future knowledge accumulation.
10. Close with the Qwen/QwenPaw roadmap: advisory text first, structured planning assist second, merchant onboarding third, knowledge-grounded cultural narrative fourth.

## Role-by-Role Value

| Role | Value |
| --- | --- |
| Organizer | Reduces coordination burden by turning planning, approval, incident recovery, and review into one operational flow. |
| Merchant | Provides a low-friction workbench for assigned tasks, stock status, queue status, and organizer notices. |
| Visitor | Receives simple route guidance, stories, tasks, and live updates without seeing backend operations. |

## Closing Statement

The contribution of Zhiyin Haojiang is not a single generated itinerary. The contribution is a controlled event orchestration loop where AI assistance, merchant feedback, organizer approval, public route updates, and review evidence are connected. The current MVP intentionally keeps this loop deterministic and reliable; model-based Qwen/QwenPaw assistance can be layered on top after the product flow is stable.

## Implemented / Partial / Future Matrix

# v0.7 Competition Brief

Date: 2026-06-12
Status: draft for competition material preparation

## Project Title

Zhiyin Haojiang: AI-Assisted Event Orchestration for Macau Old-District Cultural Tourism

## Target Users

| User | Need |
| --- | --- |
| Organizer | Plan, approve, monitor, recover, and review cultural-tourism events. |
| Merchant | Receive event tasks and report live stock, queue, and visitor capacity. |
| Visitor | Follow a lightweight H5 route with stories, tasks, and live updates. |

## Pain Points

1. Old-district events require heavy manual coordination.
2. Small merchants have low digital capacity and cannot easily join complex systems.
3. Static route publishing responds poorly to stock, weather, and queue changes.
4. Event experience is hard to review and reuse after the activity ends.

## Product Flow

```text
Organizer event workspace
-> route plan and merchant tasks
-> merchant live status update
-> incident and recovery proposal
-> organizer approval
-> public H5 update
-> metric-backed review
```

## Current Demo Scope

The current MVP focuses on one Macau old-district night-walk event with simulated merchants and deterministic incidents. It proves the closed loop rather than full city-scale SaaS coverage.

## Technical Architecture

```text
React/Vite role-separated frontend
+ FastAPI backend
+ Pydantic schemas
+ SQLite persistence
+ deterministic agent backend
+ optional Qwen adapter for future advisory generation
+ audit, approval, plan versioning, incident, recovery, public notice, and review objects
```

## Innovation Points

1. The project targets event orchestration rather than generic tourist recommendation.
2. The MVP connects organizer, merchant, visitor, incident recovery, and review as one loop.
3. It treats AI output as structured operational objects instead of free-form text.
4. It keeps human approval for public-facing changes.
5. It provides a clear path for Qwen/QwenPaw advisory capability without making model APIs required for the demo.

## Current Limitations

1. Qwen/QwenPaw multi-agent orchestration is not yet implemented end to end.
2. Merchant onboarding uses demo data rather than natural-language, image, or voice parsing.
3. Route and weather behavior are deterministic or mocked, not real external integrations.
4. Knowledge accumulation is represented by review outputs and curated files, not a full historical learning system.
