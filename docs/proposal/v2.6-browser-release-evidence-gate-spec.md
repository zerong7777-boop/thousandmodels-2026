# v2.6 Browser Release Evidence Gate Spec

Date: 2026-06-28

Status: implemented.

## Purpose

v2.6 adds browser-level release evidence for the deterministic local beta. v2.5 proves the core product state machine through live API calls. v2.6 proves that the same state machine is reachable through the real Vite frontend and a real FastAPI backend, without request mocking.

This phase closes the gap between API readiness and operator-visible release confidence.

## Readiness Target

v2.6 moves the project from "release-gated local API beta" to "release-gated local browser beta".

## In Scope

- A Playwright browser gate that runs against a real local FastAPI API and real local Vite app.
- Organizer login and workspace visibility through the UI.
- Merchant login, workbench visibility, and interaction package visibility through the UI.
- Merchant live-status report and organizer exception suggestion visibility through the UI.
- Public H5 mobile scan, coupon claim, and coupon redemption through the UI.
- Organizer review report visibility through the UI after real public interactions.
- A structured JSON evidence artifact with gate steps, durations, browser route coverage, screenshot hashes, and failure details.
- Limited local screenshot generation for run-time inspection, with screenshot files treated as local generated artifacts rather than committed evidence churn.
- Live Playwright config hardening so the browser gate can use explicit API/web ports and an isolated SQLite database.
- Documentation updates to README, release runbook, and `docs/ai/`.

## Out Of Scope

- New business logic.
- New visual redesign.
- Cloud browser testing.
- Browser tests in the default GitHub Actions workflow.
- Committing PNG screenshots under `docs/research/assets/`.
- Real Qwen, DashScope, or QwenPaw calls.
- Real merchant, POS, payment, map, weather, traffic, or visitor identity integrations.
- Production monitoring or SLO proof.

## Required Behavior

### Invocation

The browser gate should be runnable from `apps/web` as:

```powershell
npm.cmd run test:e2e:live:v26
```

Equivalent direct command:

```powershell
npm.cmd exec playwright test tests/e2e/v26-browser-release-evidence.spec.ts --config playwright.live.config.ts
```

Optional environment variables:

- `PLAYWRIGHT_API_PORT`, default `8000`;
- `PLAYWRIGHT_WEB_PORT`, default `5179`;
- `VITE_API_BASE_URL`, default `http://127.0.0.1:<PLAYWRIGHT_API_PORT>`;
- `PLAYWRIGHT_BASE_URL`, default `http://127.0.0.1:<PLAYWRIGHT_WEB_PORT>`;
- `DATABASE_URL`, default isolated SQLite database under the OS temp directory;
- `V26_BROWSER_EVIDENCE_OUTPUT`, default `docs/research/assets/v2.6-browser-release-evidence-gate/browser-release-gate-result.json`;
- `V26_BROWSER_SCREENSHOT_DIR`, default local temp directory.

### Gate Steps

The browser gate must perform these steps in order:

1. Check `GET /api/health`.
2. Check `GET /api/ready`.
3. Prepare the demo event through real API calls: organizer login, seed, generate plan, approve plan v1, draft/publish event page, and generate merchant-edge packages.
4. Log in through the UI as organizer and verify the event workspace.
5. Log in through the UI as merchant and verify the interaction package.
6. Report merchant sold-out status through the UI, then verify organizer exception suggestion flow.
7. Open the public H5 route on a mobile viewport and complete scan, claim, and redeem through the UI.
8. Generate the review report through the API, then verify the organizer review page through the UI.
9. Check `/api/metrics` for public interaction and review counters touched by the run.

### Evidence

The JSON evidence artifact must include:

- `status`: `passed` or `failed`;
- `event_id`;
- `api_base_url`;
- `web_base_url`;
- `started_at`;
- `finished_at`;
- `duration_ms`;
- `steps`, with each step containing `step`, `ok`, `duration_ms`, and `summary`;
- `screenshots`, with file name, step, viewport, byte size, and SHA-256 digest;
- `failure`, when failed;
- `artifact_policy`, documenting that PNG files are local run artifacts and are not committed by default.

Evidence must not include passwords, session cookies, bearer tokens, provider payloads, secret values, or local absolute paths.

### Failure Semantics

The browser gate exits successfully only when all required browser and API checks pass.

It fails when:

- the API or web app cannot start;
- required UI routes do not render;
- required UI controls are missing or disabled;
- public H5 interactions do not mutate state through the real API;
- public, merchant, or tourist surfaces expose forbidden internal terms;
- expected process-local metrics are missing;
- JSON evidence cannot be written.

## Acceptance Criteria

- v2.6 spec and implementation plan exist.
- `playwright.live.config.ts` supports env-driven ports and isolated SQLite.
- The default Playwright config excludes the live browser gate from ordinary visual runs.
- `apps/web/tests/e2e/v26-browser-release-evidence.spec.ts` runs against real FastAPI and Vite without request mocking.
- `npm.cmd run test:e2e:live:v26` passes locally and writes JSON evidence.
- Full backend pytest passes serially.
- Full web Vitest and build pass.
- Repository hygiene and whitespace checks pass.
- Docs record the v2.6 result and boundary.
