Date: 2026-06-27
Project root: `<PROJECT_ROOT>`
Status: implemented and verified by v1.8.1 parser-fix live matrix

## 1. Purpose

v1.7 proved a sharper boundary than earlier live QwenPaw work:

```text
local QwenPaw provider reachable
+ HTTP 200 SSE response
+ no provider-level blocked reason
+ strict advisory qualification
-> advisory_unqualified / missing_fields
```

That is the correct failure mode. The project should not turn a non-empty
model response into a product success unless the response satisfies the
advisory-only adapter contract.

v1.8 should therefore optimize the real QwenPaw advisory qualification path
without weakening the contract:

```text
v1.7 local failure attribution
+ external mature-pattern research
+ prompt / agent / adapter experiment matrix
+ bounded repair design
+ evidence-bound smoke reporting
-> at least one qualified candidate or a precise blocker classification
```

The product claim after v1.8 should be:

```text
The project has a researched and testable plan for improving real QwenPaw
advisory qualification while preserving the non-authoritative product boundary.
```

The product claim must not be:

```text
QwenPaw now runs the live business workflow or can publish, approve, redeem,
or mutate merchant / tourist / organizer state.
```

## 2. Baseline and failure attribution

The current recommended next step is already narrow: improve prompt and agent
selection until the guarded real-QwenPaw smoke returns `advisory_qualified`,
while keeping the v1.4 fake adapter as the accepted product path.

Primary local evidence:

- `docs/ai/NEXT.md` records that the latest v1.7 response stayed in
  preamble / tool-introspection text and did not provide the three required
  fields.
- `docs/ai/VERIFY.md` records that v1.7 unit tests and focused backend tests
  passed, while the live smoke returned `advisory_unqualified`.
- `docs/research/assets/v1.5-real-qwenpaw-guarded-smoke/live-qwenpaw-smoke-result.json`
  records `response_status_code=200`, `provider_reachable=true`,
  `advisory_status=unqualified`, and
  `qualification_failure_kind=missing_fields`.
- `docs/research/v1.5-real-qwenpaw-guarded-smoke.md` records a sanitized
  response preview that restates the task and tries to inspect context instead
  of emitting the advisory payload.
- `apps/api/scripts/live_qwenpaw_smoke.py` contains the current prompt,
  localhost-only request path, SSE / JSON extraction, provider-error
  classification, advisory parsing, and qualification logic.

The failure is not:

- endpoint reachability
- local provider startup
- model configuration
- HTTP status
- provider-error classification
- a missing `X-Agent-Id` header in the smoke script

The failure is the final response contract. `qualify_advisory_text()` accepts
only parseable non-empty values for:

```text
recovery_rationale
visitor_safe_notice_draft
safety_notes
```

It correctly rejects mere mentions of those field names inside a preamble. The
current live response mentions the requested fields but never emits JSON keys
or Markdown sections that the adapter can qualify.

## 3. External patterns checked

The external research line reviewed mature patterns that map directly to this
failure mode. These are patterns for the v1.8 design, not new runtime
dependencies unless a later implementation plan explicitly adopts one.

### 3.1 Schema-first structured output

OpenAI Structured Outputs distinguishes valid JSON from schema adherence and
supports strict schema-following in supported APIs. LangChain also separates
provider-native structured output from tool-based structured output. The
transferable lesson is not "use this exact provider API"; it is:

```text
success = parsed object + schema-valid + required non-empty fields + safety pass
```

For QwenPaw v1.8, the adapter must keep `HTTP 200` separate from advisory
qualification. If QwenPaw cannot enforce structured output natively, the
project should use prompt contract plus deterministic validation.

### 3.2 Tool-use gating

OpenAI and Anthropic tool-use documentation both expose controls for tool
choice and tool availability. The transferable lesson is to minimize the tool
surface during the final answer phase.

The v1.7 response looks like an agent preparing to inspect tools or workspace
context. v1.8 should test a final formatter phase that receives only distilled
facts and no tool inventory, or a QwenPaw agent/profile that behaves as a
chat-only formatter.

### 3.3 Role isolation

Agent orchestration patterns commonly split planning, execution, and final
answer formatting. The v1.8 mapping is:

```text
planner / advisor: may reason about recovery options
formatter: must only produce the advisory contract object
validator: deterministic pass / fail gate
```

The final formatter must not mention tools, prompts, schemas, agent state, or
internal reasoning.

### 3.4 Few-shot contract examples

Prompting guidance from major model providers recommends examples when a
stable output format matters. v1.8 should not rely on one abstract sentence
that says "Return a concise advisory response." It should test positive and
negative examples:

- valid JSON advisory with all three fields
- valid conservative advisory when evidence is incomplete
- valid advisory with explicit non-authoritative safety language
- invalid preamble-only response
- invalid tool-introspection response
- invalid missing-field response

Negative examples must be labeled as invalid so the model does not imitate
them.

### 3.5 Validator-in-the-loop repair

Guardrails-style validators and agent evaluator loops both support the same
practical pattern:

```text
generate
-> validate deterministically
-> repair with exact validation errors
-> revalidate
-> stop after a bounded number of attempts
```

v1.8 should allow at most two repair attempts in the research path. If the
final attempt still lacks required fields or contains unsafe authority claims,
the outcome remains `advisory_unqualified`.

### 3.6 Evidence-bound evaluation

The smoke should continue to report concrete evidence, not narrative
confidence. v1.8 should record:

- provider HTTP status and content type
- prompt variant
- agent selection
- session isolation id
- response hash and bounded sanitized preview
- parsed JSON / Markdown extraction result
- required-field presence
- safety validator result
- repair attempt count
- final qualification

## 4. Scope decision

v1.8 is:

- a prompt, agent, and adapter qualification optimization spike
- a dual-track research synthesis from local failure evidence and external
  mature patterns
- an experiment design for improving real QwenPaw advisory output
- a strict validation and evidence-reporting phase

v1.8 is not:

- a production QwenPaw orchestration migration
- a relaxation of the v1.7 advisory qualification contract
- a required dependency for the deterministic demo
- a replacement for the v1.4 fake adapter product path
- a merchant, tourist, payment, POS, map, traffic, weather, or visitor identity
  integration
- a claim that QwenPaw can approve, publish, mutate, create coupons, or redeem
  anything

## 5. Target advisory contract

The v1.8 preferred contract is JSON-first:

```json
{
  "recovery_rationale": "Why the organizer should take the suggested recovery path, grounded in the provided incident.",
  "visitor_safe_notice_draft": "Visitor-facing wording that avoids operational over-claims.",
  "safety_notes": "Explicit notes that this is advisory only and requires organizer approval before any authoritative action."
}
```

Required invariants:

- The object must include exactly the three required top-level fields, or those
  fields plus explicitly approved diagnostics if the implementation plan adds
  them.
- Each required field must be a non-empty string after trimming.
- No text before or after the JSON object should be accepted in the JSON-only
  experiment variant.
- The content must not claim that it approved, published, applied, updated,
  created, redeemed, or mutated authoritative state.
- Safe negations remain valid, for example saying that no state was changed or
  that organizer approval is still required.

Markdown remains a regression-compatible parser path because v1.7 tests already
cover valid Markdown headings. It should not be the preferred new prompt path
unless the JSON-only path fails due QwenPaw provider behavior.

## 6. Experiment matrix

v1.8 should run a compact matrix before any product wiring.

### 6.1 Prompt variants

| Variant | Purpose | Expected output |
| ------- | ------- | --------------- |
| `prompt_a_json_only` | Force final answer into one JSON object | Three required keys, no prose |
| `prompt_b_markdown_sections` | Test parser-compatible headings | Exact three headings |
| `prompt_c_few_shot_json` | Stabilize format with examples | JSON object matching examples |
| `prompt_d_error_repair` | Test validator feedback loop | Corrected JSON after precise error list |

`prompt_a_json_only` should be the first candidate because it is easiest to
validate deterministically.

### 6.2 Agent variants

| Variant | Purpose | Notes |
| ------- | ------- | ----- |
| `agent_qa_current` | Preserve v1.7 baseline | Uses `QwenPaw_QA_Agent_0.2` |
| `agent_default` | Test local default routing | No explicit `QWENPAW_AGENT_ID` |
| `agent_formatter_candidate` | Test no-tool formatter behavior | Only if a local chat-only or formatter agent is available |

Each run must use a fresh `QWENPAW_SMOKE_SESSION_ID` so prior introspection does
not leak into the next prompt.

### 6.3 Adapter variants

| Variant | Purpose | Acceptance |
| ------- | ------- | ---------- |
| `strict_existing_validator` | Keep current v1.7 qualification | No parser loosening |
| `json_no_preamble_validator` | Reject text outside JSON | Required for JSON-only path |
| `repair_validator` | Emit exact validation errors for bounded repair | Never marks failed repair as success |

Adapter changes must be diagnostic or stricter. They must not convert a
preamble into a valid advisory by guessing missing fields.

### 6.4 Negative controls

The matrix must keep these controls:

- v1.7 recorded preamble-only output remains `missing_fields`
- valid JSON advisory qualifies
- valid Markdown advisory qualifies only when required sections are present
- authority claims remain `unsafe_authority_claim`
- coupon and redemption authority claims remain unqualified
- provider error payload remains blocked and not advisory-qualified

## 7. Requirements

### 7.1 Contract requirements

- `REQ-STRUCT-001`: `advisory_qualified` requires a parsed advisory object or
  parser-compatible Markdown sections with all required fields present.
- `REQ-STRUCT-002`: JSON-only experiment variants reject non-whitespace content
  before or after the JSON object.
- `REQ-STRUCT-003`: Field-name mentions in prose do not satisfy field presence.
- `REQ-STRUCT-004`: Empty strings, whitespace-only strings, or truncated fields
  are missing fields.

### 7.2 Tool and role requirements

- `REQ-TOOL-001`: Final formatting experiments must avoid exposing tool
  inventory, workspace inspection instructions, or provider debug context.
- `REQ-TOOL-002`: If QwenPaw supports a no-tool or chat-only agent/profile, the
  experiment matrix must include it.
- `REQ-ROLE-001`: Planning and final formatting should be separated in prompt
  design even if they are executed in a single QwenPaw call.
- `REQ-ROLE-002`: The final answer must not mention tools, schemas, prompts,
  internal agent state, or self-inspection.

### 7.3 Repair requirements

- `REQ-REPAIR-001`: Repair is bounded to at most two attempts.
- `REQ-REPAIR-002`: Repair prompts receive exact validation errors, not a vague
  "try again" instruction.
- `REQ-REPAIR-003`: Each repair attempt is validated from scratch.
- `REQ-REPAIR-004`: Failure after bounded repair remains
  `advisory_unqualified`.

### 7.4 Evidence requirements

- `REQ-EVIDENCE-001`: HTTP 200 and non-empty response remain provider
  reachability evidence only.
- `REQ-EVIDENCE-002`: Smoke output records prompt variant, agent variant,
  validator variant, session id, response hash, bounded sanitized preview, field
  presence, safety result, repair count, and final outcome.
- `REQ-EVIDENCE-003`: Evidence artifacts must remain sanitized and bounded.
- `REQ-EVIDENCE-004`: No secret, raw token, full local path, private working
  directory, or raw unbounded stream may be written to docs.

## 8. Acceptance criteria

The phase is successful if one of these outcomes is reached:

### 8.1 Qualified outcome

At least one real local QwenPaw smoke run records:

```text
provider_reachable=true
advisory_status=qualified
outcome=advisory_qualified
qualification_failure_kind=null
all required advisory fields present
no unsafe authority claim
```

The successful run must be accompanied by sanitized evidence and must not
modify any product workflow boundary.

### 8.2 Precise unqualified outcome

If no variant qualifies, v1.8 is still useful if it records a precise blocker:

- prompt contract ignored
- agent selection not honored
- tool-introspection dominates final answer
- provider output cannot be parsed into contract JSON or Markdown
- repair loop fails after bounded attempts
- local QwenPaw agent configuration is not externally controllable from the
  current script

This outcome should recommend the next technical step without claiming product
success.

## 10. Open implementation decisions

The implementation plan resolved:

1. v1.8 writes experiment evidence into a new
   `v1.8-qwenpaw-advisory-optimization` artifact path.
2. Prompt variants are implemented as a small internal registry in
   `live_qwenpaw_smoke.py`.
3. Repair is implemented as a bounded second or third QwenPaw call, disabled
   unless `QWENPAW_MAX_REPAIR_ATTEMPTS` is set.
4. The JSON-only validator becomes the preferred path while
   Markdown support remains only for backwards-compatible tests.
5. Local QwenPaw formatter-agent configuration remains outside repo state.

## 11. Risks and guardrails

Risk: QwenPaw may not honor the selected `X-Agent-Id`.
Guardrail: Record the requested agent id and classify this as an agent-routing
unknown if behavior does not change.

Risk: A stricter prompt may still be overridden by QwenPaw system-level
tool-introspection behavior.
Guardrail: Include a no-tool / formatter agent variant if available, and keep
tool-introspection as an explicit failure category.

Risk: Few-shot examples may cause copied boilerplate.
Guardrail: Include semantic review checks and vary incident details in the
matrix.

Risk: Repair could make unsafe content more fluent.
Guardrail: Re-run authority-claim and field validators after every repair.

Risk: The team may overstate live-provider evidence.
Guardrail: Keep `provider_reachable`, `advisory_status`, and product-readiness
as separate fields in docs and evidence.
