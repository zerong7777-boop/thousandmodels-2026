# Zhiyin Haojiang v1.5 Real QwenPaw Guarded Smoke Spec

Date: 2026-06-22
Project root: `<PROJECT_ROOT>`
Status: design spec for next phase

## 1. Purpose

v1.4 proved a QwenPaw-style orchestration boundary with a deterministic fake adapter:

```text
merchant incident
-> organizer-only shadow orchestration
-> Leader / Worker evidence
-> permissioned tool decisions
-> advisory bundle
-> no authoritative mutation
```

The remaining credibility gap is narrower:

```text
The project has not yet shown that a real QwenPaw runtime can be reached from this workspace.
```

v1.5 should close that gap without weakening the current demo. The goal is a guarded live-provider smoke, not production integration:

```text
Call a real locally running QwenPaw service once
+ send only a sanitized incident prompt
+ record sanitized smoke evidence
+ keep the v1.4 fake adapter as the accepted product path
+ keep v1.3 deterministic live demo as the reliable demo path
```

The product claim after v1.5 should be:

```text
The project can optionally probe a real local QwenPaw service and record sanitized evidence.
The business workflow still runs deterministically and QwenPaw remains non-authoritative.
```

The product claim must not be:

```text
QwenPaw now orchestrates the production business workflow.
```

## 2. External facts checked

The implementation plan is based on the current public QwenPaw documentation checked on 2026-06-22:

- QwenPaw is installable with `pip install qwenpaw`, then `qwenpaw init --defaults`, then `qwenpaw app`.
- The local server listens on `127.0.0.1:8088` by default.
- The quick-start docs expose `POST /api/agent/process` as the install-verification Agent API.
- Cloud models require provider configuration and API keys; local model providers can avoid cloud API keys.
- QwenPaw includes security controls such as Tool Guard, File Guard, Skill Scanner, and optional web authentication.
- QwenPaw Team uses QwenPaw plus HiClaw for multi-agent teams, with additional deployment and lifecycle scope.

## 3. Scope decision

v1.5 is:

- a real local QwenPaw connectivity smoke
- a sanitized prompt and response evidence phase
- a timeout, failure-classification, and redaction phase
- a documentation and handoff phase

v1.5 is not:

- a production QwenPaw runtime migration
- a HiClaw / QwenPaw Team deployment
- a required dependency for the deterministic demo
- a frontend product feature
- an autonomous operations phase
- a real merchant, POS, payment, hardware, map, weather, traffic, or visitor identity integration phase

## 4. Recommended approach

Use a standalone backend script:

```text
apps/api/scripts/live_qwenpaw_smoke.py
```

The script should:

1. require `RUN_LIVE_QWENPAW_SMOKE=1`;
2. read `QWENPAW_BASE_URL`, defaulting to `http://127.0.0.1:8088`;
3. optionally read `QWENPAW_SMOKE_SESSION_ID`, defaulting to a deterministic smoke session id;
4. build a sanitized incident prompt from deterministic demo state;
5. call `POST /api/agent/process`;
6. classify the result as `live_success`, `blocked`, or `failed`;
7. write sanitized JSON evidence and a short Markdown smoke note;
8. avoid writing raw provider streams, API keys, Authorization headers, local absolute paths, or full unredacted prompts.

This keeps the risk boundary clear. The v1.4 product API and UI stay deterministic. Real QwenPaw is probed through a manual smoke command only.

## 5. Runtime contract

### 5.1 Environment gate

The script must refuse to call QwenPaw unless:

```text
RUN_LIVE_QWENPAW_SMOKE=1
```

If the flag is missing, the script exits non-zero and writes a sanitized blocked result:

```json
{
  "outcome": "blocked",
  "blocked_reason": "RUN_LIVE_QWENPAW_SMOKE is required"
}
```

### 5.2 Service URL

The base URL is:

```text
QWENPAW_BASE_URL or http://127.0.0.1:8088
```

The script must reject non-localhost URLs by default. Only these hosts are allowed:

```text
127.0.0.1
localhost
::1
```

This prevents accidental transmission of demo state to remote QwenPaw instances. If a later phase needs remote service support, it must define a separate approval gate.

### 5.3 Endpoint

The script calls:

```text
POST {QWENPAW_BASE_URL}/api/agent/process
```

Request body:

```json
{
  "input": [
    {
      "role": "user",
      "content": [
        {
          "type": "text",
          "text": "<sanitized smoke prompt>"
        }
      ]
    }
  ],
  "session_id": "zhiyin-v15-qwenpaw-smoke"
}
```

The endpoint may return streaming SSE or plain JSON depending on QwenPaw configuration. v1.5 should capture only a bounded preview and summary, not a full raw transcript.

### 5.4 Timeout

Use short explicit timeouts:

```text
connect timeout: 3 seconds
read timeout: 20 seconds
overall smoke expectation: under 30 seconds for a healthy local setup
```

If QwenPaw is not running, not configured with a model, or returns an auth/configuration failure, the smoke should be `blocked`, not a product failure.

## 6. Sanitized prompt contract

The prompt should be small and bounded. It should not include cookies, session ids, Authorization headers, API keys, local paths, database paths, visitor identifiers, or raw event dumps.

The prompt should include only:

- product context: "Zhiyin Haojiang incident advisory smoke"
- event id: `demo-night-tour`
- incident kind: merchant sold-out signal
- affected merchant id: `m001`
- current boundary: advisory only, no mutation, no approval, no publishing
- requested output shape: brief JSON-like advisory with rationale, safe public copy, and safety notes

Example semantic shape:

```text
You are helping verify a local QwenPaw integration for a demo operations system.
Incident: event demo-night-tour has merchant m001 sold_out.
Return only advisory analysis. Do not claim you applied changes.
Summarize:
1. recovery_rationale
2. visitor_safe_notice_draft
3. safety_notes
```

The exact prompt may be persisted only after redaction and truncation.

## 7. Evidence contract

Create:

```text
docs/research/v1.5-real-qwenpaw-guarded-smoke.md
docs/research/assets/v1.5-real-qwenpaw-guarded-smoke/live-qwenpaw-smoke-result.json
```

The JSON artifact should include:

```json
{
  "outcome": "live_success | blocked | failed",
  "base_url_host": "127.0.0.1",
  "endpoint": "/api/agent/process",
  "session_id": "zhiyin-v15-qwenpaw-smoke",
  "request_sent": true,
  "response_status_code": 200,
  "response_content_type": "text/event-stream",
  "latency_ms": 1234,
  "sanitized_prompt_preview": "...",
  "sanitized_response_preview": "...",
  "response_preview_sha256": "...",
  "blocked_reason": null,
  "failure_kind": null,
  "evidence_generated_at": "ISO-8601 timestamp"
}
```

Do not include:

- full raw response stream
- provider API keys
- Authorization headers
- cookies
- local absolute paths
- QwenPaw config files
- QwenPaw secret paths
- unredacted exception tracebacks
- private visitor identity

## 8. Failure classification

Use these outcomes:

| Outcome | Meaning |
| --- | --- |
| `live_success` | Local QwenPaw accepted the request and returned a non-empty response preview. |
| `blocked` | Preconditions are missing: smoke flag absent, QwenPaw not running, model not configured, localhost-only guard rejected URL, auth/config required, or endpoint unavailable. |
| `failed` | The script ran under valid preconditions but hit an unexpected implementation error or unsafe evidence condition. |

This distinction matters because a missing local QwenPaw process is not a business regression.

## 9. Safety and redaction

The script must redact common sensitive patterns before writing evidence:

- `sk-...`
- `Bearer ...`
- `Authorization: ...`
- `DASHSCOPE_API_KEY=...`
- QwenPaw auth password environment-variable values
- Windows local user paths
- project-local absolute paths under drive letters

The evidence writer must run strict scans after writing artifacts:

```powershell
rg -n "sk-[A-Za-z0-9._-]{20,}|Bearer\s+sk-[A-Za-z0-9._-]{20,}|DASHSCOPE_API_KEY\s*=\s*sk-|QWENPAW_AUTH_PASSWORD\s*=" docs/research/v1.5-real-qwenpaw-guarded-smoke.md docs/research/assets/v1.5-real-qwenpaw-guarded-smoke
rg -n "([A-Z]:\\Users\\[^\\]+|[A-Z]:\\rz\\|[A-Z]:/rz/)" docs/research/v1.5-real-qwenpaw-guarded-smoke.md docs/research/assets/v1.5-real-qwenpaw-guarded-smoke
```

Expected: no matches.

## 10. Integration boundaries

v1.5 must not change:

- default `AgentRuntime(mode="qwenpaw_workflow")` fake adapter behavior;
- organizer shadow orchestration endpoint response contract;
- merchant, tourist, or public page source;
- v1.3 live smoke configuration;
- deterministic demo reset behavior.

The live QwenPaw smoke may reuse deterministic demo setup code to create the incident context, but it must use an isolated temporary SQLite database if it imports FastAPI/store modules.

## 11. Tests

Add backend tests for:

1. missing `RUN_LIVE_QWENPAW_SMOKE` returns blocked evidence and no network call;
2. non-localhost `QWENPAW_BASE_URL` is rejected;
3. prompt builder does not include secrets, cookies, Authorization headers, or local paths;
4. response redaction removes key-like and bearer-token strings;
5. SSE response parsing yields bounded sanitized preview;
6. JSON response parsing yields bounded sanitized preview;
7. successful fake HTTP transport writes `live_success` evidence to a temp artifact path;
8. connection failure writes `blocked` evidence with clear reason.

Full regression should include:

```powershell
cd <PROJECT_ROOT>\apps\api
python -m pytest tests/test_v15_live_qwenpaw_smoke_script.py -q
python -m pytest -q

cd <PROJECT_ROOT>\apps\web
npm.cmd run test
npm.cmd run build
npm.cmd exec -- playwright test
npm.cmd exec -- playwright test tests/e2e/v13-live-demo-smoke.spec.ts --config playwright.live.config.ts
```

If local QwenPaw is not installed or not configured, the optional live smoke should record `blocked` honestly. That is acceptable for v1.5 as long as the guard and evidence path are verified.

## 12. Manual smoke commands

Suggested manual flow:

```powershell
qwenpaw --version
qwenpaw init --defaults
qwenpaw app
```

In another shell:

```powershell
cd <PROJECT_ROOT>\apps\api
$env:RUN_LIVE_QWENPAW_SMOKE = "1"
$env:QWENPAW_BASE_URL = "http://127.0.0.1:8088"
python scripts\live_qwenpaw_smoke.py
```

If QwenPaw has no model configured, the evidence should say `blocked`. If a model is configured and `/api/agent/process` returns content, the evidence may say `live_success`.

## 14. Acceptance criteria

v1.5 is complete when:

1. `docs/proposal/v1.5-real-qwenpaw-guarded-smoke-spec.md` exists.
2. `docs/proposal/v1.5-real-qwenpaw-guarded-smoke-implementation-plan.md` exists.
3. The script refuses to call non-localhost QwenPaw by default.
4. The script refuses to call QwenPaw unless `RUN_LIVE_QWENPAW_SMOKE=1`.
5. The script can classify missing service/model/auth as `blocked`.
6. The script can classify a fake successful HTTP response as `live_success`.
7. Evidence artifacts are sanitized and bounded.
8. Tests cover prompt sanitization, URL guard, response parsing, success, and blocked paths.
9. The v1.4 fake adapter remains default for product/API behavior.
10. v1.3 live smoke still passes.
11. Final scans find no real key, bearer token, password, local absolute path, or internal term leakage to merchant/public/tourist pages.

## 15. Stop gates

Stop and ask for review if:

- QwenPaw requires disabling local security controls to run the smoke;
- the only available service URL is remote or public;
- evidence would need to store API keys, cookies, Authorization headers, raw streams, or local absolute paths;
- live QwenPaw output would need to approve, publish, or mutate business state;
- implementation requires changing merchant, tourist, or public pages;
- implementation requires HiClaw, Docker, Kubernetes, or QwenPaw Team deployment;
- v1.3 deterministic live smoke breaks for reasons unrelated to test environment cleanup.

## 16. Summary

v1.5 should prove real QwenPaw reachability, not real QwenPaw authority.

The right target is:

```text
manual guarded smoke
+ localhost-only QwenPaw service
+ sanitized incident prompt
+ bounded sanitized evidence
+ honest blocked/live_success classification
+ unchanged deterministic product path
```

That is the smallest useful step from v1.4 toward real QwenPaw usage.
