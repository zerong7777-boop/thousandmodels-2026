# Zhiyin Haojiang v0.9 Agent Evidence UI Spec

Date: 2026-06-12
Project root: `<PROJECT_ROOT>`
Status: design spec for next phase

## 1. Purpose

v0.8 made the Agent layer concrete in the backend:

- `AgentRun`
- enriched `AgentStep`
- `AgentToolCall`
- `AgentDraft`
- `AgentModelCall`
- `AgentEvaluation`
- deterministic planning, recovery, and review Agent runs
- organizer-only read endpoints for Agent evidence

However, this evidence is still mostly visible through tests and API calls. For a competition demo, that is not enough. Judges should not need to inspect backend tests to understand that the system has real Agent orchestration evidence.

The purpose of v0.9 is to expose the v0.8 Agent evidence through focused organizer-facing product UI.

The target message is:

```text
The system does not only show a generated route.
It shows which specialist Agents ran, which tools they called,
what drafts they produced, what safety checks passed,
and where human approval remains mandatory.
```

v0.9 must remain a frontend evidence and presentation layer over the already verified deterministic backend. It must not introduce mandatory model APIs, Qwen/QwenPaw dependency, broad UI redesign, or new backend workflow semantics.

## 3. Problem Statement

The project now has a stronger Agent backend than the UI communicates.

Without v0.9, the demo can still be challenged as:

```text
a polished event operations product with hidden Agent evidence
```

That is a presentation and product problem, not a backend capability problem. The backend already records specialist steps and tool calls. The missing layer is a judge-readable organizer UI that makes the Agent construction visible without leaking raw internals to the wrong users.

v0.9 should solve three problems:

1. Competition legibility:
   Judges can see multi-Agent orchestration evidence in the running product.
2. Product credibility:
   Organizer pages show evidence as an operational review surface, not a marketing panel.
3. Boundary clarity:
   Merchant, tourist, and public H5 pages remain clean user products and do not expose raw Agent internals.

## 4. Positioning Decision

v0.9 is:

- an organizer-facing Agent evidence UI phase
- a frontend integration phase over v0.8 backend contracts
- a demo legibility phase for Agent capability
- a safety-boundary visualization phase

v0.9 is not:

- a Qwen/QwenPaw integration phase
- a new Agent backend implementation phase
- a broad page redesign phase
- a new login/auth/role model phase
- a real merchant, map, weather, POS, payment, or traffic integration phase
- a marketing landing page

The main product stance:

```text
Organizer sees Agent evidence.
Merchant sees assigned work and live status.
Tourist/public sees route guidance and visitor-safe notices.
```

## 5. Scope

### 5.1 In Scope

v0.9 includes:

- frontend TypeScript contracts for v0.8 Agent evidence
- API client methods for Agent runs, drafts, and tool calls
- organizer workspace Agent evidence panel
- organizer exception center recovery Agent evidence panel
- organizer review center review Agent evidence panel
- localized labels for `zh-Hans`, `zh-Hant`, and `en`
- tests that organizer can see Agent evidence
- tests that merchant, tourist, and public H5 do not expose raw Agent internals
- optional v0.9 screenshot capture for the Agent evidence surfaces
- docs/ai handoff updates after implementation

### 5.2 Out Of Scope

v0.9 must not include:

- mandatory `DASHSCOPE_API_KEY`
- Qwen/QwenPaw required demo path
- optional Qwen draft backend Task 8 from v0.8 unless separately approved
- backend core workflow redesign
- new authoritative Agent mutations
- autonomous approval or publication
- real merchant onboarding
- hardware, POS, payment, real map, real weather, or real traffic prediction
- vector database or RAG implementation
- broad UI shell redesign
- full SaaS admin console
- registration or account management expansion

Backend changes are allowed only if frontend integration discovers a concrete API response gap. Such changes must be small, compatible, and tested.

| Backend object | Organizer product term | Allowed detail |

| `AgentDraft` | Controlled draft | Draft type, content, source run, status |
| `AgentEvaluation` | Safety check | schema, public-copy safety, approval boundary |
| `AgentModelCall` | Model call record | Provider/mode status only, if present |

## 7. Target User Experience

### 7.1 Organizer Workspace

Route:

```text
/organizer/events/demo-night-tour
```

Goal:

Show how the initial route plan was produced by a deterministic multi-Agent run.

Required additions:

- Agent run summary for `trigger=planning_generation`
- run mode and status
- fallback indicator, usually "No fallback used" for deterministic P0
- specialist step timeline
- at least six meaningful planning steps if present in backend data
- tool-call summary, especially:
  - `route.build_static_route`
  - `merchant.select_night_merchants`
  - `budget.split_budget`
- clear human approval boundary:
  - the Agent can prepare plan evidence
  - organizer still confirms the route plan

Required visual behavior:

- The panel should sit inside the event workspace as operational evidence.
- It should not dominate the page above plan status and approval.
- It should support empty/loading/error states.
- It should not require opening browser devtools or reading JSON.

Recommended layout:

```text
Agent evidence summary strip
+ specialist step timeline
+ selected run details
+ tool-call evidence drawer or compact table
```

Acceptance:

- A judge can see which Agents participated in planning.
- The UI makes tool calls first-class evidence.
- The approval boundary is visible.
- Existing plan generation and approval actions still work.

### 7.2 Organizer Exception Center

Route:

```text
/organizer/events/demo-night-tour/exceptions
```

Goal:

Show how a merchant sold-out signal becomes an incident, recovery draft, public notice draft, and human-approved route update.

Required additions:

- Agent run summary for `trigger=incident_recovery`
- recovery Agent steps:
  - incident interpretation
  - recovery proposal draft
  - public notice draft
  - safety evaluation
- controlled draft cards:
  - `recovery_explanation`
  - `public_notice`
- tool-call evidence for `recovery.build_recovery_proposal`
- safety/approval indicators:
  - draft only
  - requires organizer approval
  - does not publish automatically
- link the evidence to the existing recovery approval panel

Required visual behavior:

- The evidence should explain why the suggested recovery is credible.
- The panel must not imply the Agent has already approved the recovery.
- The confirmation button remains the only visible path to route v2 publication.

Acceptance:

- After merchant sold-out triggers an incident, the organizer can see recovery Agent evidence.
- The recovery draft and public notice draft are clearly marked as drafts.
- The page still shows v1 -> v2 impact.
- Confirming recovery still uses the existing backend approval endpoint.

### 7.3 Organizer Review Center

Route:

```text
/organizer/events/demo-night-tour/review
```

Goal:

Show that the review report is not a generic summary. It is supported by an Agent draft linked to operational metrics, incidents, proposals, and notices.

Required additions:

- Agent run summary for `trigger=review_generation`
- review Agent draft card for `draft_type=review_summary`
- evidence references from draft payload:
  - metrics
  - incidents
  - public notices
  - recovery proposals
- distinction between:
  - measured evidence
  - recommendations
  - narrative summary

Acceptance:

- Review page shows metric-backed report and Agent evidence together.
- The UI makes it clear that recommendations are derived from recorded operational evidence.
- It does not look like a static text summary.

### 7.4 Organizer Dashboard

Route:

```text
/organizer/dashboard
```

Goal:

Keep dashboard focused on operations. Do not overload it with full Agent details.

Allowed additions:

- compact "Agent evidence ready" status in attention queue or activity timeline
- link to event workspace evidence panel
- link to exception center if recovery evidence exists

Not required:

- full Agent run timeline
- tool-call drawer
- draft cards

Acceptance:

- Dashboard remains an operations home.
- Agent evidence is discoverable but not the main dashboard content.

### 7.5 Merchant Surfaces

Routes:

```text
/merchant/dashboard
/merchant/events/demo-night-tour/tasks
/merchant/events/demo-night-tour/status
/merchant/notifications
```

Goal:

Preserve merchant product simplicity.

Required boundary:

- Merchant pages must not show raw Agent evidence.
- Merchant pages may show organizer-facing outcome language:
  - "Organizer review requested"
  - "Route update pending"
  - "Task updated after organizer confirmation"

Acceptance:

- Sold-out quick action still triggers incident and organizer review.
- Merchant does not see AgentRun, ToolCall, Qwen, schema, fallback, or backend terminology.

### 7.6 Tourist And Public H5

Routes:

```text
/user/events/demo-night-tour
/user/events/demo-night-tour/route
/user/events/demo-night-tour/points/rp001
/user/events/demo-night-tour/notices
/public/events/demo-night-tour
```

Goal:

Preserve visitor-safe route guidance.

Required boundary:

- No raw Agent evidence.
- No backend/model terms.
- Public H5 remains unauthenticated.
- Updated route and notices are expressed as visitor guidance.

Acceptance:

- Tourist/public pages still pass forbidden-term tests.
- v0.9 does not turn visitor pages into backend previews.

## 8. Data And API Requirements

### 8.1 Frontend Types

Add TypeScript contracts matching backend v0.8 objects:

```text
AgentRun
AgentToolCall
AgentDraft
AgentModelCall
AgentEvaluation
AgentRunTrigger
AgentRunMode
AgentRunStatus
AgentDraftType
```

`AgentStep` should be enriched in a backward-compatible way:

```text
existing fields stay required or accepted
new v0.8 fields are optional where old traces may omit them
```

Required compatibility:

- Existing `AgentTrace` rendering still works.
- New `AgentRun` rendering does not require old `AgentTrace`.
- Tests can mock both old and new evidence payloads.

### 8.2 API Client Methods

Add methods to `apps/web/src/api.ts`:

```text
getAgentRuns(eventId)
getAgentDrafts(eventId, draftType?)
getAgentToolCalls(eventId, runId)
```

Optional if the backend later exposes them:

```text
getAgentModelCalls(eventId, runId)
getAgentEvaluations(eventId, runId)
```

For v0.9 P0, do not add backend endpoints only to satisfy optional UI fields. Render missing optional evidence as "not recorded for this run".

### 8.3 Filtering Rules

Organizer workspace should prefer:

```text
trigger=planning_generation
```

Exception center should prefer:

```text
trigger=incident_recovery
draft_type in recovery_explanation, public_notice
```

Review center should prefer:

```text
trigger=review_generation
draft_type=review_summary
```

If multiple runs exist, show the latest completed run first and make older runs secondary.

- mode


### 9.2 AgentStepTimeline

Shows:

- agent name
- objective
- decision reason
- confidence
- input refs count
- tool calls count
- human approval flag

Primary purpose:

```text
Which specialist Agents contributed?
```

### 9.3 AgentToolCallList

Shows:

- tool name
- status
- summarized inputs
- summarized outputs
- step association

Primary purpose:

```text

```

### 9.4 AgentDraftCard

Shows:

- draft type
- locale
- status
- content
- source run

```text

```

### 9.5 AgentSafetyChecklist

Shows:

- schema passed
- public copy safe
- human approval required
- unsafe mutation blocked or not attempted
- fallback status

If persisted `AgentEvaluation` is not exposed through API yet, derive a limited display from available run/draft fields and label it as UI-side boundary summary, not backend evaluation evidence.

## 12. State Requirements

Each Agent evidence surface must handle:

```text

empty

no selected run

```

Empty states must be product-specific:

- workspace: "Generate a route plan to create planning evidence."

- review center: "Generate a review report to create review evidence."

Do not show blank panels or generic "No data".

## 13. Safety And Boundary Requirements

### 13.1 No Autonomous Mutation

The frontend must not add any action that lets Agent evidence directly:

- approve a plan
- approve a recovery proposal
- publish a public notice
- update merchant runtime state
- mutate auth/session

Existing organizer approval buttons can remain, but the Agent panel must explain that approval happens through the existing approval workflow.

### 13.2 No Public Leakage

Merchant, tourist, and public H5 routes must not include raw evidence components, imports, or labels.

Required source-level expectation:

```text
apps/web/src/pages/merchant/*
apps/web/src/pages/tourist/*
apps/web/src/pages/public/*
```

must not import components from:

```text
apps/web/src/components/agent/*
```

unless a future spec explicitly designs a visitor-safe derived explanation component.

### 13.3 No Model Overclaim

If `mode=deterministic`, the UI should say deterministic or rule-backed only in organizer evidence context.

Do not imply:

- Qwen was used
- QwenPaw orchestration ran
- LLM output made the decision

If model call records are absent, display:

```text
No model call recorded for this deterministic run.
```

## 16. Stop Gates

Stop and ask for review if:

- implementing v0.9 requires a large backend redesign
- Agent evidence cannot be read through current v0.8 endpoints
- the UI would require Qwen/QwenPaw or `DASHSCOPE_API_KEY`
- the design starts exposing raw Agent internals on merchant/tourist/public pages
- the Agent panel implies autonomous approval or publication
- public H5 shows backend/model terms
- source changes require a new major UI framework
- mobile screenshots show unresolved overflow or overlapping text
- the page becomes a generic AI dashboard rather than an organizer operation surface
- frontend tests or backend pytest cannot be brought back to green within the scoped changes

v0.9 should make v0.8 visible.

The target is:

```text
v0.8 backend Agent evidence
+ organizer-facing evidence panels
+ specialist steps and tool calls visible in product UI
+ controlled drafts visible before approval
+ human approval boundary explicit
+ merchant/tourist/public surfaces kept clean
```

This is the right next step because the Agent capability now exists in the backend, but competition judges need to see it in the running product without sacrificing product credibility or public-facing safety.
