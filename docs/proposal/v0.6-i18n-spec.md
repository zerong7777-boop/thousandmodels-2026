# Zhiyin Haojiang v0.6 i18n Spec

Date: 2026-06-12
Project root: `<PROJECT_ROOT>`
Status: design spec for next phase

## 1. Purpose

v0.5 made the product surfaces visually credible enough to pause UI polish. v0.6 should make the interface usable for a Macau/UM competition context by adding a real language strategy instead of hardcoding all visible product copy in English.

The goal is not to redesign the UI again. The goal is to turn the current v0.5 product demo into a multilingual product surface:

- Default interface language should be Simplified Chinese.
- Traditional Chinese should be available for Macau-facing review and demo use.
- English should remain available as a fallback and for technical review.
- The language switch should work across organizer, merchant, tourist, public H5, and login surfaces.
- Internal backend names must remain hidden from tourist/public pages in every supported language.

v0.6 should keep the deterministic demo path stable. It must not introduce Qwen/QwenPaw, model APIs, backend workflow changes, real merchants, hardware, payment, map APIs, or registration.

## 3. Problem Statement

The current product is visually acceptable, but the language experience is not aligned with the likely demo audience.

Problems:

- A Macau/UM competition demo will be easier to understand in Chinese.
- English-only UI makes merchant and tourist surfaces feel less locally grounded.
- Directly replacing English strings with Chinese would lose the useful English fallback.
- Ad hoc translation inside components would make later Qwen/QwenPaw wording integration harder.
- Simplified and Traditional Chinese have different audience expectations; Traditional Chinese should not be treated as a blind script conversion only.
- Tests currently assert English UI text, so a language pass can easily break route and role behavior if not planned.

## 4. Language Decision

### 4.1 Supported Locales

v0.6 supports exactly three locales:

| Locale key | Display name | Intended use | Status |
| --- | --- | --- | --- |
| `zh-Hans` | 简体中文 | Default demo language and primary development target | Required |
| `zh-Hant` | 繁體中文 | Macau-facing demo/review language | Required |
| `en` | English | Fallback, technical review, and source comparison | Required |

### 4.2 Default Locale

Default locale:

```text
zh-Hans
```

Fallback order:

```text
selected locale -> en -> key name in development
```

### 4.3 Locale Persistence

Language preference may be stored in browser local storage because it is a UI preference, not an identity/session mechanism.

Rules:

- Do not reintroduce localStorage-based role switching or auth.
- Locale storage key must be explicit, for example `zhiyin.locale`.
- If stored locale is invalid, fall back to `zh-Hans`.
- Public H5 can read and write the same locale preference.

### 4.4 Browser Language Detection

Browser language detection is optional in v0.6.

Recommended behavior:

1. If `zhiyin.locale` exists and is valid, use it.
2. Otherwise use `zh-Hans`.

Do not make browser language detection a blocker for v0.6.

## 5. Scope

### 5.1 In Scope

v0.6 includes:

- Lightweight frontend i18n infrastructure.
- Locale dictionaries for `zh-Hans`, `zh-Hant`, and `en`.
- Language switcher in login, organizer shell, merchant shell, tourist shell, and public H5.
- Translation of all visible product UI copy in active v0.5 routes.
- Translation of deterministic demo content where it is shown as product content.
- Tests for locale switching, fallback behavior, and role boundaries.
- Text gates preventing public/tourist leakage of internal backend names across all locales.
- Updated screenshots showing Simplified and Traditional Chinese.
- Documentation handoff updates.

### 5.2 Out Of Scope

v0.6 must not include:

- Qwen/QwenPaw integration.
- Model-generated translation as a required path.
- Backend core workflow changes.
- Database schema changes for localization.
- Public registration or user profile language settings.
- Full CMS/content-management workflow.
- Real merchant onboarding.
- Real map, POS, payment, traffic, hardware, or IoT integration.
- New UI redesign beyond the minimal layout work needed for language overflow.
- Adding a large i18n framework unless explicitly justified in the implementation plan.

## 6. Product Language Strategy

### 6.1 Preserve Product Meaning, Not Literal English

Translations should preserve role and workflow meaning. Do not blindly translate every English label word-for-word.

Examples:

| English | Simplified Chinese | Traditional Chinese | Notes |
| --- | --- | --- | --- |
| Organizer workspace | 主办方工作台 | 主辦方工作台 | Use for organizer role entry. |
| Merchant workbench | 商户工作台 | 商戶工作台 | Use for merchant surfaces. |
| Visitor route | 游客路线 | 遊客路線 | Use for public/tourist entry. |
| Event workspace | 活动工作区 | 活動工作區 | Organizer planning surface. |
| Exception center | 异常中心 | 異常中心 | Product term for incidents. |
| Recovery suggestion | 恢复建议 | 恢復建議 | Product term for recovery proposal. |
| Evidence trail | 决策依据 | 決策依據 | Better than literal "证据轨迹". |
| Route plan | 路线方案 | 路線方案 | Product term for plan version. |
| Live status | 实时状态 | 即時狀態 | Merchant runtime state. |
| Assigned task | 执行任务 | 執行任務 | Merchant task. |
| Public notice | 游客通知 | 遊客通知 | Visitor-facing notice. |
| Review report | 复盘报告 | 復盤報告 | Organizer review. |

### 6.2 Terms That Must Stay Stable

These product nouns should be translated consistently:


## 7. UX Requirements

### 7.1 Language Switcher

Every major surface must expose a clear language switcher:

| Surface | Placement |
| --- | --- |
| `/login` | Top-right or form-header area; must be visible before login. |
| Organizer shell | Header account/action area. |
| Merchant shell | Header area; must fit mobile width. |
| Tourist shell | Header or bottom-nav-adjacent utility area; must not crowd route actions. |
| Public H5 | Top area near route status; must not require login. |

Switcher labels:

```text
简
繁
EN
```

or:

```text
简体
繁體
English
```

Acceptance:

- Current selected locale is visibly marked.
- Switching locale updates the current page without navigation loss.
- Switching locale does not log the user out.
- Switching locale on public H5 does not require a session.

### 7.2 Responsive Text Requirements

Chinese strings can be shorter than English, but Traditional Chinese labels can still overflow in compact mobile controls.

Requirements:

- Buttons and status pills must not overflow at `390x844`.
- Merchant quick actions must remain touch-friendly in all locales.
- Tourist/public route cards must remain readable in all locales.
- Sidebar/nav labels must not wrap into incoherent two-line fragments on desktop.
- Use layout constraints and wrapping intentionally; do not shrink fonts with viewport width.

### 7.3 Locale-Specific Tone

Simplified Chinese tone:

- Clear, product-oriented, concise.
- Suitable for internal demo walkthrough.
- Avoid overly formal government/report phrasing on merchant/tourist pages.

Traditional Chinese tone:

- Macau-facing but still product-oriented.
- Use `主辦方`, `商戶`, `遊客`, `活動`, `路線`.
- Avoid mainland-only phrasing where a neutral Traditional Chinese phrase is better.

English tone:

- Keep current v0.5 product language as fallback.
- Minor copy cleanup is allowed only if it improves consistency.

## 8. Technical Design

### 8.1 Preferred Implementation

Use a lightweight owned i18n layer:

```text
apps/web/src/i18n/
  I18nProvider.tsx
  locales.ts
  types.ts
  dictionaries/
    en.ts
    zh-Hans.ts
    zh-Hant.ts
```

Recommended API:

```ts
const { locale, setLocale, t } = useI18n();
t("login.title");
t("organizer.dashboard.needsAttention");
```

The first implementation should avoid adding a large dependency unless the implementation plan proves the local layer is insufficient.

### 8.2 Dictionary Shape

Use typed nested dictionaries or flat typed keys. The implementation plan should choose one and stay consistent.

Recommended key categories:

```text
common.*
auth.*
shell.*
organizer.dashboard.*
organizer.workspace.*
organizer.exceptions.*
organizer.review.*
merchant.dashboard.*
merchant.tasks.*
merchant.status.*
merchant.notices.*
tourist.event.*
tourist.route.*
tourist.notices.*
public.event.*
demo.*
errors.*
```

### 8.3 Fallback Behavior

Fallback behavior must be deterministic:

- Missing key in selected locale falls back to English.
- Missing key in English returns the key name in development/test.
- Tests should fail if required production keys are missing from any supported locale.

### 8.4 Dynamic Values

Translations must support interpolation:

```text
current plan v2
7/8 merchants ready
2 story stops
Response time 4 min
```

Recommended API:

```ts
t("organizer.dashboard.currentPlan", { version: 2 })
```

Do not build a full ICU message parser in v0.6 unless necessary.

### 8.5 Dates, Times, Numbers

v0.6 may keep deterministic date/time strings from the backend, but visible labels around them must be localized.

Recommended minimum:

- Keep event date `2026-07-18`.
- Keep time window `18:00-21:30`.
- Localize labels such as `Date`, `Time`, `Area`, `Current plan`.

Do not add complex timezone/date formatting in v0.6.

### 8.6 Demo Data And Content

The deterministic backend can continue returning English-like demo objects. The frontend may map known demo route point/task/notice content through locale dictionaries for the demo event.

Recommended approach:

- UI labels always come from dictionaries.
- Known demo content can be localized in frontend dictionaries under `demo.*`.
- Backend mock JSON does not need to be localized in v0.6.

This avoids backend schema changes while making screenshots and demo flows Chinese-ready.

## 13. Stop Gates

Stop and ask for review if:

- Implementing i18n requires a large dependency or router rewrite.
- Chinese text causes mobile overflow that cannot be fixed with local layout adjustments.
- The implementation starts replacing role/auth logic instead of translating product copy.
- Public/tourist pages expose internal backend names in any locale.
- A translation changes workflow meaning, especially approval, recovery, publication, or merchant status semantics.
- Locale switching breaks login, session persistence, route guards, or public H5 access.
- Backend schema changes appear necessary.
- Qwen/QwenPaw becomes required for translation.

## 14. Recommended Implementation Phases

### Phase A: i18n Infrastructure

Add the provider, locale types, dictionaries, switcher component, fallback behavior, and persistence.

### Phase B: Shells And Login

Translate login and role shells first. This proves language switching works before touching every page.

### Phase C: Organizer Pages

Translate organizer dashboard, events, workspace, exception center, and review.

### Phase D: Merchant Pages

Translate merchant dashboard, tasks, status, and notifications. Check mobile overflow carefully.

### Phase E: Tourist And Public Pages

Translate logged-in tourist route and public H5. Localize known demo route content and notices.

### Phase F: Tests, Screenshots, Handoff

Run unit/build/Playwright/backend regression, inspect screenshots, create `v0.6-i18n-smoke.md`, and update `docs/ai`.
