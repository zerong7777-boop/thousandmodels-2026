# v3.3 Merchant Portal Setup Pack Spec

Status: implemented on 2026-06-30.

## Goal

v3.3 turns merchant participation setup from an organizer-only demo control into a merchant-facing preparation workflow. Merchants should see event context for assigned events, submit event-specific setup evidence, and make readiness gaps visible to organizers before planning.

This phase keeps organizer roster authority, v2.9 planning gates, v3.0 merchant eligibility, v3.1 merchant-fit evidence, v3.2 route-fit evidence, and the deterministic demo path intact.

## Background

v2.9 added organizer-managed event merchant rosters.
v3.0 added merchant network profiles and eligibility checks.
v3.1 used merchant profile data for deterministic planning fit.
v3.2 assembled route points with selected merchant coverage.

The remaining collaboration gap is merchant-side setup. Today the organizer can mark a merchant ready without merchant-submitted evidence. The merchant workbench can show assigned tasks and runtime status, but it does not let a merchant confirm participation constraints, capacity notes, operating-window fit, or preparation readiness for a specific event.

## Scope

v3.3 delivers a Merchant Portal Setup Pack:

1. Event-specific merchant setup fields on `EventMerchantParticipant`.
2. Merchant-owned API endpoints to read assigned event setup context and submit setup evidence.
3. Organizer roster summaries that include merchant-submitted setup evidence and gap reasons.
4. Merchant portal UI for assigned event setup.
5. Organizer workspace UI that shows setup evidence before planning.
6. Planning gate compatibility: non-demo planning remains blocked until participants are confirmed, ready, eligible, and have required setup evidence.
7. Regression coverage for demo compatibility, authorization boundaries, and adjacent v2.9-v3.2 workflows.

## Non-Goals

- No public merchant onboarding, self-registration, invitation emails, password reset, OAuth, SSO, tenant administration, or compliance workflow.
- No contract, settlement, payment, POS, CRM, ERP, inventory, hardware, map, weather, traffic, visitor identity, or real external integration.
- No automatic organizer approval based on merchant input.
- No model/Qwen/QwenPaw-driven readiness approval.
- No automatic plan publication.
- No event archive/delete/clone workflow.

## Product Requirements

### Merchant Setup Data

- `REQ-MSETUP-001`: The backend must store event-specific setup evidence for each selected merchant.
- `REQ-MSETUP-002`: Merchant setup evidence must include:
  - `setup_status`: `not_started`, `submitted`, or `approved`;
  - `capacity_commitment`: `low`, `medium`, or `high`;
  - `staffing_ready`: boolean;
  - `stock_ready`: boolean;
  - `indoor_backup_ready`: boolean;
  - `operating_window_confirmed`: boolean;
  - `merchant_contact_name`;
  - `merchant_contact_phone`;
  - `merchant_notes`;
  - `submitted_at`;
  - `submitted_by`.
- `REQ-MSETUP-003`: Existing records without v3.3 fields must validate with backward-compatible defaults.
- `REQ-MSETUP-004`: Organizer roster replacement must preserve existing setup evidence for merchants that remain selected.
- `REQ-MSETUP-005`: Removing a merchant from an event roster removes that event-specific participant record.

### Merchant-Facing API

- `REQ-MAPI-001`: A merchant must be able to list event assignments that include their participant record, event summary, eligibility, current plan version, public release status, and setup gaps.
- `REQ-MAPI-002`: A merchant must be able to read setup context for one assigned event.
- `REQ-MAPI-003`: A merchant must be able to submit setup evidence only for their own merchant id.
- `REQ-MAPI-004`: A merchant cannot mutate `participation_status`, `readiness_status`, organizer notes, selected roster membership, or plan state through merchant setup endpoints.
- `REQ-MAPI-005`: Organizer users may read merchant setup through organizer roster APIs, but merchant setup submission should remain merchant-owned in this phase.
- `REQ-MAPI-006`: Merchant setup submission must use the existing session and CSRF/mutation-origin boundary.

### Setup Gaps And Planning Gate

- `REQ-GATE-001`: `summarize_event_merchants()` must expose setup gaps per participant.
- `REQ-GATE-002`: Required setup evidence for planning is:
  - setup submitted;
  - contact name and phone present;
  - capacity commitment present;
  - staffing ready;
  - stock ready;
  - operating window confirmed.
- `REQ-GATE-003`: Indoor backup readiness is required only when the event goal, constraints, theme preferences, or priority rules mention rain, rainy-day, indoor, covered, backup, family, or rest.
- `REQ-GATE-004`: Non-demo planning must remain blocked until every selected participant is confirmed, organizer-ready, eligible, and setup-complete.
- `REQ-GATE-005`: Organizer mark-ready remains an explicit organizer decision and must not be replaced by merchant submission.
- `REQ-GATE-006`: Demo event generation remains backward compatible when `demo-night-tour` has no manual roster.

### Organizer UI

- `REQ-OUI-001`: Organizer event workspace merchant setup panel must show merchant-submitted setup status, contact, capacity commitment, readiness booleans, notes, and setup gaps.
- `REQ-OUI-002`: Organizer can still select merchants and mark eligible participants ready, but the panel should explain when setup evidence is missing.
- `REQ-OUI-003`: The Build route plan action must remain disabled for non-demo events until roster and merchant setup are ready.
- `REQ-OUI-004`: New visible copy must use the existing i18n dictionary layer.

### Merchant UI

- `REQ-MUI-001`: Merchant portal must show assigned events rather than only the hard-coded demo event.
- `REQ-MUI-002`: Merchant setup UI must show event title, area, date, time window, role hint, participation status, readiness status, setup status, eligibility, gaps, and current notes.
- `REQ-MUI-003`: Merchant setup UI must let merchants submit capacity commitment, contact name, contact phone, staffing readiness, stock readiness, indoor backup readiness, operating window confirmation, and notes.
- `REQ-MUI-004`: Merchant setup UI must not expose raw backend/model/agent terms.
- `REQ-MUI-005`: Merchant navigation can keep demo defaults, but assigned-event setup must be reachable for the authenticated merchant.

### Authorization And Safety

- `REQ-AUTH-001`: Merchant users can only read and submit setup for `user.merchant_id`.
- `REQ-AUTH-002`: Organizer users cannot use merchant-owned endpoints to impersonate merchant setup submission unless a separate audited proxy flow is designed later.
- `REQ-AUTH-003`: Tourist and unauthenticated users cannot access merchant setup.
- `REQ-AUTH-004`: Setup submission must not approve plans, publish pages, generate packages, or mutate runtime state.

## Proposed Data Model

### EventMerchantParticipant Additions

- `setup_status: Literal["not_started", "submitted", "approved"] = "not_started"`
- `capacity_commitment: Literal["low", "medium", "high"] | None = None`
- `staffing_ready: bool = False`
- `stock_ready: bool = False`
- `indoor_backup_ready: bool = False`
- `operating_window_confirmed: bool = False`
- `merchant_contact_name: str = ""`
- `merchant_contact_phone: str = ""`
- `merchant_notes: str = ""`
- `submitted_at: str | None = None`
- `submitted_by: str | None = None`
- `setup_gaps: list[str] = []`

`setup_gaps` is response evidence and can be computed during summaries rather than persisted.

### MerchantSetupSubmitRequest

- `capacity_commitment: Literal["low", "medium", "high"]`
- `staffing_ready: bool`
- `stock_ready: bool`
- `indoor_backup_ready: bool`
- `operating_window_confirmed: bool`
- `merchant_contact_name: str`
- `merchant_contact_phone: str`
- `merchant_notes: str = ""`

### MerchantAssignedEvent

- `event: EventSummary`
- `participant: EventMerchantParticipant`
- `eligibility: MerchantEligibility | None`
- `setup_gaps: list[str]`
- `ready_for_planning: bool`

## API Behavior

New merchant endpoints:

- `GET /api/merchants/me/events`
  - Requires merchant session.
  - Returns assigned event setup contexts for the authenticated merchant.

- `GET /api/merchants/me/events/{event_id}/setup`
  - Requires merchant session.
  - Returns one assigned event setup context.
  - Returns 404 if the merchant is not selected for that event.

- `PATCH /api/merchants/me/events/{event_id}/setup`
  - Requires merchant session and mutation origin/CSRF verification.
  - Updates only merchant-owned setup evidence fields.
  - Sets `setup_status="submitted"`, `submitted_at`, and `submitted_by`.
  - Returns the updated setup context.

Enhanced organizer endpoints:

- `GET /api/events/{event_id}/merchant-roster`
  - Adds setup evidence and setup gaps inside each participant.

- `PATCH /api/events/{event_id}/merchant-roster/{merchant_id}`
  - Keeps organizer authority over participation/readiness/role/organizer notes.
  - Must not erase merchant setup evidence unless the participant is removed from roster.

Enhanced behavior:

- `POST /api/events/{event_id}/generate-plan`
  - Keeps v2.9-v3.2 gates.
  - Blocks non-demo planning if any selected participant has required setup gaps.

## Error Handling

- Missing event assignment returns 404.
- Merchant id mismatch returns 403 through the existing authenticated merchant boundary.
- Missing required setup fields returns 422 through Pydantic validation.
- Planning with incomplete setup returns 400 with `event merchant setup required`.
- Ineligible merchants still cannot be marked ready.
- Existing demo paths continue to seed and plan without a manual merchant setup flow when no demo roster exists.

## Acceptance Criteria

v3.3 is complete when:

1. Merchants can read event assignment context and submit event-specific setup evidence.
2. Merchant setup evidence is visible in organizer roster setup.
3. Non-demo planning remains blocked until setup evidence and organizer readiness are both complete.
4. Merchant setup submission does not approve readiness, plans, publication, or runtime state.
5. Auth boundaries prevent merchants from reading or updating another merchant's setup.
6. Existing v2.9-v3.2 planning, merchant-fit, and route-fit evidence remain compatible.
7. The deterministic demo path remains reliable.
8. Full verification passes.
