# v3.4 Event Operations Command Center Pack Spec

Status: implemented on 2026-06-30.

## Goal

v3.4 turns the organizer event workspace into an operations command center for one event. The product should answer a concrete operator question: "Can this event go live now, what blocks it, and what evidence led here?"

This phase consolidates existing event setup, route plan, event page, merchant package, incident, notice, and audit-log state into one readiness surface. It does not add real external integrations and does not weaken existing human approval gates.

## Background

v2.9 introduced event merchant setup.
v3.0-v3.2 strengthened merchant and route planning quality.
v3.3 added merchant-side setup evidence.

The remaining product gap is operational command. A commercial organizer should not need to infer readiness by jumping between route plans, merchant setup, event page cards, package cards, incidents, and audit trails. The system should produce a direct launch readiness summary with blockers, warnings, and recent action evidence.

## Scope

v3.4 delivers an Event Operations Command Center Pack:

1. Event-level operations summary API for organizers.
2. Pre-launch checks over merchant setup, plan approval, visitor page publication, merchant package readiness, incidents, notices, and audit evidence.
3. Structured readiness status: `ready`, `warning`, or `blocked`.
4. Action items that explain the next operator steps without mutating state automatically.
5. Event-scoped action timeline derived from existing audit logs.
6. Organizer workspace command-center panel near the top of the event workspace.
7. Regression coverage for demo compatibility and adjacent v2.9-v3.3 workflows.

## Non-Goals

- No POS, inventory, weather, maps, traffic, CRM, ERP, payment, identity, email, SMS, or public provider integration.
- No automatic plan approval, page publication, package generation, notice publication, or incident recovery.
- No new operational workflow engine.
- No new database table unless existing records cannot represent the evidence.
- No tenant administration, RBAC expansion, billing, audit export, or compliance workflow.
- No QwenPaw live-provider optimization in this phase.

## Product Requirements

### Operations Summary

- `REQ-OPS-001`: The backend must expose a single event operations summary for organizer users.
- `REQ-OPS-002`: The summary must include the event summary when the event exists.
- `REQ-OPS-003`: Missing events must return 404.
- `REQ-OPS-004`: The summary must classify overall status as:
  - `blocked` if any blocking launch check is blocked;
  - `warning` if no blocking check is blocked but at least one check is warning;
  - `ready` when all launch-critical checks are ready.
- `REQ-OPS-005`: The summary must include separate blocker and warning counts.

### Launch Checks

- `REQ-CHECK-001`: Merchant setup check must use existing event merchant setup readiness. Non-demo events are blocked until selected merchants are planning-ready.
- `REQ-CHECK-002`: Plan approval check is blocked until the current plan exists and is approved.
- `REQ-CHECK-003`: Event page check is blocked until a page for the current plan is published.
- `REQ-CHECK-004`: Merchant package check is blocked until every current-plan merchant has an active package for that plan.
- `REQ-CHECK-005`: Incident check is blocked when high-severity incidents are open or proposal-ready, and warning when lower-severity incidents are open or proposal-ready.
- `REQ-CHECK-006`: Notice check is warning-only when there are active incidents but no public notice evidence.
- `REQ-CHECK-007`: Audit evidence check is warning-only when there is no recent event action history.
- `REQ-CHECK-008`: Demo event compatibility must be preserved: demo fallback behavior may show warnings but must not break existing demo planning and page flows.

### Action Items

- `REQ-ACTION-001`: The summary must provide operator-readable action items derived from failed or warning checks.
- `REQ-ACTION-002`: Action items must identify severity, status, and optional target area.
- `REQ-ACTION-003`: Action items are advisory; they must not approve, publish, generate, or recover anything.

### Timeline Evidence

- `REQ-TIME-001`: The summary must include a bounded event timeline derived from existing `AuditLog` records.
- `REQ-TIME-002`: Timeline items must preserve actor type, actor id, action type, timestamp, note, and an evidence reference.
- `REQ-TIME-003`: Timeline sorting must be newest first.
- `REQ-TIME-004`: Timeline must not expose raw model-provider terms to public or merchant pages. This endpoint is organizer-only.

### Organizer UI

- `REQ-UI-001`: The event workspace must show a command-center panel above lower-level workflow cards.
- `REQ-UI-002`: The panel must show overall readiness, blocker/warning counts, launch checks, action items, and recent timeline.
- `REQ-UI-003`: The panel must use existing design primitives and i18n dictionaries.
- `REQ-UI-004`: The panel must fit desktop and mobile layouts without overlapping text.
- `REQ-UI-005`: Existing route plan, merchant setup, event page, merchant package, and evidence sections must remain available.

### Safety

- `REQ-SAFE-001`: v3.4 must preserve organizer-only access to operations state.
- `REQ-SAFE-002`: No operation summary call may mutate event state.
- `REQ-SAFE-003`: Human approval remains required for plan approval, page publication, package generation, operation-suggestion approval, and recovery approval.
- `REQ-SAFE-004`: Existing deterministic demo and CI release gates must remain green.

## Proposed API

`GET /api/events/{event_id}/operations-summary`

Organizer-only response shape:

```json
{
  "event": {},
  "overall_status": "blocked",
  "blocker_count": 2,
  "warning_count": 1,
  "checks": [],
  "action_items": [],
  "timeline": [],
  "incident_summary": {},
  "package_summary": {},
  "notice_summary": {}
}
```

### New Schemas

- `OperationReadinessCheck`
  - `check_id`
  - `label`
  - `status`: `ready`, `warning`, `blocked`
  - `summary`
  - `blocking`
  - `evidence_refs`

- `OperationActionItem`
  - `action_id`
  - `label`
  - `status`: `todo`, `watch`, `done`
  - `severity`: `info`, `warning`, `critical`
  - `target`
  - `evidence_refs`

- `OperationTimelineItem`
  - `item_id`
  - `actor_type`
  - `actor_id`
  - `action_type`
  - `summary`
  - `timestamp`
  - `evidence_ref`

- `EventOperationsSummary`
  - `event`
  - `overall_status`
  - `blocker_count`
  - `warning_count`
  - `checks`
  - `action_items`
  - `timeline`
  - `incident_summary`
  - `package_summary`
  - `notice_summary`

## Acceptance Criteria

v3.4 is complete when:

1. Organizer can fetch one event-level operations summary.
2. The summary correctly blocks incomplete merchant setup, missing approved plan, missing published page, missing active merchant packages, and unresolved high-severity incidents.
3. Lower-severity incidents, missing notice evidence, and missing timeline evidence surface as warnings.
4. Organizer workspace renders the command-center panel with checks, action items, and timeline.
5. Summary and panel do not mutate event state.
6. Existing v2.9-v3.3 workflows and deterministic demo behavior remain compatible.
7. Focused and full verification pass.
