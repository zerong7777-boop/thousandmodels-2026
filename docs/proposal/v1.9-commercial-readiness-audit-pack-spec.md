# Zhiyin Haojiang v1.9 Commercial Readiness Audit Pack Spec

Date: 2026-06-27
Project root: `<PROJECT_ROOT>`
Status: design spec for next phase

## 1. Purpose

v1.8.1 closed the QwenPaw advisory qualification spike with real local
QwenPaw evidence. The next useful phase is not another feature spike. It is a
commercial-readiness audit pack that turns the current implementation into a
clear investment and remediation map.

The current project is a strong competition MVP:

```text
deterministic business loop
+ role-separated organizer / merchant / tourist / public surfaces
+ backend-backed demo login
+ verified API and frontend test suites
+ optional Qwen and QwenPaw guarded evidence paths
```

It is not yet a commercial production application:

```text
demo accounts
+ SQLite JSON record store
+ fixed demo CSRF header
+ anonymous public mutation endpoints
+ fake QwenPaw product path
+ no CI/CD, deployment, migrations, observability, or production secrets model
```

v1.9 should create a formal audit pack that names these gaps precisely, ranks
them, and gives future implementation phases a grounded work queue.

## 3. Commercial readiness levels

v1.9 defines readiness levels so the project does not blur demo, pilot, and
production claims.

| Level | Name | Definition | Current fit |
| --- | --- | --- | --- |
| `CR0` | Competition demo | Local deterministic flow with controlled evidence and no real external business dependencies | Current default path |
| `CR1` | Hardened demo | CI, clean repo state, explicit risk register, sanitized evidence, repeatable local runbook | v1.9 target |
| `CR2` | Pilot-ready | Production-like auth, database, migrations, deployment, observability, rate limits, and provider governance | Future target |
| `CR3` | Commercial production | Multi-tenant operations, real integrations, security review, incident response, support, data governance, SLAs | Out of current scope |

The v1.9 pack should move documentation and evidence from `CR0` toward `CR1`.
It should not claim `CR2` or `CR3` readiness.

## 4. Scope decision

v1.9 is:

- a commercial-readiness audit pack
- a P0/P1 gap register
- a current-state evidence capture
- a GitHub and repo hygiene assessment
- a production-readiness roadmap input
- a boundary document that prevents overclaiming

v1.9 is not:

- a production migration
- a Postgres or cloud deployment implementation
- a QwenPaw product integration
- a real merchant, POS, payment, map, weather, traffic, or identity integration
- a new feature surface for judges or users
- a replacement for the deterministic demo path
- a claim that optional Qwen/QwenPaw evidence is business success

## 5. Audit dimensions

The audit pack must grade each dimension as `pass`, `partial`, `gap`, or
`blocked`.

| Dimension | Current signal | Commercial target | Priority |
| --- | --- | --- | --- |
| Repo hygiene | branch pushed; main merged; dirty PNG files remain | clean working tree, branch policy, release tags | P0 |
| CI/CD | no `.github` workflows | enforced API, web, build, Playwright, secret-scan gates | P0 |
| Data persistence | SQLite JSON payload store | relational schema, migrations, backup/restore, transaction strategy | P0 |
| Auth and authorization | demo accounts and roles | account lifecycle, resource-level ACL, secure cookies, rate limits | P0 |
| CSRF and public mutation | fixed demo CSRF header | real CSRF, abuse control, idempotency, visitor/session model | P0 |
| AI/provider governance | deterministic fallback plus guarded live smoke | provider registry, evals, budget, replay, approval boundary | P0 |
| Observability | audit records only | structured logs, metrics, traces, error reporting, alerting | P0 |
| Deployment | local dev commands | production deployment topology, env separation, health/readiness | P0 |

| Frontend performance | build passes with large chunk warning | route code splitting, bundle budget, accessibility smoke | P1 |
| Documentation | strong proposal history | commercial runbook, risk register, release notes, operator guide | P1 |

## 6. P0 requirements

### 6.1 Repository and CI requirements

- `REQ-CI-001`: v1.9 must record whether `.github/workflows` exists.
- `REQ-CI-002`: v1.9 must recommend a minimal CI workflow that runs API tests,
  web tests, web build, and strict secret scans.
- `REQ-CI-003`: v1.9 must record dirty working-tree files and explicitly
  exclude historical PNG diffs from audit commits unless the user asks.
- `REQ-CI-004`: v1.9 must record current branch, upstream, default branch, open
  PR count, open issue count, and latest remote heads.

### 6.2 Data and state requirements

- `REQ-DATA-001`: v1.9 must classify the current SQLite JSON record store as
  acceptable for `CR0` and insufficient for `CR2`.
- `REQ-DATA-002`: v1.9 must define the target data plan: relational tables,
  migrations, backup/restore, and transaction boundaries.
- `REQ-DATA-003`: v1.9 must identify demo seed data and mock metrics as
  non-production data.

### 6.3 Security requirements

- `REQ-SEC-001`: v1.9 must map demo auth, sessions, cookies, CSRF, CORS, and
  public mutation endpoints to commercial risks.
- `REQ-SEC-002`: v1.9 must recommend OWASP ASVS as the future security control
  checklist, without claiming ASVS compliance.
- `REQ-SEC-003`: v1.9 must require rate limiting and abuse controls before any
  public claim, coupon, redeem, scan, or anonymous interaction endpoint is
  exposed beyond local demo.
- `REQ-SEC-004`: v1.9 must keep the existing secret-scan discipline and avoid
  storing provider keys, local absolute paths, authorization headers, or raw
  provider payloads in evidence.

### 6.4 AI governance requirements

- `REQ-AI-001`: v1.9 must distinguish deterministic product path, optional
  Qwen draft path, optional QwenPaw live advisory smoke, and fake QwenPaw
  shadow orchestration.
- `REQ-AI-002`: v1.9 must state that QwenPaw `advisory_qualified` evidence is
  not production orchestration.
- `REQ-AI-003`: v1.9 must require provider budget, latency, prompt/version
  tracking, replay, failure classification, and approval boundaries before any
  live provider becomes product runtime.

### 6.5 Operations requirements

- `REQ-OPS-001`: v1.9 must identify missing deployment, environment, health,
  readiness, logging, metric, tracing, and alerting controls.
- `REQ-OPS-002`: v1.9 must recommend production runbooks before pilot use.
- `REQ-OPS-003`: v1.9 must identify the Vite bundle warning as a P1
  performance follow-up, not a P0 blocker for the audit pack.

## 7. P1 requirements

- `REQ-PROD-001`: Define integration contracts for merchants, touchpoints,
  coupon claims, redemption, POS/payment, maps, weather, and traffic before
  implementing real integrations.
- `REQ-PROD-002`: Add a product analytics and event taxonomy plan before
  claiming operational intelligence.
- `REQ-PROD-003`: Add accessibility and mobile-device QA targets for public H5
  and merchant flows.
- `REQ-PROD-004`: Add operator documentation for reset, demo, seed, rollback,
  and evidence capture.
- `REQ-PROD-005`: Add a release-readiness checklist that separates competition
  claims, pilot claims, and production claims.

## 9. Acceptance criteria

v1.9 is complete when:

1. The audit report records the exact current test/build/GitHub evidence.
2. The readiness matrix covers every P0 and P1 dimension in this spec.
3. Each gap has a current-state citation to a local file, command result, or
   GitHub state.
4. Each P0 gap has a recommended owner phase and a concrete exit criterion.
5. The report explicitly says the project is `CR0` with a path toward `CR1`,
   not `CR2` or `CR3`.
6. The report preserves the Qwen/QwenPaw boundary: optional advisory evidence
   is not product orchestration.
7. Secret/local-path scans pass for the new v1.9 evidence artifacts.
8. API tests, web tests, and web build are rerun or explicitly recorded as not
   rerun with a reason.
9. Historical PNG diffs are not staged into the v1.9 commit unless the user
   explicitly requests it.
10. `docs/ai/NEXT.md` recommends the next remediation phase after the audit
    pack, not another ambiguous spike.

## 10. Stop gates

Stop and ask for review if:

- the implementation would add production infrastructure instead of auditing it
- the audit requires storing real provider keys or raw provider payloads
- the worktree contains unrelated changes that would be mixed into the audit
  commit
- a planned claim would imply commercial or pilot readiness without evidence
- a QwenPaw result would be described as authoritative workflow execution
- CI or deployment work would require external credentials not already present
