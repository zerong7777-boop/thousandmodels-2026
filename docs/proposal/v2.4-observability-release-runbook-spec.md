# v2.4 Observability And Release Runbook Spec

Date: 2026-06-27

Status: proposed.

## Purpose

v2.4 makes failures diagnosable and releases repeatable. After CI, auth boundary, persistence migrations, and deployment readiness are in place, the project needs a lightweight operational layer:

- structured logs;
- request IDs;
- consistent error payloads;
- release checklist;
- smoke evidence retention;
- incident response runbooks.

This phase avoids a full monitoring vendor integration. It establishes local and staging-friendly observability primitives that can later be wired into managed tooling.

## Readiness Target

v2.4 moves the project from "staging-deployable" to "operable beta candidate".

## In Scope

- Request ID middleware.
- Structured JSON logs for API requests and key domain actions.
- Consistent API error envelope.
- Basic counters emitted through logs or a lightweight `/api/metrics` JSON endpoint.
- Release checklist document.
- Incident response runbook for CI failure, API startup failure, auth failure, migration failure, QwenPaw live-smoke failure, and public H5 failure.
- Smoke evidence retention policy.
- Tests for request ID propagation, redaction, error envelope, and runbook presence.

## Out Of Scope

- Vendor monitoring setup.
- OpenTelemetry collector deployment.
- Prometheus/Grafana hosting.
- PagerDuty or alert routing.
- SLO commitments.
- Real customer incident management.

## Required Behavior

### Request IDs

Every API response should include `X-Request-ID`.

If a request includes `X-Request-ID`, reuse it after validating it is bounded and safe. Otherwise generate one.

### Structured Logs

API logs should include:

- timestamp;
- level;
- request ID;
- method;
- path template when available;
- status code;
- duration;
- actor role and user ID when authenticated;
- operation name for domain actions.

Logs must not include cookies, bearer tokens, API keys, raw provider payloads, or local absolute paths.

### Error Envelope

Unhandled errors and HTTP exceptions should follow a consistent shape:

```json
{
  "error": {
    "code": "string",
    "message": "string",
    "request_id": "string"
  }
}
```

The public H5 must not expose internal stack traces.

### Metrics Surface

Add a minimal JSON endpoint or log counters for:

- health checks;
- auth failures;
- agent run outcomes;
- QwenPaw advisory outcomes;
- public touchpoint interactions;
- public coupon claims/redemptions;
- recovery approvals;
- review report generation.

### Release Runbook

The runbook must list:

- pre-release commands;
- CI checks;
- migration command;
- deployment smoke command;
- rollback notes;
- known acceptable warnings;
- release evidence paths.

### Incident Response Runbook

The runbook must list first checks and escalation paths for:

- CI red;
- API will not start;
- login/session failure;
- migration failure;
- QwenPaw local service unreachable;
- public H5 unavailable;
- repo hygiene failure.

## Tests

Tests must cover:

- request IDs are returned and logged;
- unsafe request IDs are replaced;
- error envelope includes request ID;
- logs redact sensitive headers;
- runbook files exist and include required sections;
- metrics endpoint or log counters do not expose secrets.

## Acceptance Criteria

- API tests pass.
- Web tests and build pass.
- Repo hygiene passes.
- Release and incident runbooks exist.
- A failing request can be traced by request ID.
- Documentation states that v2.4 is a beta operations baseline, not full observability maturity.

## Commercial Readiness Impact

v2.4 closes the first operational gap from v1.9 `CR-P1-002` and supports deployment discipline from `CR-P0-005`. It does not close external alerting, formal SLOs, or customer incident operations.
