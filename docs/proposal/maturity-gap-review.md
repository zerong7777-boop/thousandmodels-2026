# 智引濠江 MVP Maturity Gap Review

版本：v0.1  
日期：2026-06-07  
任务：`20260518-thousandmodels-ai-competition`  
项目根目录：`<PROJECT_ROOT>`  
审查对象：当前 deterministic clickable prototype  
结论：当前版本是技术脚手架和可点击演示壳，不是成熟 MVP，也不是可直接用于比赛展示的最终候选版本。

## 1. Review 目的

这份文档拆解当前“玩具版”原型的问题，避免继续在一个薄弱方向上堆 UI 或接模型。

当前版本已经证明：

- FastAPI + React/Vite/Ant Design 可以跑起来。
- 固定 demo loop 可以串通。
- 不依赖 `DASHSCOPE_API_KEY` 的 deterministic fallback 可用。

但它还没有证明：

- 这是一个真实 Agent 系统，而不是规则模板生成器。
- 它解决了澳门旧区文旅活动组织中的真实协同问题。
- 主办方、商户、游客三方协同具有不可替代性。
- 动态恢复是真正的重规划，而不是追加通知。
- 复盘报告来自运行数据，而不是总结话术。
- 这个项目相对普通旅游推荐、活动策划生成器有足够差异化。

因此，当前阶段应重新定义为：

> `v0.1 technical scaffold / clickable prototype`

不应称为 mature MVP。

## 2. 总体判断

当前项目最大问题不是工程不通，而是“评委证据链”不成立。

系统目前更像：

1. 固定活动 brief。
2. 固定规则生成一套方案。
3. 按钮触发两个异常。
4. 追加一条游客通知。
5. 输出一份复盘文本。

这条链路可演示，但缺少智能体项目应有的关键证据：

- 可观察的任务拆解。
- 子 Agent 的结构化输出。
- 工具调用与状态读取。
- 人工确认前后的对象版本变化。
- 基于运营指标的复盘。
- 场景数据可信度。

换句话说，当前版本完成了“demo 的壳”，还没有完成“项目的理由”。

## 4. 逐项问题拆解

### G1. 问题定义不够真实

当前定位是“旧区文旅活动编排 Agent”，方向是对的，但问题表述还停留在概念层。

当前 spec 强调“不是普通旅游推荐 App”，但没有足够说明：

- 旧区活动主办方为什么难。
- 商户协同为什么难。
- 为什么库存、天气、排队、居民影响会让活动执行失控。
- AI Agent 在其中替代或增强了哪类人工工作。

当前 demo 给人的感觉更像“输入一个活动主题，生成一个路线和文案”。这和普通旅游推荐、活动策划生成器的边界不够清楚。

v0.2 需要补强：

- 定义一个更具体的业务问题：旧区活动不是路线推荐，而是多方执行协调。
- 说明主办方要同时处理：活动目标、预算、商户承载、路线节奏、天气、库存、游客通知、复盘。
- 给出没有 Agent 时的低效流程：人工表格、微信群沟通、临时改路线、无法沉淀经验。
- 给出 Agent 的价值：把策划、执行、异常、复盘变成可追踪的工作流。

### G2. Agent 编排不可见

spec 写了“总策划 Agent 调度子 Agent”，但当前实现没有真正呈现这个机制。

当前代码中的核心生成逻辑集中在：

- `apps/api/app/agents/deterministic.py`
- `DeterministicAgentBackend.plan_event()`

这更接近一个 deterministic rule planner，而不是多 Agent 系统。

主要缺口：

- 没有 `CulturalNarrativeAgent`、`RoutePlannerAgent`、`MerchantMatcherAgent`、`RiskAgent`、`RecoveryAgent` 的独立结构。
- 没有每个 Agent 的输入/输出 schema。
- 没有 Agent 之间的依赖关系。
- 没有 Agent 调用 trace。
- 没有“工具调用结果如何影响 Agent 输出”的展示。

v0.2 需要补强：

- 引入 `AgentStep` / `AgentTrace` 数据对象。
- 每次生成方案时记录：
  - Agent 名称。
  - 输入引用。
  - 调用的工具。
  - 结构化输出。
  - 决策理由。
  - 是否需要人工确认。
- 前端增加“Agent 工作流”面板，而不是只展示最终方案。

最低可接受方案不是立即接大模型，而是让 deterministic path 也模拟真实 Agent trace。否则即使后面接 Qwen，也只是把模板文本替换成模型文本。

### G3. 动态恢复不是真重规划

当前异常恢复能触发 `RecoveryAction`，但它没有真正改变活动对象。

当前行为大致是：

- 库存不足或强降雨触发恢复建议。
- 主办方确认。
- 系统向游客通知列表追加一条 notice。

这不足以支撑“动态重规划”的说法。

真正的动态恢复至少应改变以下对象之一：

- `EventPlan.version`
- `EventPlan.route`
- `MerchantExecutionPack`
- `MerchantRuntimeState`
- `PublicEvent`
- `AuditLog`

当前版本只更新通知，评委会很容易认为这是 if/else 文案切换。

v0.2 需要补强：

- `RecoveryAction` 应包含 `patch` 或 `proposed_changes`：
  - 替换路线节点。
  - 替换商户。
  - 降低某商户导流权重。
  - 修改商户执行包。
  - 修改游客 H5 点位和任务。
- 主办方确认后生成 `EventPlan v2`。
- 前端展示 v1 -> v2 diff。
- 游客端展示实际路线变化，而不是只展示通知。

### G4. 商户端没有形成闭环

spec 要求商户能更新库存和排队状态，但当前商户端更像展示页。

当前后端有状态更新接口：

- `POST /api/merchants/{merchant_id}/runtime-state`

但前端商户端的状态控件没有真正提交更新。当前库存不足异常主要由主办方后台按钮触发，而不是由商户状态触发。

这会削弱“三方协同”的核心叙事。

v0.2 需要补强：

- 商户端可以更新：
  - 库存：正常 / 紧张 / 售罄。
  - 排队：正常 / 忙碌 / 过载。
  - 是否可接待游客。
  - 临时备注。
- 状态变更后，主办方 Dashboard 出现待处理异常。
- Recovery Agent 根据商户状态生成候选恢复动作。
- 主办方确认后才对游客端发布。

这样才是“商户 -> Agent -> 主办方 -> 游客”的闭环。

### G5. 数据模型太薄

当前 mock 数据能支撑页面显示，但不足以支撑决策。

商户字段已有基础信息，但缺少运营决策需要的数据：

- 当前库存量或库存等级。
- 每 15 分钟可接待人数。
- 当前排队人数。
- 每个活动任务所需准备时间。
- 是否适合亲子、雨天、夜间、无障碍。
- 商户优先级和主办方约束。
- 居民影响风险。
- 预算补贴需求。

路线也只是字符串数组，没有结构化点位对象。

v0.2 需要补强：

- 引入 `RoutePoint`：
  - point_id
  - name
  - type
  - indoor/outdoor
  - estimated_stay_minutes
  - story
  - crowd_risk
  - rainy_day_score
- 引入 `OperationalMetric`：
  - expected_visitors
  - completed_checkins
  - queue_minutes
  - merchant_load
  - budget_spent_mop
- 引入 `MerchantConstraint` 或扩展 `MerchantProfile`。

没有这些数据，Agent 决策没有依据。

### G6. 游客 H5 缺少文化内容

当前游客 H5 能显示路线、通知和打卡任务，但文化和商户内容偏空。

spec 里要求：

- 文化故事。
- 商户优惠。
- 打卡任务。
- 临时通知。

当前实现主要是：

- `route: string[]`
- 固定 `checkin_tasks`
- `marketing_content`
- `notices`

这会让游客端看起来像路线列表，而不是澳门旧区活动入口。

v0.2 需要补强：

- 每个路线点显示：
  - 站点故事。
  - 推荐停留时间。
  - 关联商户。
  - 打卡任务。
  - 优惠或互动。
  - 当前状态。
- 异常恢复后，游客端应显示替换后的点位和原因。
- 游客端应更像活动页面，而不是后台数据预览。

### G7. 复盘报告没有运营证据

当前复盘报告能生成，但证据较弱。

主要问题：

- 没有游客访问量。
- 没有打卡完成率。
- 没有商户接待量。
- 没有预算执行。
- 没有异常前后的效果对比。
- 没有“下次活动参数如何调整”的具体建议。

这使复盘报告更像自然语言总结，而不是运营智能体的输出。

v0.2 需要补强：

- 增加 mock 指标：
  - H5 访问量。
  - 各点位打卡数。
  - 商户接待量。
  - 平均排队时间。
  - 异常处理耗时。
  - 预算计划 vs 预算消耗。
- 复盘报告引用这些指标。
- 输出“下次活动建议”时要绑定指标原因。

例如：

> m001 在 19:10 前库存耗尽，说明试吃任务配额过高；下次应降低导流权重或提前设置备用商户。

这才有复盘价值。

### G8. 审计记录不能说明 Agent 推理

当前审计记录已经有动作流水，但不够解释智能体决策。

现有审计更像：

- seed demo
- generate plan
- approve plan
- trigger inventory
- approve recovery

缺少：

- Agent 为什么选这个商户。
- 工具返回了什么。
- 哪些约束被满足或违反。
- 主办方确认前后的差异。
- 哪些信息被发布到游客端。

v0.2 需要补强：

- `AuditLog` 应继续保留。
- 新增 `AgentTrace`，不要把所有东西塞进审计流水。
- 审计回答“谁做了什么”；Agent trace 回答“AI 为什么这么建议”。
- 前端应分别展示：
  - Agent 工作流。
  - 工具调用结果。
  - 人工确认记录。

这对官方“智能体能力”和“实践文章”都很关键。

### G9. Qwen/QwenPaw 价值不明确

当前实现有 Qwen adapter skeleton，但它没有真正参与核心能力。

这不是短期错误，因为 deterministic fallback 对演示稳定很重要。但从比赛角度看，必须说明 Qwen/QwenPaw 最终承担什么角色。

当前缺口：

- Qwen 只是可选 API adapter。
- QwenPaw 与多 Agent 编排关系不清。
- 没有说明哪些模块使用模型，哪些模块保持规则。
- 没有结构化输出校验和 fallback 策略。

v0.2 需要补强：

- 定义模型使用边界：
  - 文化叙事 Agent：适合 Qwen。
  - 宣传内容 Agent：适合 Qwen。
  - 风险说明 Agent：可用 Qwen + 规则校验。
  - 预算、库存、路线约束：应由规则工具校验。
- 定义 QwenPaw 的价值：
  - Agent 编排。
  - Skills / tools 调用。
  - trace 和任务管理。
- 保留 deterministic fallback，避免现场演示被外部 API 影响。

### G10. UI 像后台模板，不像决策工作台

当前 Ant Design 页面比手写 CSS 更专业，但信息架构还偏“后台模板”。

问题不在组件库，而在演示视角：

- 评委看到的是按钮和卡片，不一定看到 AI 如何工作。
- 方案、恢复、复盘没有形成强叙事。
- Agent trace、状态变化、版本 diff 不突出。
- 商户端和游客端更像附属页，没有体现三方协同。

v0.2 需要补强：

- 主办方页改成“运营指挥台”：
  - 左侧：活动状态和异常。
  - 中间：Agent 工作流和方案版本。
  - 右侧：待确认动作和影响范围。
- 明确展示：
  - 当前计划版本。
  - 当前异常。
  - Agent 建议。
  - 主办方确认。
  - 对游客端/商户端影响。
- 商户端要能触发状态变化。
- 游客端要展示变化后的实际体验。

### G11. Spec 和实现有多处不一致

当前 spec 写得比实现强很多。这本身不是问题，但如果不标注状态，会误导后续开发和评审准备。

典型不一致：

- spec 要自然语言或表单创建，当前是固定 brief。
- spec 要多 Agent 调度，当前是单 planner。
- spec 要商户端状态更新，当前前端未形成更新闭环。
- spec 要游客端文化故事，当前没有结构化站点故事。
- spec 要复盘预算执行概览，当前没有预算执行数据。
- spec 要展示 Agent 调用记录，当前审计记录不足以证明调用链。

v0.2 需要补强：

- 将 spec 分成三层：
  - 已实现。
  - v0.2 必须实现。
  - 后续扩展。
- 每个验收项必须能对应：
  - 数据对象。
  - API。
  - UI 展示。
  - 测试或 demo 证据。

### G12. 演示脚本缺少冲突和决策张力

当前 demo script 是流程脚本，不是评审故事。

它展示“我点击了这些按钮”，但没有展示“为什么这个系统有价值”。

v0.2 演示应该围绕一个有冲突的运营场景：

1. 主办方要办旧区夜游，希望带动 8 个商户，但不能造成局部拥堵。
2. Agent 读取商户容量、营业时间、雨天适配和文化点位，生成方案。
3. 活动进行到 19:10，老字号库存售罄。
4. Agent 判断继续导流会造成体验下降，建议转移到备用手信商户。
5. 主办方确认，游客端路线和商户端执行包同步变化。
6. 20:00 出现强降雨，Agent 切换室内路线。
7. 复盘显示库存异常、雨天切换、打卡完成率和下次优化建议。

这个故事能让评委看到：

- Agent 有任务拆解。
- 工具有数据。
- 人有确认权。
- 系统有恢复能力。
- 结果能沉淀。

## 5. 当前版本可以保留的部分

虽然当前版本不成熟，但不是无用。

可保留：

- React + Vite + Ant Design 前端技术栈。
- FastAPI + Pydantic 后端技术栈。
- deterministic fallback 思路。
- 基础数据对象：
  - `EventBrief`
  - `EventPlan`
  - `MerchantProfile`
  - `MerchantRuntimeState`
  - `MerchantExecutionPack`
  - `RecoveryAction`
  - `ReviewReport`
  - `AuditLog`
- 主办方确认这个责任边界。
- 不接硬件、不接真实商户、不做真实交通预测的范围控制。

应重做或显著增强：

- Agent 编排结构。
- 事件恢复对象和 plan diff。
- 商户端状态更新。
- 游客端内容结构。
- 复盘指标。
- 演示故事。

## 6. v0.2 Redesign 的最低目标

v0.2 不应该追求功能更多，而应该追求证据更硬。

最低目标：

1. 一个明确真实痛点：旧区活动执行协同，而不是游客路线推荐。
2. 一个可观察 Agent workflow：至少 5 个 Agent step。
3. 一个可解释工具链：商户查询、路线约束、预算、状态、天气。
4. 一个真实状态触发：商户端库存变化触发恢复建议。
5. 一个真正 plan update：确认恢复后生成 `EventPlan v2`。
6. 一个游客端实际变化：路线、点位或任务发生可见变化。
7. 一个商户端实际变化：备用商户执行包发生变化。
8. 一个带指标的复盘：访问、打卡、商户负载、异常耗时、预算。
9. 一个评委可读的 Agent trace：输入、工具、输出、人工确认。

如果只做这 9 件事，项目就会从“玩具壳”接近“有说服力的竞赛 MVP”。

## 7. 建议的下一步

不要马上接 Qwen，也不要继续堆 UI。

建议下一步写：

`docs/proposal/v0.2-redesign.md`

该文档应明确：

- v0.2 的产品叙事。
- v0.2 的 Agent workflow。
- v0.2 的新数据对象。
- v0.2 的页面结构。
- v0.2 的验收标准。
- 哪些 v0.1 代码保留，哪些重构。

然后再写新的 implementation plan。

## 8. 一句话结论

当前版本不是失败，但它只是“能跑的壳”。如果继续沿着当前版本堆功能，会越来越像一个后台模板。下一步必须把项目核心从“页面流程”转成“可证明的多 Agent 运营协同系统”。
