# v2.7 Real Product Logic Foundation Spec

Date: 2026-06-28

Status: implemented.

## Purpose

v2.5 and v2.6 proved that the deterministic local beta can pass live API and
browser release gates. Those gates are still centered on the seeded
`demo-night-tour` fixture. v2.7 starts the first real product-logic phase:
organizer-created events must have their own lifecycle and must not silently
fall back to demo seed data.

The goal is not to replace every fixture or build a full commercial product in
one phase. The goal is to separate demo mode from real event state, make the
event lifecycle explicit, and harden the public interaction semantics that a
real merchant/tourist flow would immediately depend on.

## Readiness Target

v2.7 moves the project from "release-gated local browser beta" to
"product-logic beta foundation".

This means:

- a non-demo event can be created by an organizer;
- plan generation for that event uses an explicit event brief, not a demo seed
  fallback;
- approval and publication statuses describe real state transitions;
- public scan, claim, and redeem flows are repeat-safe enough for beta use;
- the deterministic demo remains available as a fixture and release-gate path.

v2.7 does not claim pilot or production readiness.

## Current Baseline

The current implementation already contains useful product entities:

- `EventBrief`
- `EventSummary`
- `PlanVersion`
- `MerchantTask`
- `MerchantInteractionPackage`
- `InShopTouchpoint`
- `CouponRule`
- `TouchpointInteraction`
- `CouponRedemption`
- `ReviewReport`
- `AuditLog`

The current weakness is that the main path still behaves like a demo system:

- `/api/events/demo/seed` creates the canonical event.
- `/api/events/{event_id}/generate-plan` calls `seed_demo(...)` when the event
  brief is missing.
- `seed_demo(...)` loads a fixed `demo_event.json` payload and also populates
  merchant, route, runtime, and mock metric records.
- approving a plan sets `public_release_status` to `published` before an event
  page is actually published.
- public touchpoint scans and coupon claims are not fully idempotent.
- several frontend API helpers and route defaults still use
  `demo-night-tour`.

## Scope

### In Scope

- Organizer-created non-demo events.
- Pre-plan event brief updates for non-demo events.
- Explicit lifecycle transitions for event, plan, and public release state.
- Demo seed isolation from real event creation.
- A shared local merchant/route catalog fixture for planning until real
  integrations exist.
- Non-demo public projection requiring an actually published event page.
- Idempotent public touchpoint scan, coupon claim, and coupon redeem semantics.
- Review report metrics derived from stored public interaction records.
- Minimal frontend API typing for creating/updating events.
- Focused API tests that prove the non-demo path and demo path both work.

### Out Of Scope

- Full organizer event creation UI redesign.
- Multi-tenant accounts, organizations, or billing.
- Real merchant onboarding and resource-level ACL beyond the current beta auth
  model.
- Real POS, payment, map, weather, traffic, QR hardware, or visitor identity
  integrations.
- Replacing SQLite JSON records with a relational production schema.
- Making Qwen, DashScope, or QwenPaw authoritative for product mutations.
- Changing v2.5 or v2.6 release gates away from the deterministic demo path.

## Product Model

### Event Lifecycle

v2.7 uses the existing `EventSummary.status` values:

| Status | Meaning |
| --- | --- |
| `draft` | Organizer has created an event brief, but no plan is ready for approval. |
| `pending_approval` | A plan version exists and needs organizer approval. |
| `active` | An approved current plan exists. |
| `ended` | Event operations are closed. Ending an event is not implemented in v2.7. |

### Public Release Lifecycle

v2.7 uses the existing `public_release_status` values:

| Status | Meaning |
| --- | --- |
| `draft` | No public event page has been published for the current event. |
| `published` | The current public page matches the current approved plan. |
| `stale` | The current approved plan changed after the last public page publish. |

The important correction is that plan approval is not the same as public
publication.

## Requirements

### Real Event Creation

- `REQ-EVENT-001`: Add `POST /api/events` for organizer-created events.
- `REQ-EVENT-002`: The create payload must include `title`, `area`, `date`,
  `time_window`, `budget_mop`, `target_audience`, `event_goal`,
  `theme_preferences`, `constraints`, and `priority_rules`.
- `REQ-EVENT-003`: The create payload may include `event_id`. Tests should use
  an explicit event id for deterministic assertions.
- `REQ-EVENT-004`: Creating an event must save both `EventBrief` and
  `EventSummary`.
- `REQ-EVENT-005`: New events start as
  `status=draft`, `current_plan_version=0`, and
  `public_release_status=draft`.
- `REQ-EVENT-006`: Duplicate event ids must return a conflict response instead
  of overwriting the existing event.
- `REQ-EVENT-007`: Add `PATCH /api/events/{event_id}` for pre-plan event brief
  updates.
- `REQ-EVENT-008`: Patching is allowed only while the event is still `draft`
  and `current_plan_version=0`.

### Demo Boundary

- `REQ-DEMO-001`: `/api/events/demo/seed` remains the only endpoint that seeds
  the demo event.
- `REQ-DEMO-002`: `POST /api/events/{event_id}/generate-plan` must return
  `404` when the event brief does not exist. It must not call `seed_demo(...)`
  for missing non-demo events.
- `REQ-DEMO-003`: Planning may use a shared local merchant/route catalog
  fixture until real integrations exist, but that catalog seed must not create
  `demo-night-tour`, demo operational metrics, or demo event records.
- `REQ-DEMO-004`: Existing demo gates and demo tests should keep working.

### Lifecycle Transitions

- `REQ-LIFE-001`: Generating the first plan for a real event creates
  `PlanVersion(version=1, status=draft)` and moves the event to
  `pending_approval`.
- `REQ-LIFE-002`: Approving a plan version moves the event to `active`, sets
  `current_plan_version`, and records approval metadata on the plan version.
- `REQ-LIFE-003`: Approving a plan must not set `public_release_status` to
  `published` unless a current public page has already been published for that
  same plan version.
- `REQ-LIFE-004`: Publishing an event page is the only operation in v2.7 that
  sets `public_release_status=published`.
- `REQ-LIFE-005`: If a current plan changes after publication, the event must
  become `stale` until the page is republished.

### Merchant Edge Foundation

- `REQ-MERCHANT-001`: Merchant interaction packages may only be generated for
  an approved current plan.
- `REQ-MERCHANT-002`: Generated packages for the current plan remain `active`.
- `REQ-MERCHANT-003`: v2.7 does not require package history retention. Keeping
  one current package per event and merchant is acceptable.
- `REQ-MERCHANT-004`: Merchant workbench summaries must continue to be scoped
  by `event_id` and current package children.

### Public Projection And Interaction Safety

- `REQ-PUBLIC-001`: For non-demo events, `GET /api/public/events/{event_id}`
  must require a published event page for the current approved plan.
- `REQ-PUBLIC-002`: The demo legacy public fallback can remain for
  `demo-night-tour` so existing release gates stay deterministic.
- `REQ-PUBLIC-003`: Public touchpoint interactions require an active
  touchpoint under a published event.
- `REQ-PUBLIC-004`: Public coupon claims require an active coupon rule under a
  published event.
- `REQ-PUBLIC-005`: Repeating the same touchpoint action for the same
  `(event_id, touchpoint_id, anonymous_interaction_id, interaction_type)` must
  return the existing interaction and must not increment metrics twice.
- `REQ-PUBLIC-006`: Repeating the same coupon claim for the same
  `(event_id, coupon_rule_id, anonymous_interaction_id)` must return the
  existing active redemption and must not create another claim metric.
- `REQ-PUBLIC-007`: Repeating coupon redeem on an already redeemed redemption
  must return the redeemed record and must not create another redeem metric.
- `REQ-PUBLIC-008`: Existing metadata sanitization must continue to remove
  visitor identity keys from stored public interaction metadata.

### Review Metrics

- `REQ-REVIEW-001`: Review reports must continue to include
  `touchpoint_summary`, `merchant_outcomes`, and `extension_tasks`.
- `REQ-REVIEW-002`: Those fields must derive from stored
  `TouchpointInteraction` and `CouponRedemption` records.
- `REQ-REVIEW-003`: v2.7 does not need to remove every hardcoded frontend
  review metric, but the API report must be correct for real event records.

### Frontend Contract

- `REQ-WEB-001`: Add TypeScript request types for real event create/update.
- `REQ-WEB-002`: Add `api.createEvent(...)` and `api.updateEvent(...)`.
- `REQ-WEB-003`: Keep demo defaults for existing release-gate routes unless a
  focused route is deliberately changed and covered by tests.
- `REQ-WEB-004`: A full event creation UI is out of scope for v2.7.

## Acceptance Criteria

v2.7 is complete when:

1. The v2.7 spec and implementation plan exist.
2. Focused API tests prove that organizer-created non-demo events can be
   created, updated before planning, planned, approved, published, interacted
   with publicly, and reviewed.
3. Focused API tests prove that missing non-demo events do not call
   `seed_demo(...)` or create `demo-night-tour`.
4. Focused API tests prove that plan approval and public publication are
   separate lifecycle transitions.
5. Focused API tests prove that repeat scan, claim, and redeem calls are
   idempotent and do not double-count metrics.
6. Existing demo flow tests still pass.
7. v2.5 release gate and v2.6 browser gate commands remain demo-based and are
   not broadened into real external integrations.
8. Full backend pytest passes.
9. Web tests and build pass if frontend contract files are changed.
10. Repository hygiene and whitespace checks pass.
11. `docs/ai/STATUS.md`, `docs/ai/VERIFY.md`, and `docs/ai/NEXT.md` record the
    new boundary and recommended next phase.

## Stop Gates

Stop and ask for review if:

- implementation would remove or destabilize the deterministic demo release
  gates;
- real event creation would require real merchant, POS, payment, map, weather,
  traffic, QR hardware, or visitor identity integrations;
- the work starts a broad frontend redesign instead of the API-first logic
  foundation;
- Qwen, DashScope, or QwenPaw output would approve, publish, or mutate
  authoritative state;
- public endpoints would expose operator-only package ids, tokens, evidence
  refs, raw metrics, or internal model terms;
- unrelated repository changes would need to be staged with v2.7.
