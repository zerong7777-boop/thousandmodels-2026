- deterministic FastAPI backend loop

- Chinese, Traditional Chinese, and English UI

The purpose of v0.7 is to turn the current running system into a competition-ready narrative:

- explain the original vision in a judge-readable way

- distinguish deterministic demo capability from future full-platform ambition

- make the Agent value legible before adding Qwen/QwenPaw
- document gaps honestly without making the project look unfinished

v0.7 is a documentation, storytelling, and evidence-packaging phase. It must not introduce Qwen/QwenPaw, model APIs, backend workflow rewrites, real merchant onboarding, hardware, payment, map APIs, or traffic prediction.

## 3. Problem Statement

The initial engineering overview describes a broad AI collaborative operating system for Macau old-district cultural-tourism events. The current product is a high-completion MVP of that vision, but not the complete platform.

If this distinction is not explained, the project can be misunderstood in two opposite ways:

1. It may look smaller than the original vision because real multi-agent Qwen/QwenPaw orchestration, RAG, merchant onboarding, and knowledge accumulation are not fully implemented yet.
2. It may look overclaimed if the presentation implies that every full-platform capability already exists.

v0.7 must solve this communication problem.

The desired message is:

```text
Zhiyin Haojiang is not trying to be a tourist recommendation app.
It is an AI-assisted event orchestration system for old-district cultural-tourism operations.
The current MVP proves the most important engineering claim:
an event can move through planning, merchant execution, live exception recovery,
visitor update, and metric-backed review as one controlled loop.
```

## 4. v0.7 Positioning Decision

### 4.1 What v0.7 Is

v0.7 is:

- a demo narrative package
- a judge walkthrough
- a screenshot-backed evidence layer
- a bridge from current MVP to future Qwen/QwenPaw integration
- a clear explanation of what is implemented, simulated, and planned

### 4.2 What v0.7 Is Not

v0.7 is not:

- a Qwen/QwenPaw integration phase
- a new UI redesign phase
- a backend workflow redesign phase
- a real merchant onboarding phase
- a production SaaS hardening phase
- a pitch deck full of marketing claims without runnable evidence

### 4.3 Main Recommendation

The recommended v0.7 narrative stance is:

```text
Complete deterministic MVP first, model-assisted intelligence second.
```

This means the presentation should explicitly say that the required demo path is deterministic and stable, while Qwen/QwenPaw can later be added as an advisory layer for:

- route wording suggestions
- recovery explanation drafts
- public notice variants
- review prose
- merchant onboarding text extraction
- cultural narrative generation

Model output should not be described as directly mutating approvals, inventory, route publication, or plan persistence.

## 5. Audience

v0.7 materials should serve four audiences.

| Audience | What they need to understand | Required narrative emphasis |
| --- | --- | --- |
| Competition judges | Why the project is useful and technically relevant | closed loop, role separation, Agent orchestration, local Macau scenario |
| Technical reviewers | What is actually implemented | deterministic backend, structured objects, tests, route-level demo evidence |
| Non-technical reviewers | What problem it solves | organizer burden, merchant participation, visitor updates, post-event review |
| Future implementers/agents | What to build next | Qwen/QwenPaw advisory layer after demo narrative is stable |

## 6. Core Story

The demo story should be one concrete event, not a generic platform tour.

Recommended story:

```text
A weekend night-walk event is planned for Fulong New Street, Senado/New Road, and the Inner Harbour.
The organizer wants to connect old shops, cultural stories, visitor check-ins, and merchant capacity.
During execution, one merchant sells out.
The system detects the incident, proposes a recovery plan, asks the organizer to approve,
publishes an updated visitor route, updates merchant tasks, and generates a review report.
```

The story should avoid feature listing. Every screen should answer one operational question:

| Step | Operational question |
| --- | --- |
| Login | Which role is entering the system? |
| Organizer dashboard | What event is live and what needs attention? |
| Event workspace | What plan was generated and why is it credible? |
| Merchant status | What changed on the ground? |
| Exception center | What should the organizer approve? |
| Public H5 | What does the visitor see after recovery? |
| Review center | What evidence did the system retain after the event? |

## 9. Competition Claim Boundaries

v0.7 materials must use precise language.

Allowed claims:

- The MVP demonstrates a deterministic closed loop for old-district event orchestration.
- The system separates organizer, merchant, tourist, and public surfaces.
- The backend stores structured planning, incident, recovery, public notice, and metric objects.
- Merchant status updates can trigger organizer-facing incidents.
- Recovery approval creates a new plan version and visitor-facing notice.
- The system can run locally without `DASHSCOPE_API_KEY`.
- Qwen/QwenPaw is a planned advisory layer, not a required demo dependency.

Avoid these claims unless implementation changes later:

- "The system already uses QwenPaw multi-agent collaboration end to end."
- "The system supports real merchant onboarding."
- "The system predicts real traffic or crowd flow."
- "The system integrates real maps, POS, weather, or payments."
- "The system has production-grade privacy compliance."
- "The system has learned reusable knowledge from many historical events."

## 12. Technical Story Requirements

v0.7 must explain the system architecture in competition-friendly terms.

Recommended technical framing:

```text
React/Vite frontend with role-specific product surfaces
+ FastAPI backend with structured Pydantic objects
+ SQLite persistence for deterministic demo state
+ deterministic agent backend for repeatable local demonstration
+ optional Qwen adapter reserved for later advisory generation
+ audit, approval, plan versioning, incident, recovery, public notice, and review objects
```

The architecture explanation should emphasize:

- structured objects over raw chat output
- human approval for public-facing changes
- deterministic demo path for reliability
- swappable model backend for future Qwen/QwenPaw work
- public H5 projection separated from internal operations

## 13. Qwen/QwenPaw Roadmap Narrative

v0.7 should include a cautious roadmap, not an implementation promise.

Recommended phases:

| Stage | Model role | Risk control |
| --- | --- | --- |
| Stage 1: Advisory text generation | draft recovery explanations, public notices, review prose | model output stored as draft only |
| Stage 2: Structured planning assist | propose route/merchant/risk JSON candidates | schema validation and deterministic fallback |
| Stage 3: Merchant onboarding assist | parse merchant natural language/image/voice descriptions | human merchant confirmation |
| Stage 4: Knowledge-grounded narrative | generate cultural stories from curated knowledge base | source citation and content review |

The roadmap must say:

```text
The current competition demo does not require model API availability.
```
