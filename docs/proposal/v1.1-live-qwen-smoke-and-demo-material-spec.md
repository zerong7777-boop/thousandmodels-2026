## 1. Purpose

v1.0 made Qwen integration real at the architecture level:

```text

+ controlled draft generation
+ schema/public-copy/unsafe-mutation guards
+ deterministic fallback
+ AgentModelCall evidence
+ organizer-only model evidence UI
```

The remaining gap is not broad model capability. The remaining gap is proof and communication:

```text

Can judges understand why this is an Agent system instead of a static demo or chatbot?
```

v1.1 should answer both questions without weakening the deterministic demo.

The target stance:

```text
Run one controlled live Qwen smoke path.
Record exactly what happened.
Package the existing Agent evidence into competition-ready material.
Keep Qwen optional, constrained, and non-authoritative.
```

## 3. Problem Statement

v1.0 proves the controlled Qwen lane through mocked provider tests and UI evidence. That is enough for software correctness, but not enough for a competition presentation.

If the project claims "Qwen-assisted Agent" without one real API smoke, judges may see the Qwen layer as theoretical. If the project jumps directly to QwenPaw or model-generated route plans, the system risks losing the strongest boundary it currently has: deterministic, auditable operations.

v1.1 should therefore be a narrow proof-and-packaging phase:

```text
one live Qwen controlled draft smoke
+ evidence document
+ screenshots
+ demo script
+ architecture/story material
+ no new authoritative model powers
```

## 4. Positioning Decision

v1.1 is:

- a live API smoke phase
- a secrets-hygiene phase
- a judge-facing evidence packaging phase
- a demo narrative and architecture explanation phase

v1.1 is not:

- a QwenPaw workflow implementation
- a model-generated route planning phase
- a prompt engineering sprint for open-ended autonomy
- a broad frontend redesign
- a product feature expansion phase
- a real merchant, hardware, POS, payment, map, weather, or traffic integration phase

The v1.1 claim should be:

```text
The system can call Qwen through DashScope for controlled Agent drafts.
The call is optional, guarded, recorded, and never authoritative.
The stable demo still runs without any external model API.
```

## 5. Scope

### 5.1 In Scope

v1.1 includes:

- a manual live Qwen smoke procedure
- secret handling instructions that keep keys out of git, docs, screenshots, logs, and test fixtures
- one or more manual live smoke scripts or commands that are never part of default automated tests
- a recorded live smoke result document
- organizer screenshots showing live model evidence when the live smoke succeeds
- deterministic fallback verification after the live smoke
- competition demo script
- judge-facing architecture explanation
- slide/storyboard outline
- screenshot index for the existing v0.5, v0.9, and v1.0 evidence
- concise explanation of why Qwen is limited to controlled drafts

### 5.2 Out Of Scope

v1.1 must not include:

- mandatory `DASHSCOPE_API_KEY`
- storing real API keys in repo files
- printing real API keys in logs or docs
- live DashScope calls in normal `pytest`, Vitest, build, or Playwright suites
- QwenPaw workflow orchestration
- Qwen-generated `PlanVersion`
- Qwen-generated route ordering as authoritative state
- model approval or publication actions
- model mutation of merchant inventory, queue, availability, auth, session, or database state
- real merchant onboarding
- real map, weather, traffic, POS, payment, or hardware integrations
- broad UI redesign
- new public-facing model UI

## 6. Live Qwen Smoke Design

### 6.1 Configuration

The live smoke uses the existing v1.0 configuration:

```text
AGENT_DRAFT_BACKEND=qwen
DashScope API key supplied from the local process environment, outside git
QWEN_MODEL=qwen-plus
QWEN_TIMEOUT_SECONDS=30
```

Default local development remains:

```text
AGENT_DRAFT_BACKEND=deterministic
```

The implementation plan should avoid `.env` commits. If a local `.env` is used, it must be ignored by git and never copied into docs.

### 6.2 Smoke Target

The live smoke should exercise the narrowest useful path:

```text
seed demo
-> create or reuse demo event
-> trigger merchant sold-out incident
-> generate incident recovery proposal/drafts with AGENT_DRAFT_BACKEND=qwen
-> generate review report/draft with AGENT_DRAFT_BACKEND=qwen
-> read organizer-only AgentModelCall and AgentEvaluation evidence
```

The smoke should prefer these draft types:

```text
public_notice
review_summary
```

Rationale:

- `public_notice` proves public-copy guard behavior.
- `review_summary` proves metric/evidence-grounded model drafting.
- Both are text-heavy and useful to show judges.

### 6.3 Accepted Outcomes

The ideal live smoke outcome:

```text
provider=dashscope
model=qwen-plus or configured QWEN_MODEL
response_status=success
fallback_used=false
AgentDraft(status=draft)
AgentEvaluation passes guard checks
```

Acceptable non-success outcomes:

```text
response_status=invalid_json
response_status=schema_failed
response_status=provider_error
response_status=skipped
fallback_used=true
```

Those outcomes are acceptable only if they are documented clearly as live smoke results. They do not support a claim that Qwen produced the final draft, but they do support the claim that the system safely falls back and records evidence.

For competition material, at least one successful live Qwen draft is strongly preferred before claiming "live Qwen-assisted draft generation" in slides.

### 6.4 Evidence To Capture

Create:

```text
docs/research/v1.1-live-qwen-smoke.md
docs/research/assets/v1.1-live-qwen-smoke/
```

The smoke document should record:

- date and local environment
- exact commands without secret values
- whether `AGENT_DRAFT_BACKEND=qwen` was active
- provider/model/status for each `AgentModelCall`
- whether fallback was used
- validation/evaluation notes
- whether any public-copy guard failed
- whether deterministic fallback still works after unsetting model env vars
- screenshots or text evidence paths

Do not record:

- raw API key
- Authorization header
- full raw provider request body if it includes sensitive context
- full raw provider response if later prompts include sensitive user data

### 6.5 Failure Handling

If live Qwen smoke fails due to key, network, quota, model output format, or provider error:

1. Record the failure as evidence.
2. Verify deterministic fallback still works.
3. Do not change public demo behavior to depend on Qwen.
4. Do not broaden prompts into unsafe state mutation just to force a success.
5. Do not claim live Qwen success in demo materials.

## 8. Safety And Claim Boundaries

v1.1 materials must preserve these exact boundaries:

```text
Qwen may draft controlled text.
Qwen may summarize evidence.
Qwen may help explain recovery decisions.
Qwen may not approve, publish, mutate, route, or create authoritative versions.
```

Forbidden slide/demo claims:

- "QwenPaw orchestrates the workflow" unless QwenPaw is implemented later.
- "Qwen autonomously updates the route."
- "Qwen approves recovery."
- "Qwen publishes visitor notices."
- "The demo requires DashScope."
- "The system is connected to real merchants/hardware/maps/traffic."

Preferred claims:

- "Qwen is used as an optional controlled draft generator."
- "Every model output is recorded as evidence."
- "Unsafe or invalid model output falls back deterministically."
- "Operational state changes only through backend workflow and organizer approval."
- "The public H5 receives only approved visitor-safe projection data."

## 9. Testing And Verification Requirements

### 9.1 Normal Regression

Normal verification must still pass without live model access:

```powershell
cd <PROJECT_ROOT>\apps\api
python -m pytest -q

cd <PROJECT_ROOT>\apps\web

```

### 9.2 Live Smoke Verification

The implementation plan should define one explicit manual command path for live smoke. It may be a script, a marked test, or a documented API sequence, but it must not run during normal regression.

Recommended guard:

```text
RUN_LIVE_QWEN_SMOKE=1
AGENT_DRAFT_BACKEND=qwen
DashScope API key set outside repo
```

The smoke should fail fast if:

- `RUN_LIVE_QWEN_SMOKE` is not set
- `DASHSCOPE_API_KEY` is not set
- the target backend is not configured for `AGENT_DRAFT_BACKEND=qwen`

## 10. Acceptance Criteria

v1.1 spec is complete when:

1. `docs/proposal/v1.1-live-qwen-smoke-and-demo-material-spec.md` exists.
2. It defines live Qwen smoke as manual and optional.
3. It keeps deterministic demo behavior as the default.
4. It forbids committing or documenting real API keys.
5. It defines accepted success and fallback outcomes for live smoke.
6. It requires a smoke evidence document and screenshot folder.
7. It defines demo script, architecture brief, slide outline, and screenshot index deliverables.
8. It explicitly says QwenPaw is not implemented in this phase.
9. It preserves organizer-only model evidence and public/merchant/tourist non-exposure.
10. It defines verification commands and boundary scans.

v1.1 implementation is complete only when:

1. A manual live Qwen smoke procedure exists and is documented.
2. The live smoke is run once or explicitly recorded as blocked with reason.
3. No real API key appears in git diff, docs, logs, or screenshots.
4. Deterministic regression still passes without `DASHSCOPE_API_KEY`.
5. If live Qwen succeeds, organizer screenshots show live model evidence.
6. If live Qwen fails, deterministic fallback evidence is recorded and no live-success claim is made.
7. Demo script, architecture brief, slide outline, and screenshot index exist.
8. docs/ai handoff files are updated with actual results.

## 11. Stop Gates

Stop and ask for review if:

- live API validation would require storing the key in repo files
- live API validation would require weakening schema, public-copy, or unsafe-mutation guards
- Qwen output would need to create or approve `PlanVersion`
- Qwen output would need to publish `PublicNotice`
- Qwen output would need to mutate merchant runtime state
- live smoke requires changing normal regression tests to depend on DashScope
- public/merchant/tourist pages would expose raw model/backend terms
- the work drifts into QwenPaw implementation
- the work drifts into real merchant/hardware/map/payment/traffic integrations
