# v0.7 Competition Brief

Date: 2026-06-12
Status: draft for competition material preparation

## Project Title

Zhiyin Haojiang: AI-Assisted Event Orchestration for Macau Old-District Cultural Tourism

## Target Users

| User | Need |
| --- | --- |
| Organizer | Plan, approve, monitor, recover, and review cultural-tourism events. |
| Merchant | Receive event tasks and report live stock, queue, and visitor capacity. |
| Visitor | Follow a lightweight H5 route with stories, tasks, and live updates. |

## Pain Points

1. Old-district events require heavy manual coordination.
2. Small merchants have low digital capacity and cannot easily join complex systems.
3. Static route publishing responds poorly to stock, weather, and queue changes.
4. Event experience is hard to review and reuse after the activity ends.

## Product Flow

```text
Organizer event workspace
-> route plan and merchant tasks
-> merchant live status update
-> incident and recovery proposal
-> organizer approval
-> public H5 update
-> metric-backed review
```

## Current Demo Scope

The current MVP focuses on one Macau old-district night-walk event with simulated merchants and deterministic incidents. It proves the closed loop rather than full city-scale SaaS coverage.

## Technical Architecture

```text
React/Vite role-separated frontend
+ FastAPI backend
+ Pydantic schemas
+ SQLite persistence
+ deterministic agent backend
+ optional Qwen adapter for future advisory generation
+ audit, approval, plan versioning, incident, recovery, public notice, and review objects
```

## Innovation Points

1. The project targets event orchestration rather than generic tourist recommendation.
2. The MVP connects organizer, merchant, visitor, incident recovery, and review as one loop.
3. It treats AI output as structured operational objects instead of free-form text.
4. It keeps human approval for public-facing changes.
5. It provides a clear path for Qwen/QwenPaw advisory capability without making model APIs required for the demo.

## Current Limitations

1. Qwen/QwenPaw multi-agent orchestration is not yet implemented end to end.
2. Merchant onboarding uses demo data rather than natural-language, image, or voice parsing.
3. Route and weather behavior are deterministic or mocked, not real external integrations.
4. Knowledge accumulation is represented by review outputs and curated files, not a mature cross-event memory system.
5. Privacy and audit boundaries are MVP-level rather than production compliance.

## Qwen/QwenPaw Roadmap

| Stage | Model role | Safety boundary |
| --- | --- | --- |
| 1 | Draft recovery explanations, public notices, and review prose. | Draft only; deterministic workflow remains source of truth. |
| 2 | Suggest structured route, merchant, risk, and budget candidates. | Schema validation and human approval required. |
| 3 | Parse merchant natural language, image, or voice onboarding input. | Merchant confirms parsed profile before use. |
| 4 | Generate cultural stories from curated local knowledge. | Source review and public copy approval required. |
