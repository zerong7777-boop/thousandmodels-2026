## CI Workflow Details

The workflow has three jobs:

1. `api-tests`: uses Python 3.11, installs `apps/api` with `.[dev]`, and runs `python -m pytest -q`.
2. `web-tests`: uses Node 20, runs `npm ci`, `npm run test`, and `npm run build` in `apps/web`.
3. `repo-hygiene`: uses Python 3.11 and runs `python scripts/repo_hygiene.py --base "$BASE_SHA"`.

The workflow uses `fetch-depth: 0` for the hygiene job so changed-file comparisons have enough history.

## Hygiene Boundaries

The secret scanner is intentionally high-confidence. It is not a replacement for a dedicated enterprise secret scanner, but it catches the common failure modes relevant to this repo:

- raw `sk-...` style keys,
- high-entropy bearer tokens,
- provider key/password assignments with concrete values,
- AWS-style access key IDs.

The local-path scanner targets user home paths and known local workspace roots. It avoids failing on placeholder paths such as `<PROJECT_ROOT>`.

The generated-artifacts guard checks committed changes relative to the provided base. It does not fail on unstaged local files. This is deliberate because v2.0 must not hide or mutate the current historical PNG byproducts.

Those files need a separate explicit decision:

1. Restore them to the committed baseline.
2. Commit them as a named evidence refresh.
3. Regenerate and replace them through a dedicated visual evidence task.

Until that decision is made, future v2.0 commits should keep them unstaged.

## Remaining Boundary

v2.0 closes the local implementation part of the CI/repo-hygiene gap. It does not prove remote CI is green until the branch is pushed and GitHub Actions completes.

After remote CI is observed, the recommended next hardening phase is v2.1 auth/session/CSRF hardening.
